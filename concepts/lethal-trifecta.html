Lethal Trifecta

Simon Willison's risk framework: an agent combining private-data access, exposure to untrusted content, and an external communication channel is structurally exploitable — remove one leg to defuse it.

  • Running Python code in a sandbox with MicroPython and WASM Simon Willison · 2026-06-06 Reader 🖥️
    WebAssembly-embedded MicroPython is the first Python sandbox that meets all the criteria Willison thinks a trustworthy in-process sandbox must satisfy — clean PyPI install, memory/CPU limits, controlled file and network access, host-function interop, and active upstream maintenance — making it the right substrate for safely running plugin and agent-generated Python code.

Open gaps (14)

  • incompleteFowler offers no mechanism for detecting when 'disposable' software is silently graduating into consequential software. The disposable/consequential boundary is treated as legible to the author, but in practice it is one of the most common failure modes in software organisations — internal scripts becoming critical infrastructure.
    Bliki: Vibe Coding — read next: Cutler on institutional overload and shadow systems; classic discussions of 'tactical' vs 'strategic' programming (Ousterhout, Philosophy of Software Design); literature on shadow IT and end-user computing.
  • incompleteThe security treatment is gestural — pointing at the Lethal Trifecta — and does not address the asymmetry that non-programmers, the very population vibe coding most empowers, are least equipped to recognise security trifectas in their own systems.
    Bliki: Vibe Coding — read next: Simon Willison on prompt injection and lethal trifectas; agentic-AI security taxonomies; end-user security literature.
  • incompleteNo engagement with the cultural/professional question of whether vibe coding deserves the name 'coding' at all, or whether legitimising it as a coding practice (rather than, say, 'spec-driven prototyping') invites exactly the diffusion problem Fowler is trying to push back against.
    Bliki: Vibe Coding — read next: Naur on programming as theory building; the broader 'what is programming' debate that has resurfaced post-LLM.
  • incompleteThe 'block the data path, not the model' thesis governs what data reaches a model but leaves inference-time and output-side risk uncontrolled. The article concedes that models 'can exfiltrate entire databases through inference APIs if improperly secured' and never returns to it — prompt injection, model-output filtering, and egress through inference sit outside the data-path frame yet are where much AI-specific risk lives.
    Governing AI in the Cloud: A Practical Guide for Architects — read next: Read alongside agent-security / lethal-trifecta and runtime output/egress controls to cover inference-time and behavioral risk.
  • omissionThe article mentions gRPC in the title but never discusses it in the body. The Transport Working Group's SEP-1442 is about stateless HTTP, not gRPC. The title implies transport diversification that the content does not substantiate.
    AAIF's MCP Dev Summit: Gateways, gRPC, and Observability Signal Protocol Hardening — read next: Check whether gRPC transport was discussed in sessions not covered by the article; review the Transport Working Group charter.
  • tensionThe framing of context bloat as 'solved' at the client layer via progressive discovery creates tension with interoperability: if each client implements discovery differently, tool authors cannot predict which tools will be visible. No standardization of progressive discovery is mentioned.
    AAIF's MCP Dev Summit: Gateways, gRPC, and Observability Signal Protocol Hardening — read next: Look for MCP spec proposals standardizing tool discovery negotiation between client and server.
  • implicit_premiseThe article assumes that the Kubernetes analogy (interceptors as admission webhooks, gateways as ingress controllers) is the right mental model for MCP governance. This frames MCP as infrastructure middleware, which may not be the only valid framing -- it could also be framed as a capability/permission system, a marketplace, or a service mesh.
    AAIF's MCP Dev Summit: Gateways, gRPC, and Observability Signal Protocol Hardening — read next: Compare with capability-based security models (e.g., object-capability model) and service mesh patterns (Istio/Envoy) for alternative framings.
  • omissionThe article does not address how MCP Apps (interactive UI in iframes) interacts with the gateway pattern -- does UI traffic flow through the gateway? Are there separate security considerations for bidirectional postMessage communication?
    AAIF's MCP Dev Summit: Gateways, gRPC, and Observability Signal Protocol Hardening — read next: Review the MCP Apps specification for security model details.
