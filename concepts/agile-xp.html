Tensions

What replaces PRs?
from: Are pull requests still necessary in software engineering, given trunk-based development and pair programming alternatives, AND given AI-accelerated code generation that radically changes the volume and nature of code authorship?
  • Laforgia (Four Eyes): Replace PRs with a composite control: policy + commit trailers + enforcement hooks + telemetry + quarterly sampling audit. Pair programming is the substance; structured artefacts capture the evidence.
  • Laforgia (T*D): The convergent T*D pattern (TDD + TBD + TFD) makes the question moot. Review is continuous; no separate replacement needed.
  • Podcast: The AI Joy Gap: Why Some Developers Thrive While Others Struggle (Parker): Fundamental SDLC redesign required, but the endpoint is unclear. Names the problem rather than the replacement.
  • Developer ramp-up time continues to accelerate with AI (Engineering Enablement): Silent — operates inside PR paradigm. Implicit: keep PRs, just compress the cycle.
  • Henrik Kniberg's Post (Kniberg): Agents author the PRs; humans review the output. The PR becomes a transactional artefact between agent producer and human verifier.
  • On Pair Programming (Siessegger): Pair by default, PR for exceptions (solo production changes). Explicit middle path.
  • From Async Code Reviews to Co-Creation Patterns (Stepanović): Maximal co-creation. Drive PR transaction cost to zero by making review continuous. Optimal batch size approaches 1 line.
  • Trust Factory (Beck): No direct prescription for a PR replacement, but 'slow development to create space for person-to-person interaction' implies pairing and mob programming as the structural answer. Beck's logic further implies that solo PR review — whether reviewing human or AI-authored code — is itself trust-negative: the single-player problem applies to the reviewer, not just the author. Laforgia's Four Eyes composite control is the closest existing instantiation of what Beck is gesturing at: it operationalizes the trust-preserving form of review (pair programming as substance) rather than solo PR review as artefact-checking.
Assessment: Three axes of disagreement. (1) How much PR ritual to keep — Laforgia Four Eyes keeps the artefact-shape via trailers; Siessegger keeps PR for solo exceptions; Stepanović eliminates the PR-as-event entirely. (2) Whether to engage AI as part of the answer — Kniberg pushes furthest; Parker names the problem; Beck provides the theoretical basis for why the social architecture around the LLM determines the answer. (3) What the unit of value is — for Laforgia it's the evidence of control; for Stepanović it's throughput; for Beck it's trust accumulation; for Kniberg it's human cognitive bandwidth. The Kniberg position (agent-authored PRs, human verifier) is now further complicated by Beck: a human reviewing an agent's PR alone is in single-player mode — trust-negative even when the compliance and quality boxes are checked. Kniberg's more expansive vision (humans focused on higher-level design in a multi-agent team) may preserve trust; the PR-as-solo-review-act does not.
What is code review primarily FOR?
from: Are pull requests still necessary in software engineering, given trunk-based development and pair programming alternatives, AND given AI-accelerated code generation that radically changes the volume and nature of code authorship?
  • Siessegger / Stepanović: Quality production + accelerated learning. Review's purpose is built in at the moment of writing.
  • Laforgia (Four Eyes): Compliance evidence is real; quality is what pairing produces; the two are separable controls.
  • Podcast: The AI Joy Gap: Why Some Developers Thrive While Others Struggle (Parker): Quality at scale. Worries about reviewing 10,000-line PRs OR 10,000 single-line PRs — neither is feasible. Quality is the dominant lens.
  • Henrik Kniberg's Post (Kniberg): Verification of agent output. Review is the human safety check on autonomous production.
  • Trust Factory (Beck): Trust manufacture between practitioners. Every team practice is primarily a trust production mechanism: trustworthy behavior → trustworthy software → trust between humans. Quality and learning are downstream effects, not primary purposes. The practitioner is changed by the practice (trustworthy behavior becomes structural, not effortful), not just the product.
Assessment: Beck's addition is the most significant divergence contribution in v2. He fills the 'review-as-cultural-induction' gap the original panel flagged: pairing produces trust-as-induction (the practitioner is changed by being held accountable in structural ways) rather than merely information-transfer-as-induction. This is not in conflict with the other positions — it provides the mechanism that makes those answers work and sustainable. Quality (Siessegger/Stepanović), compliance (Laforgia), and verification (Kniberg) are all downstream of trust accumulation, on Beck's account. The live disagreement is no longer simply 'PR vs not PR' — it is: if trust is the primary function, and trust requires person-to-person interaction that cannot be manufactured by code artefacts or automated review, what is the minimum set of practices that preserves trust in the AI-augmented era? Architecture-coherence review (the senior reviewer who refuses design-wrong changes) remains unaddressed by all panel members including Beck.
Is AI-accelerated coding a feature or a threat to existing workflows?
from: Are pull requests still necessary in software engineering, given trunk-based development and pair programming alternatives, AND given AI-accelerated code generation that radically changes the volume and nature of code authorship?
  • Podcast: The AI Joy Gap: Why Some Developers Thrive While Others Struggle (Parker): Threat. The joy gap is asymmetric harm; AI overwhelms the parts of the workflow it doesn't help.
  • Henrik Kniberg's Post (Kniberg): Feature. Agents become teammates; the human team gets to focus on higher-level work.
  • Developer ramp-up time continues to accelerate with AI (Engineering Enablement): Feature. Faster ramp-up is intrinsically good (with caveats).
  • Laforgia (both), Siessegger, Stepanović: Silent / predates AI. Their frameworks still apply but they don't engage the question.
  • Trust Factory (Beck): Structurally a threat when deployed in single-player mode (one human + LLM genie, bypassing team interaction). Potentially trust-preserving when deployed in multi-player or team-mediated form (pairing with LLMs inside the team structure, not replacing it). Beck distinguishes the technology from the deployment architecture — the LLM is not inherently anti-trust; the single-player usage pattern is.
Assessment: Beck triangulates between Parker (threat) and Kniberg (feature) with the most analytically precise position in the corpus: the social architecture around the LLM determines whether it's a trust factory or a trust destroyer. This has a direct implication for the PR question: AI-authored code reviewed by a solo human is the worst configuration — two single-player failure modes stacked (AI authoring + human solo review). The crucial remaining gap: no one has produced a synthesis treatment that combines Stepanović's Reinertsen math with Beck's trust-asymmetry argument and Kniberg's agent-as-author shift. Stepanović's framework directly implies AI in-context review could be a near-zero-cost reviewer; Beck's framework implies that same review is trust-negative if done solo. The two arguments together demand a configuration — team-mediated AI code review — that none of the panel members has yet theorised.

  • Podcast: The AI Joy Gap: Why Some Developers Thrive While Others Struggle (Parker): No — the joy gap is context-dependent. Greenfield and legacy diverge sharply.
  • Henrik Kniberg's Post (Kniberg): Implicit yes — Abundly's practice extrapolates.

  • Trust Factory (Beck): Silent on context-dependence; the trust factory claim is presented as universal. However, Beck's own unaddressed gaps (the cultural specificity of trust, whether XP worked because of practices or pre-existing high-trust subculture) suggest a context-dependence he does not engage.
Assessment: Parker is the lone explicit voice for context-dependence. Beck's trust factory framing is implicitly universalist — but his unaddressed gaps echo Parker's concern from a different angle: if trust factories only work in pre-existing high-trust organizational contexts, the adoption prescription fails precisely where it is most needed.

  • Software Design in the Agentic Age: Place Your Bets (Mathias Verraes): Evidenced: feeding an agent an explicit domain model first yields better code structure and database performance, and bounded contexts also reduce token cost.

What is the right metaphor for the human-AI relationship in coding?
from: What is the right approach to building harnesses around AI coding agents, and where are the limits?
  • Patterns for Reducing Friction in AI-Assisted Development (Garg): AI is a teammate — treat it like a junior pair programmer who needs onboarding, context, and rituals.
  • Harness engineering for coding agent users (Böckeler): AI is a system to be governed — the cybernetic governor model with feedforward and feedback controls.
  • Agent Harness Engineering (Osmani): AI is a runtime to be engineered — the harness is an engineering artifact, not a social relationship.
  • Looking back at the Future of Software Engineering unconference (Thoughtworks): AI is a new organizational participant — Conway's Law applies, and agent topologies must be designed.
Assessment: These metaphors are not mutually exclusive but they lead to different design decisions. Garg's teammate metaphor emphasizes collaboration rituals and may underplay the need for hard constraints (you don't sandbox a teammate). Böckeler's governor metaphor is the most theoretically rigorous but may overclaim the applicability of control theory to creative work. Osmani's engineering metaphor is the most actionable but may miss the social and organizational dimensions. Thoughtworks' organizational metaphor captures the broadest scope but is the hardest to operationalize.
Will better models make current harness practices obsolete?
from: What is the right approach to building harnesses around AI coding agents, and where are the limits?
  • Agent Harness Engineering (Osmani): Harnesses don't shrink, they move — better models create new failure modes at higher capability levels.
  • Genie Tarpit (Beck): The Bitter Lesson (scale + compute) is listed as one possible solution — implying better models might eventually solve the problem without harnesses.
  • Patterns for Reducing Friction in AI-Assisted Development (Garg): Does not address model improvement — the framework lacks a theory of its own obsolescence.
Assessment: Osmani makes the strongest case for permanence with concrete evidence of scaffolding migration. Beck keeps the door open for the Bitter Lesson hypothesis. The truth likely depends on the type of harness: mechanical safeguards (sandboxes, permission controls) may always be needed, while knowledge-transfer patterns (AGENTS.md, priming docs) may become less necessary as models improve at in-context learning.
Where does the non-delegable human asset now live — the harness/platform, the oracle, the fundamentals, or the developer's understanding?
from: What is the future of software development in the agentic/AI age?
  • Notes from FOSE Europe (Giles Edwards-Alexander): In the harness — the continuously-evolving scaffolding of architecture guidance and rigour around the agents — though it must stay team-distributed, not centralized.
  • What Survived Contact (Sam Ruby): In the oracle — the objective's evaluation is the single thing the person directing the model cannot delegate back to it.
  • Jimmy Nilsson's Post — reflections after the Thoughtworks 'Future of Software Development' retreat (Jimmy Nilsson): In the fundamentals plus a renewed 'Extreme Humanism' — the durable disciplines and the conspicuously human element.
  • AI Is Changing Software Engineering. The Fundamentals Still Matter. (Abby Bangser): In right-sized boundaries and the operational loop — humans must stay in operations because their learning is the irreplaceable asset.
  • Fragments: February 9 (Martin Fowler (Feb 9)): In the developer's model-building and shared understanding — the craft of devising abstractions to reason about the domain.
Assessment: This is the deepest structural disagreement — every author agrees a human residue remains but nominates a different residue, and the choice determines what to invest in and what to teach. The harness-centric (Edwards-Alexander), oracle-centric (Ruby), fundamentals-centric (Nilsson/Bangser), and understanding-centric (Fowler) camps are not merely emphases: they imply different career ladders, different hiring, and different failure modes. Ruby's oracle and Bangser's operational-learning are the most operationally concrete; Nilsson's is the least argued. The absence in the corpus of anyone who holds 'nothing is non-delegable' is itself telling (see unexamined assumptions).
Does the current evidence favor optimism or doubt about agentic engineering's payoff?
from: What is the future of software development in the agentic/AI age?
  • Jimmy Nilsson's Post — reflections after the Thoughtworks 'Future of Software Development' retreat (Jimmy Nilsson): Optimistic: the mature, nuanced, human-centred discourse among leading practitioners is a good predictor of a healthy trajectory.
  • The Future Of Software Engineering (Thoughtworks): Pessimist-realist: there is an active regression underway that most optimism ignores.
  • What Survived Contact (Sam Ruby): Doubt is currently better-evidenced than hope, and that asymmetry is the uncomfortable thing to watch.
Assessment: The sharpest divergence in the corpus, because it is about the epistemic ground of every other claim. The whole cohort's cautious optimism may be an artifact of who was in the room.
Does AI adoption improve output and developer experience together, or do they decouple — with more output bought at the cost of the developers?
from: What is the future of software development in the agentic/AI age?
  • Jimmy Nilsson's Post — reflections after the Thoughtworks 'Future of Software Development' retreat (Jimmy Nilsson): Harmony: adoption is energizing and the human element flourishes ('Extreme Humanism'); reasons for optimism.
  • Why did Martin Fowler invite 60 engineers to a ski resort? (Ivett Ördög): Liberation: adoption frees engineers from CRUD toil to focus on hard problems — 'exciting times.'
  • The Future Of Software Engineering (Thoughtworks): Decoupling: organizations can gain output while developers report lower satisfaction — productivity and experience are coming apart.
  • Finding Comfort in the Uncertainty (Annie Vella): Hidden cost: the engineers going hardest on AI are being exhausted by it, and the warning signs are easy to miss.
  • Fragments: February 13 (Martin Fowler (Feb 13)): Intensification risk: teams stay the same size and 'do much more,' but doing much more is exactly what drives burnout.
Assessment: This is the specific, evidence-bearing form of the optimism/pessimism split, and it cuts directly against any simple reading of 'DevEx = AgentEx' as good news. The harmony camp (Nilsson, Ördög) speaks from felt experience and in-group mood; the decoupling camp (Thoughtworks, Vella) and Fowler's own cited HBR study bring the harder data, and it shows output and wellbeing pulling apart. Notably Fowler holds both 'same-size teams do much more' and the burnout finding without resolving them — the single clearest internal contradiction in the corpus, and a sign the harmony framing is doing more reassuring than the evidence supports.
Is preserving human understanding of the system a first-order value, or is comprehension safely offloadable so long as output is verifiable?
from: What is the future of software development in the agentic/AI age?
  • Fragments: February 4 (Martin Fowler (Feb 4)): Preserve understanding: it is a first-order value, and practices that force explanation defend it.
  • Fragments: February 9 (Martin Fowler (Feb 9)): Preserve understanding and craft: cognitive debt and the loss of model-building are the real costs to guard against.
  • Jimmy Nilsson's Post — reflections after the Thoughtworks 'Future of Software Development' retreat (Jimmy Nilsson): Preserve the human: 'Extreme Humanism' and being conspicuously human are load-bearing anchors.
  • We've Stopped Arguing About Whether AI Changes Software (Greg Herlein): Offloadable: the quality bar is 'is it verifiable enough,' not whether a human read or understood the code.
  • Looking back at the Future of Software Engineering unconference (Bartosz Ocytko): Offloadable for most builders: citizen developers need never read the code if a constrained platform makes failure modes disappear.
Assessment: The optimist-practitioner vs. skeptic-humanist axis. Fowler and Nilsson treat comprehension and craft as intrinsic goods whose loss is a harm even when output is fine; Herlein and Ocytko treat comprehension as instrumental — replaceable by verification and platform guardrails. The humanist camp has the stronger long-run argument (Fowler's cognitive-debt case and Ocytko's own 'who-guards-the-guards' problem both show verification and guardrails are themselves fallible artifacts that someone must understand), but the offloadable camp is describing what is actually happening at scale. The crux is whether verifiability can ever be robust enough to make understanding optional — which Ruby's own Goodhart gap (a passable oracle diverging from true intent) suggests it cannot fully be.

Open gaps (42)

  • missing evidenceThe entire framework is grounded in a single three-month engagement. No DORA metrics, no controlled comparison with unconstrained approaches, no data on defect rates or rework costs. The talk is honest about this, but it means the prescriptive claims rest on practitioner intuition rather than empirical evidence.
    Presentation: AI-First Software Delivery: Balancing Innovation with Proven Practices — read next: Studies measuring productivity and quality outcomes of structured vs. unstructured AI-assisted development; DORA-instrumented teams using agentic workflows.
  • implicit premiseRIPER-5's forbidden-actions model assumes that constraining the LLM to one phase at a time produces better outcomes. But the LLM may actually benefit from lookahead — considering implementation feasibility during planning, or discovering specification gaps during coding. The framework may be imposing a waterfall micro-structure on a process that benefits from iteration.
    Presentation: AI-First Software Delivery: Balancing Innovation with Proven Practices — read next: Research on exploratory programming, the role of prototyping in specification discovery, and whether phase gates help or hinder creative problem-solving in AI-assisted development.
  • tensionThe talk argues that (1) AI amplifies whatever practices you have, and (2) pair programming is essential for reviewing AI output. But if your practice is already strong enough to benefit from amplification, why is the review burden so high? The high review burden suggests the AI is generating output that doesn't match team standards — which is the amplification-of-weakness problem the talk warns about.
    Presentation: AI-First Software Delivery: Balancing Innovation with Proven Practices — read next: Research on AI code quality vs. human code quality at different experience levels; studies on code review effectiveness for AI-generated code.
  • scope limitationThe framework addresses coding agents exclusively. It does not address other AI applications in the SDLC (testing agents, deployment agents, monitoring agents, design agents) or how the supervised/unsupervised choice might differ for those. The 'shifting left' claim is broader than the framework can support.
    Presentation: AI-First Software Delivery: Balancing Innovation with Proven Practices — read next: End-to-end SDLC automation literature; agent architectures for testing, deployment, and operations.
  • incompleteBeck's individual-utility analysis ignores bequest motives. The 'billion dollars in 30 years' would still be worth something to most people because it could go to children, spouses, foundations, or causes outside one's lifespan. The essay's hard claim that 30-year money is worth 'literally one dollar in my hand today' less than $1 today only holds if one's utility function is purely selfish and lifetime-bounded.
    Extreme Time Value of Money: Late-stage Career Planning — read next: The bequest-motive literature in economics (Kotlikoff, Modigliani); empirical work on intergenerational transfers; Derek Parfit on personal identity and obligations to future selves and others
  • incompleteThe diagnosis of tech-compensation bias is sharp but the prescription is missing. Beck explicitly says he has no answer. The essay gestures at arbitrage but does not survey actual instruments — secondary equity markets (Forge, EquityZen), tender offers, equity-financing products (e.g., from Quid or ESO Fund), salary-heavy negotiations, advisory-shares structures for senior people, or simply not joining startups.
