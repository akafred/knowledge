Agent Self-Documentation

Having a coding agent produce human-reviewable artifacts of its own work — video demos, documentation, release notes — so the human's review shifts from reading code to inspecting what the agent claims it did; the inverse of agent-native design, where agents instead produce artifacts of their own work for humans.

Also known as: agent self-documentation, agentic self-documentation, agent-recorded demos, agent demos of its work, demo-driven review, self-documenting agent output, --help as embedded SKILL.md, review the docs first

  • Have your agent record video demos of its work with shot-scraper video Simon Willison (Simon Willison's Newsletter) · 2026-06-30 source ↗ Reader 🖥️
    Coding agents should record video demos of their own work, and the tooling that enables it — shot-scraper's new storyboard-driven `video` command — is itself best built by an agent and driven straight from its `--help` output, so the human's remaining job is reviewing the design rather than writing the code.

The adjacent sqlite-utils story is the counter-example the piece doesn't connect: its release-blocking data-loss bug was invisible to any demo and caught only by code review. Demos verify liveness and UX, not correctness.

The piece does not consider --help drift (help falling out of sync with behavior), how a large tool's full --help fits an agent's context budget, or when the embedded-skill approach should give way to a real SKILL.md or progressive disclosure.
