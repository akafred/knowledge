Presentation: AI-First Software Delivery: Balancing Innovation with Proven Practices

Wes Reisz · rss · 2025-01-01 source ↗ Reader 🖥️

Agentic EngineeringAgile & XPAI as AmplifierFeedback LoopsSpec-Driven DevelopmentVibe Coding
AI-first software delivery is not one-size-fits-all: teams should choose between supervised and unsupervised agentic approaches based on code longevity and their ability to verify outputs automatically, using structured human-AI collaboration (RIPER-5) to preserve engineering discipline — because AI amplifies whatever practices you already have, good or bad.

practical · conference presentation (QCon AI) with audience Q&A — practitioner report on adopting AI-assisted coding in a consultancy engagement — First-person account from a Thoughtworks technical principal describing a supervised AI-first approach on a three-month-old client project. Genre: experience report with prescriptive framework (RIPER-5). Includes live demo of building an MCP server.

Key terms

  • AI-first software delivery (AIFSD) Agentic Engineering
    An approach that shifts AI usage as far left as possible in the SDLC — not synonymous with fully autonomous agents, but a spectrum from supervised to unsupervised depending on context.
  • RIPER-5 Agentic Engineering
    A five-phase protocol (Research, Innovate, Plan, Execute, Review) that puts the LLM into the same mental mode as the developer, with explicit forbidden actions per phase to prevent the LLM from jumping ahead.
  • supervised vs. unsupervised agents Agentic Engineering
    Supervised: developer actively pairs with and reviews agent output at each step. Unsupervised: agent operates autonomously with automated guardrails. The choice depends on domain knowledge and verification capability.
  • code longevity
    How long the generated code will live — from throwaway explorations to long-lived production systems. Higher longevity demands more rigor.
  • degree of automated verification Feedback Loops
    The team's ability to automatically verify that generated code is correct, which depends heavily on domain knowledge and available test infrastructure.
  • spec-driven development Spec-Driven Development
    Using Markdown specifications as the contract between developer and LLM, with three variants: spec-first (generate then discard), spec-anchored (keep and reference), spec-as-source (code is byproduct).
  • sensible defaults Agile & XP
    Thoughtworks' baseline engineering practices (CI, TDD, pair programming, trunk-based development, continuous delivery) that every engagement starts with before adapting to client context.
  • vibe coding Vibe Coding
    Exploratory, throwaway coding where speed of learning matters more than code quality — valid for POCs and information gathering, not for production delivery.

Arguments

Teams should use a supervised agentic approach when they lack domain knowledge and automated verification capabilities, and progressively shift toward unsupervised as those grow. strength: medium
Explicit premises:
  • Three months into a new client engagement, the team knows too little about the domain to safely let agents operate unsupervised.
  • Unsupervised agents require the ability to evaluate their outputs automatically, which requires domain understanding.
  • The team is already beginning to shift rightward as domain knowledge accumulates.
  • 95% of AI projects fail to deliver ROI, suggesting many organizations misapply autonomous approaches.
Implicit premises:
  • The cost of agent errors in production is high enough that unsupervised operation without verification is irresponsible.
  • Domain knowledge acquisition is inherently time-consuming and cannot be short-circuited by the AI itself.
Inferences:
  • Since verification depends on domain knowledge, and domain knowledge takes time to build, the right approach at any given moment is determined by where you are on that learning curve.
RIPER-5 produces better outcomes than unconstrained LLM interaction because it prevents premature solution generation. strength: medium
Explicit premises:
  • LLMs, unconstrained, produce over-abstracted solutions when the developer just wants to explore.
  • Forbidding the LLM from coding during research/innovate/plan phases keeps it aligned with the developer's current intent.
  • The framework mirrors how experienced engineers already work, which validates its structure.
  • Review phase catches drift between plan and implementation.
Implicit premises:
  • LLMs are stateless reasoning systems that benefit from external state management (the mode protocol) imposed by the human.
  • The cognitive overhead of enforcing modes is lower than the rework cost of unconstrained generation.
Inferences:
  • Explicit mode-setting compensates for the LLM's lack of meta-awareness about where in the development lifecycle the developer currently is.
Engineering discipline (CI, TDD, pairing, trunk-based development) is more important in AI-first delivery than in traditional delivery. strength: high
Explicit premises:
  • AI can generate code much faster than humans can review it.
  • Architecture boundaries can leak when defects scale faster than review.
  • If foundations are weak, AI amplifies the weaknesses.
  • Pair programming helps manage the review burden of AI-generated code.
Implicit premises:
  • The rate of code generation has outpaced the rate of quality assurance improvement, creating a growing gap that only discipline can close.
  • Most organizations' engineering foundations are not as strong as they believe.
Inferences:
  • The amplification property of AI means that the marginal cost of weak discipline is now higher than it was pre-AI, making investment in discipline more urgent.

Narratives

  • Reisz describes being on a new Thoughtworks client engagement building a knowledge graph for a large U.S. state, three months in, constantly asked why he isn't using fully autonomous agents.
    → Establishes the real-world tension that motivates the talk: the pressure to adopt maximally autonomous AI versus the practitioner's judgment that context matters. Makes the framework feel earned rather than theoretical.
  • During a client conversation, Reisz quickly vibe-codes a prototype while the product person talks, then shows it to the client who says 'No, not at all. That's completely wrong. Please don't ever do that again.' — and Reisz considers this a success.
    → Rehabilitates vibe coding as a legitimate information-gathering technique by reframing 'wrong' output as valuable learning. Illustrates the exploratory quadrant of the two-by-two.
  • The MCP server demo: converting a single-shot QCon question-answering tool into a multi-turn MCP server using the full RIPER-5 workflow, resulting in 5-6 files of ~50 lines each after extensive specification.
    → Makes the 10:1 spec-to-code ratio tangible and demonstrates that the framework is not just theory — it produces working software through a disciplined process.
  • Nate Schutta, a Thoughtworks colleague, sees RIPER-5 and says: 'I don't know that it has anything to do with AI. That's how I work anyway.'
    → Validates RIPER-5 by showing it codifies existing expert practice rather than imposing alien process. Makes adoption feel natural rather than burdensome.
  • The $100-bill visualization: $1.3 trillion in $100 bills laid end to end would circle the earth 50 times, yet 95% of AI projects fail to deliver ROI.
    → Creates visceral scale contrast to motivate the need for a more thoughtful approach to AI adoption.

Questions addressed

  • ✅ Why not use a fully autonomous multi-agentic approach for software delivery?
    Because the choice depends on domain knowledge and verification capabilities. Without sufficient domain understanding, unsupervised agents cannot be safely verified, so supervised approaches are appropriate.
  • ✅ How should teams decide between supervised and unsupervised agentic workflows?
    Use the two-by-two model: map code longevity (y-axis) against degree of automated verification (x-axis). Choose the quadrant that fits your current context.
  • ✅ How do you prevent LLMs from generating over-abstracted or premature solutions?
    Use RIPER-5: put the LLM into explicit modes with forbidden actions per phase so it cannot jump ahead to code generation before research, innovation, and planning are complete.
  • ⭕ Does AI-first delivery actually improve productivity metrics?
    Reisz admits he does not yet have DORA metrics for his current project. Developers 'feel faster' but he acknowledges that feeling can be wrong. A similar project elsewhere has instrumented DORA metrics showing productivity gains while holding MTTR constant.
  • ⭕ Does RIPER-5 work for large, existing (brownfield) codebases?
    Reisz has not applied RIPER-5 specifically to large brownfield codebases but sees no reason it wouldn't work, since the framework mirrors how engineers already approach such work.
  • ⭕ How do you handle the boredom/wait-time problem of supervised AI workflows?
    Partially addressed: the review stage can shortcut some iteration; agent mode allows more flow-like interaction; parallelizing tasks across pairs is the eventual goal. No fully satisfying answer given.
  • ✅ Is pair programming still necessary when the LLM is your 'pair'?
    Yes — the LLM is not a pair but a tool that generates average-of-training-data output. Human pairs are needed because reviewing AI-generated code is exhausting for a single developer.

Gaps

  • missing evidenceThe entire framework is grounded in a single three-month engagement. No DORA metrics, no controlled comparison with unconstrained approaches, no data on defect rates or rework costs. The talk is honest about this, but it means the prescriptive claims rest on practitioner intuition rather than empirical evidence.
    Read next: Studies measuring productivity and quality outcomes of structured vs. unstructured AI-assisted development; DORA-instrumented teams using agentic workflows.
  • unasked questionThe talk never asks whether the 10:1 spec-to-code ratio is efficient or merely shifts the bottleneck from writing code to writing specifications. If specifications themselves become the primary artifact, what are the failure modes of spec-driven development? Can specifications be wrong in ways that are harder to detect than buggy code?
    Read next: Literature on specification errors in formal methods; research on whether natural-language specifications introduce ambiguity that undermines the 'contract' metaphor.
    closed by[FULL WORKSHOP] AI Coding For Real Engineers - Matt Pocock, AI Hero — Pocock addresses the spec-bottleneck question directly: optimizing the PRD beyond the alignment point yields little (p28), specs-to-code without reading code fails (p5), and in-repo spec documents rot and poison later agent sessions (p17).
  • implicit premiseRIPER-5's forbidden-actions model assumes that constraining the LLM to one phase at a time produces better outcomes. But the LLM may actually benefit from lookahead — considering implementation feasibility during planning, or discovering specification gaps during coding. The framework may be imposing a waterfall micro-structure on a process that benefits from iteration.
    Read next: Research on exploratory programming, the role of prototyping in specification discovery, and whether phase gates help or hinder creative problem-solving in AI-assisted development.
  • tensionThe talk argues that (1) AI amplifies whatever practices you have, and (2) pair programming is essential for reviewing AI output. But if your practice is already strong enough to benefit from amplification, why is the review burden so high? The high review burden suggests the AI is generating output that doesn't match team standards — which is the amplification-of-weakness problem the talk warns about.
    Read next: Research on AI code quality vs. human code quality at different experience levels; studies on code review effectiveness for AI-generated code.
  • scope limitationThe framework addresses coding agents exclusively. It does not address other AI applications in the SDLC (testing agents, deployment agents, monitoring agents, design agents) or how the supervised/unsupervised choice might differ for those. The 'shifting left' claim is broader than the framework can support.
    Read next: End-to-end SDLC automation literature; agent architectures for testing, deployment, and operations.
  • unasked questionThe two-by-two model places 'degree of automated verification' on the x-axis but never asks who builds the verification infrastructure. If the team is already stretched doing 10:1 spec work and exhausting code review, when do they build the automated verification that enables the rightward shift to unsupervised agents?
    Read next: Research on testing infrastructure investment patterns; the bootstrapping problem in test automation for new domains.
    closed byHarness engineering for coding agent users — Answers who builds the verification infrastructure: coding agent users themselves build an outer harness as an ongoing engineering practice, distributing computational and inferential controls across the lifecycle by cost and criticality.
