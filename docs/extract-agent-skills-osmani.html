Agent Skills

Addy Osmani · article · 2026-05-09 source ↗ Reader 🖥️

Agent HarnessAgent SkillsCoding AgentsProgressive DisclosureScope Discipline
AI coding agents default to the shortest path to 'done' and systematically skip the invisible senior-engineer work (specs, tests, reviews, scope discipline) that separates reliable software from code that breaks — skills are portable, workflow-shaped markdown files that bolt that scaffolding back on by making it impossible for the agent to rationalize skipping it.

practical · expository/prescriptive — A practitioner manifesto and design rationale for a specific open-source project (agent-skills). Combines exposition of the design decisions with prescriptive advice on applying them beyond the project itself.

Key terms

  • skill Agent Skills
    A markdown file with frontmatter containing a workflow — a sequence of steps with checkpoints and exit criteria — that gets injected into an agent's context when relevant; explicitly NOT reference documentation or an essay
  • process over prose
    The design principle that agent instructions must be structured as actionable workflows with verifiable checkpoints, not as expository essays about best practices — because agents will read prose and still skip the actual work
  • anti-rationalization table Scope Discipline
    A table embedded in each skill that pre-pairs common excuses for skipping a workflow step with written rebuttals, designed to defeat the LLM's ability to generate plausible justifications for shortcuts
  • progressive disclosure Progressive Disclosure
    Loading only the skills relevant to the current task phase into context rather than all skills at once, managed by a meta-skill router, to avoid degrading model performance with irrelevant tokens
  • scope discipline Scope Discipline
    The non-negotiable constraint that an agent must touch only what it was asked to touch — no opportunistic refactoring, no TODO-driven rewrites, no adjacent-system changes
  • senior-engineer scaffolding
    The full set of invisible work practices (surfacing assumptions, writing specs, sizing changes, leaving evidence of correctness, refusing to merge unreviewed code) that distinguish reliable engineering from code-pushing
  • harness Agent Harness
    The complete scaffolding around an AI model — prompts, tools, hooks, context policies, skills, session logs — that determines agent behavior; skills are one layer of this

Arguments

Skills (workflow-shaped markdown files) are necessary to make agents produce reliable software
Implicit premises:
  • Reliable software requires the invisible work, not just the code
  • Agents cannot discover the need for invisible work through the reward signal alone
Anti-rationalization tables are the most important design innovation for constraining agent behavior
Implicit premises:
  • The rationalization failure mode is harder to guard against than simple omission because it produces convincing justification
  • Static rebuttals are sufficient to overcome dynamic rationalization in current models
Progressive disclosure is required for skill-based systems at scale
Implicit premises:
  • The context window is a scarce resource with diminishing returns
  • A small router skill can reliably classify tasks into the correct subset
The principles behind agent skills are equally applicable to human engineering teams
Implicit premises:
  • The failure modes of agents and tired/rushed engineers are structurally identical

Narratives

  • The agent-as-unsupervised-junior narrative: AI agents are extremely capable but lack the instincts for invisible work, just like junior engineers who haven't yet learned that shipping code without specs/tests/reviews creates incidents.
    → Frames the entire piece and makes the problem emotionally legible to senior engineers
  • Google engineering culture as portable wisdom: documented practices (Hyrum's Law, Beyoncé Rule, 100-line PRs, Chesterton's Fence) are proven solutions that agents simply fail to apply despite having read about them in training data.
    → Legitimizes the skill approach by grounding it in established, high-status engineering practice

Questions addressed

  • ✅ Why do AI coding agents produce unreliable software by default?
    They optimize for 'task complete' and skip the invisible senior-engineer work (specs, tests, reviews, scope discipline) because their reward signal doesn't include it.
  • ✅ What is the right unit of instruction for constraining agent behavior?
    Workflows with steps and exit criteria — not essays or reference documentation — because only workflows are agent-actionable.
  • ✅ How do you prevent an LLM from rationalizing its way out of process requirements?
    Anti-rationalization tables: pre-written rebuttals paired with common excuses, embedded in each skill.
  • ✅ How do you scale a library of 20+ skills without poisoning the agent's context?
    Progressive disclosure via a meta-skill router that activates only the relevant skills for the current phase.
  • ✅ What existing engineering practices should agents be forced to follow?
    Google's published practices: Hyrum's Law, test pyramid, Beyoncé Rule, 100-line PRs, Chesterton's Fence, trunk-based dev, shift-left, code-as-liability.
  • ⭕ How reliable is the meta-skill router at classifying tasks correctly?
  • ⭕ What is the empirical impact of skills on defect rates or PR quality?

Gaps

  • evidentiaryNo empirical evidence for skill effectiveness. The article claims 26K stars as social proof but provides no data on defect rates, PR merge rates, or incident reduction with vs. without skills.
  • structuralThe meta-skill router is asserted but not explained. How does the 'using-agent-skills' router actually decide which skills to activate? What happens when it misclassifies a task? The reliability of the entire system depends on this component.
  • argumentativeAnti-rationalization tables assume static excuses. The article treats rationalization as a finite, enumerable set. But LLMs can generate novel justifications not covered by any pre-written table. No discussion of what happens when the agent rationalizes in a way not anticipated by the table.
  • practicalNo discussion of the cost of false enforcement. Skills add friction. What happens when the skill system forces an agent to write a spec for a one-line typo fix? The article mentions that 'a small bug fix might use three' skills but doesn't address the overhead/accuracy tradeoff.
  • evidentiaryPortability is claimed but not demonstrated. The article asserts the same SKILL.md works across Claude Code, Cursor, Gemini CLI, Codex, etc. but provides no evidence that different harnesses interpret the same markdown workflow identically.
