Have your agent record video demos of its work with shot-scraper video

Coding agents should record video demos of their own work, and the tooling that enables it — shot-scraper's new storyboard-driven `video` command — is itself best built by an agent and driven straight from its `--help` output, so the human's remaining job is reviewing the design rather than writing the code.

Key terms

  • shot-scraper video / storyboard.yml
    A new shot-scraper command that takes a YAML 'storyboard' (server to launch, URL, viewport, scenes of click/fill/wait steps) and uses Playwright's screencast to record a controlled video of a web app being exercised.
  • agentic self-documentation (agent-recorded demos) Agent Self-Documentation
    The practice of having a coding agent produce reviewable artifacts of its own work — here a video demo — so a human can see what it built.
  • '--help as embedded SKILL.md'
    The pattern where a command's `--help` output carries enough usage detail that a coding agent can drive the tool from it alone — functioning 'like bundling a SKILL.md file directly inside the tool.'
  • cross-family review Multi-Agent Orchestration
    Habitually having one lab's best model review another's output (Anthropic's model reviews OpenAI's work and vice versa) because differing blind spots surface real issues.
  • review-the-documentation-first
    Reviewing an agent's documentation edits before its code as the fastest way to build a mental model of what a large agent-produced change actually did.

Arguments

Building tools that a coding agent can drive end-to-end from its --help output, so the agent can record demos of its own work, is a worthwhile and now-practical pattern for agentic engineering. strength: medium
Explicit premises:
  • shot-scraper video can record a controlled Playwright video of a web app from a declarative storyboard.
  • GPT-5.5 xhigh authored the entire storyboard and drove the recording from a three-line prompt, using only the command's --help.
  • The --help output worked like a SKILL.md bundled inside the tool.
  • The feature itself (code, docs, format) was written by the agent, with the human reviewing the design via its documentation.
Implicit premises:
  • A human can adequately catch design problems by reviewing an agent's documentation and watching a demo video — this reviewing capacity is the load-bearing assumption of the whole loop.
  • A scripted demo that runs to completion is taken as evidence the feature works, even though the storyboard exercises a predetermined happy path (and even stubs navigator.clipboard via injected JS) — so it demonstrates 'the flow runs as scripted', not 'the feature is correct'.
  • The command's --help will stay accurate, complete and token-cheap enough to remain a reliable agent instruction as the tool evolves.
  • The recorded artifact reflects real application behavior because Playwright drives the actual app, not a mock (mostly true here, with the clipboard exception).
Inferences:
  • From 'an agent drove a real feature build and demo from --help alone' to 'agent-drivable tooling is a generally good pattern' — an existence proof generalized to a design principle.
Cross-family model review should be a habitual step in agent-assisted software work. strength: medium
Explicit premises:
  • Having Anthropic's model review OpenAI's work and vice versa has turned up valuable results often enough to be worth it.
  • In the sqlite-utils RC, a GPT-5.5 review found two genuine P1 transaction bugs after Fable's own passes.
  • Fable's earlier code review caught five release blockers, including a data-loss delete_where bug, that no demo would surface.
Implicit premises:
  • Different model families have sufficiently different blind spots that one can catch the other's misses.
  • The reviewing model is competent enough to surface real issues without generating so many false positives that the review becomes noise.
Inferences:
  • From repeated positive anecdotes to a standing recommendation — inductive generalization from Willison's own practice.

Narratives

  • GPT-5.5 xhigh, running in Codex Desktop, is given a three-line prompt: review the branch, run `shot-scraper video --help`, then record a demo of the new bulk-insert/create-table features against a Datasette dev server it spins up itself. It produces the entire multi-scene storyboard.yml and the finished video with no human-written configuration.
    → The load-bearing existence proof for the whole piece: it demonstrates simultaneously that an agent can drive a demo pipeline end-to-end and that a tool's --help alone can serve as its agent instructions ('--help as embedded SKILL.md').
  • Getting sqlite-utils to a stable 4.0: Claude Fable's pre-ship review flags 5 release blockers (worst: delete_where never commits and poisons the connection, causing silent data loss); over 37 prompts / 34 commits / 30 files the issues are worked through; a GPT-5.5 cross-family review then finds 2 more P1 transaction bugs; Willison reviews the docs first, then the PR, occasionally prompting from his phone during the July 4th parade.
    → Complements and quietly qualifies the demo thesis: the release-critical bugs were correctness/edge-case failures invisible to any demo and catchable only by code and cross-family review; also illustrates that hard agent tasks free the human's attention for other things.
  • Willison runs AgentsView inside his own Claude Code session to compute what the sqlite-utils work would have cost unsubsidized: $149.25 total, dominated by a single $141.02 main session, with sub-agent reviews at $1.40-$2.40 each — and concludes he should have leaned more on cheaper-model subagents.
    → Grounds the abstract 'agents do the work' story in real unit economics and motivates the cheaper-model-delegation proposition.

Questions addressed

  • ✅ How can a coding agent show a human what it actually built?
    Have it record a video demo of the running application via shot-scraper video, driven by a declarative storyboard the agent itself writes.
  • ✅ How can an agent learn to use a brand-new tool without a separately authored skill file?
    Design the tool so its --help output carries enough usage detail to serve as an embedded SKILL.md that the agent reads and follows.
  • ✅ Where does the human add value when the agent writes the code?
    In design judgment exercised through review — reviewing the agent's documentation first to understand the change, watching demos, reviewing the PR, and directing cross-family review — rather than in producing code.
  • ✅ Is having one AI model review another's work actually useful, or just superstition?
    It reliably surfaces real issues; cross-family review (one lab's model checking another's) is worth doing habitually.

Gaps

  • incompleteThe piece advocates video demos as the way to show an agent's work, but never addresses that a scripted storyboard exercises a predetermined happy path — it even stubs navigator.clipboard via injected JavaScript — so it demonstrates 'the flow runs as scripted', not 'the feature is correct'. The adjacent sqlite-utils story is the counter-example the piece doesn't connect: its release-blocking data-loss bug was invisible to any demo and caught only by code review. Demos verify liveness and UX, not correctness.
    Read next: Read on evals and property-based testing as complements to demos, and Geoffrey Litt's 'understand to participate' (quoted later in this same newsletter) on why watching output cannot substitute for conceptual understanding of the change.
  • incompleteThe '--help as embedded SKILL.md' pattern assumes the --help text stays accurate, complete and token-cheap as the tool grows. The piece does not consider --help drift (help falling out of sync with behavior), how a large tool's full --help fits an agent's context budget, or when the embedded-skill approach should give way to a real SKILL.md or progressive disclosure.
    Read next: Agent-native design and progressive-disclosure/token-efficiency literature; compare with SKILL.md ecosystem write-ups on versioning and staleness.
  • incompleteThe whole reliability argument rests on the human catching design flaws by reviewing agent-written documentation, yet the piece does not examine the case where the agent's prose is as confidently wrong as its code — doc rot, plausible-but-inaccurate documentation that reads clean. Release notes are declared 'safe to outsource because boring, predictable and accurate', but boring text is also the text least likely to be scrutinized.
    Read next: Cognitive-debt and comprehension-debt writing; work on documentation as a review surface and on LLM-generated-text 'tells' and silent inaccuracy.
  • incompleteCross-family review is recommended from repeated anecdote ('it really does work'), with no false-positive rate, no cost/benefit accounting, and no discussion of when it fails to catch things or produces noise that costs more review time than it saves.
    Read next: See the GitHub Copilot 'Rubber Duck' analysis of cross-family review as cost-effective only when invoked sparingly at high-signal moments, and eval methodology for measuring reviewer models.
