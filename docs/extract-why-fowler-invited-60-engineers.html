The turn to AI-built software has reopened rather than resolved the foundational questions of the craft — should humans still read code, how do we know it works, does code quality still matter, and how do we manage newly-empowered citizen developers — and the author's connective throughline is an optimistic one: AI democratizes building while the disciplines that make code good for humans remain exactly the ones that make it good for AI.

mixed · mixed — A first-person reflection on a ThoughtWorks 'Future of Software Engineering' retreat, run under the Chatham House Rule; it reports the debates (narrative/expository) while arguing the author's own firm positions (argumentative), and its questions are practical questions of how to build software mixed with conceptual questions about what a 'developer' and 'good code' now are.

Key terms

  • dark factory Dark Factory (Fully Automated Software Pipeline)
    A fully-automated software operation, by analogy to a lights-off factory: systems written and maintained entirely by AI with no human ever reading the code
  • Citizen Developers Citizen Developer
    Non-professional people whom agentic tools now empower to build their own software, lowering the bar for who counts as a developer
  • agents as compilers vs agents as engineers
    The framing that adjudicates the dark-factory debate: whether an agent regenerates code from scratch whenever needed (compiler) or maintains an existing codebase and moulds it over time (engineer)
  • complacency
    Accepting incorrect AI results as truth — a risk of empowered non-experts that the author rates as potentially as damaging as security failures
  • specification with examples that double as tests
    The author's prescribed practice: a spec should contain concrete examples that also serve directly as automated tests, so the spec and the test suite are one artifact

Arguments

Code that is good for AI agents and code that is good for humans are the same thing. strength: medium
Explicit premises:
  • AIs are trained on human-readable code, so human-readable code is what they understand best.
  • AIs have a short attention span, so any strategy that helps short-attention-span humans also helps AI agents.
  • In the author's Habit Hooks experience, keeping code human-maintainable let the agent make more independent progress.
Implicit premises:
  • 'Good for humans' is best operationalized as readability and maintainability.
  • What a model handles best is determined by the distribution of code it was trained on.
  • The relevant respects in which AI and human cognition differ (attention span, comprehension) point the same direction for code design.
Inferences:
  • Therefore optimizing code for human maintainability simultaneously optimizes it for AI agents, so quality practices do not need to bifurcate.
The reduction in software-engineer hiring should be read optimistically as democratization, not pessimistically as declining demand. strength: low
Explicit premises:
  • In the mid-2010s Robert C. Martin estimated the number of developers was doubling roughly every 5 years, which extrapolated absurdly (everyone a developer by 2050).
  • Agentic tools have dramatically lowered the bar for someone to be considered a software developer.
  • People can now build simple useful tools themselves instead of waiting for professional engineers.
Implicit premises:
  • Overall demand for software is not shrinking, only the mix of who builds it.
  • 'Developer' is being redefined to include far more people.
Inferences:
  • Therefore fewer professional-engineer hires reflects a redistribution of who builds software rather than the profession becoming less needed.
Testing is more important than ever in the agentic era. strength: high
Explicit premises:
  • AI agents produce code at a rate that is impossible to review.
  • Human review is woefully inadequate at catching bugs.
  • The industry was never good at producing bug-free software, and the last year made it an order of magnitude worse.
Implicit premises:
  • Automated tests can catch classes of defects that human review cannot keep up with or reliably detect.
Inferences:
  • Since the traditional safeguard (human review) cannot scale to AI output volume or reliably catch bugs, the burden must shift to automated testing.

Narratives

  • The retreat is staged as a nod to a ski-resort gathering a quarter of a century ago that produced a famous manifesto; this time, though, the group deliberately left with interesting questions rather than another manifesto, and proceedings ran under the Chatham House Rule so ideas could be shared without names.
    → Frames the piece's significance and resets expectations — the point is unsettled questions, not a declaration — and licenses the author to report positions anonymously.
  • Habit Hooks, a tool the author built to keep code human-maintainable, ironically nudged them toward the dark-factory camp: because the agent keeps the code maintainable, it makes more progress independently and needs far less of the author's direct involvement.
    → Personal evidence that good-for-humans practices are precisely what let agents operate more autonomously — bridging the code-quality argument and the dark-factory question.
  • The author ended up writing their own video-editing tool and their own teleprompter software because the commercial options kept crashing to the point of being unusable, and notes GitHub currently struggles to claim even one nine of availability.
    → Illustrates the deteriorating reliability of shipped software that motivates the 'does this software work?' section.
  • Several participants told stories of colleagues in finance and customer service who accidentally exposed sensitive data or made poor decisions on the back of AI results they trusted too readily.
    → Concrete, non-theoretical evidence for the citizen-developer risks (security and complacency) that ground the section's open question.

Questions addressed

  • ⭕ Should software be built in 'dark factories' — written and maintained entirely by AI with no human ever reading the code?
    No consensus; this was the largest divide. The author's synthesis is that viability depends heavily on context — genuinely viable for transactional backends and COBOL rewrites — and the crux is whether agents behave like compilers or like engineers.
  • ✅ What kinds of tests should we write, and can a detailed enough specification substitute for tests entirely?
    The author's firm position: the specification should include examples that do double duty as automated tests, so a spec does not replace tests but carries them.
  • ✅ Does code quality still matter, and is 'code that is easy to change' the same for humans and AIs?
    Yes and yes — the author argues code good for AI agents and code good for humans are the same thing, so quality still matters.
  • ⭕ How do we manage the risk of newly-empowered citizen developers?
    The author welcomes the democratization but names security and complacency as serious risks and explicitly calls how to manage the tension a genuinely open question.

Gaps

  • incompleteThe claim that AI-built systems are 'genuinely viable' for transactional backends and COBOL rewrites is asserted from others' reported experience without stating the criteria that mark a domain as suitable for full automation versus one that needs a human-maintained, legible codebase.
    Read next: Look for case studies and taxonomies of where 100%-AI-maintained systems succeed or fail by domain, coupling, and change-rate.
  • illogicalThe argument that reduced engineer hiring reflects democratization (optimistic read) rather than declining demand (pessimistic read) rests on Robert C. Martin's developer-doubling extrapolation, which is rhetorical and does not actually adjudicate between the two readings — the same hiring data is consistent with both.
    Read next: Read labor-market analyses that separate 'demand for software' from 'demand for professional engineers' with evidence rather than extrapolation.
  • incompleteThe citizen-developer risks (security exposure and complacency) are vividly named but left entirely without proposed mitigation — the author declares the tension 'a genuinely open question' and offers no governance, guardrail, or review mechanism.
    Read next: Look for governance patterns for non-expert AI builders: access controls, human-in-the-loop checks, and complacency countermeasures.
  • uninformedThe 'AIs have a short attention span' premise carries much of the weight of the good-for-AI-equals-good-for-humans argument but is asserted as intuition rather than substantiated, and no case is examined where AI-usable code might diverge from human-readable code.
    Read next: Seek empirical work on how context length, code structure, and readability affect agent task success, and on whether agent-optimal code ever differs from human-optimal code.
