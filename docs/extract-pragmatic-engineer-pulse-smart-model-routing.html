The Pulse: a new trend, smart model routing

The Pragmatic Engineer (Gergely Orosz) · email · 2026-07-03 source 🖥️

AI Unit EconomicsModel RoutingOpen-Weight Models
A new category of 'intelligent' model-routing tools is emerging to let engineering organizations cut AI spend by automatically directing each task to the cheapest model capable of handling it, and — because cheaper open models are now considered sufficient for a large share (~60%) of coding work — this routing capability is on track to become table stakes across all AI vendors.

practical · mixed — Despite the 'The Pulse' framing, this document is NOT a multi-item newsletter — it is a free bonus excerpt covering exactly one of four topics from a prior paid Pulse issue. It is a survey (expository) of the smart-model-routing vendor landscape, wrapped in an argumentative forecast that routing will become table stakes. Classed practical because the whole framing is instrumental: helping engineering leaders act on AI-cost control by adopting a routing layer.

Key terms

  • intelligent router / smart model routing Model Routing
    A layer or tool that automatically selects the right (typically cheapest-capable) LLM for each task, prompt, or session in order to optimize cost while preserving output quality
  • AI gateway Model Routing
    A proxy layer through which a workplace funnels its LLM API calls; increasingly ships with routing that can pick models by cost, latency, and availability
  • open models / hosted open models Open-Weight Models
    Open-weight models (as opposed to proprietary frontier models), claimed to be sufficient for roughly 60% of coding work by token spend and to be strictly rising in usage
  • table stakes
    A baseline feature every vendor will be expected to have; the forecast endpoint for routing

Arguments

Intelligent model routing will become table stakes across the AI tooling market. strength: medium
Explicit premises:
  • Enterprise demand is 'off the charts' (Grinberg has met practically every bank CEO).
  • Token-price variance is large (10-20x), so the cost incentive is strong.
  • Many vendors and gateways already ship routing.
  • Open models are sufficient for ~60% of coding work, so most spend can be shifted cheaply.
Implicit premises:
  • Features that deliver obvious cost savings and are strongly demanded tend to commoditize into baseline expectations.
  • A capability many vendors can build converges to a table-stakes feature rather than a durable differentiator.
  • Routing to the cheapest capable model preserves output quality — i.e., 'good enough' is reliably detectable per task (the load-bearing hidden assumption).
Inferences:
  • If routing commoditizes and open models cover 60% of spend, frontier-model providers face volume/margin pressure on the low-complexity tier.
  • Differentiation shifts from raw model quality to routing quality and evaluation.
  • Standalone routing vendors risk being squeezed once IDEs and gateways all build routing in-house.
Enterprises can cut AI spend substantially without sacrificing code quality by adopting intelligent routing. strength: medium
Explicit premises:
  • Vendors claim 20-30% cost savings.
  • Open models are estimated sufficient for ~60% of coding work.
  • Banks want a layer to control spend 'while still generating high-quality code'.
Implicit premises:
  • Routers can reliably classify task difficulty and match it to model capability.
  • The 'sufficient' threshold is measurable and trustworthy.
  • Vendor-sourced savings figures are representative of real deployments.
Inferences:
  • The real bottleneck becomes trustworthy evaluation of 'is this model good enough for this task'.
  • Without good evals, routing risks silent quality regressions that are invisible until they cause harm.

Questions addressed

  • ✅ Do intelligent model routers that pick the right model for the right task actually exist yet?
    Yes — the piece catalogs dedicated routers, gateways with routing, and IDE 'Auto' modes.
  • ✅ How much money can routing save, and for how much of the work can cheaper/open models substitute?
    Partially — vendors claim 20-30% savings and ~60% of coding work can use open models, but these figures are vendor-sourced and not independently verified.
  • ✅ Will intelligent routing become a standard feature?
    The author forecasts yes — routing will become table stakes.
  • ⭕ How do you verify that a router actually picked a 'good enough' model — i.e., how do you guard against silent quality regressions without a reliable evaluation harness?
  • ⭕ What are the data-egress, IP, and compliance implications of routing enterprise (e.g. bank) code and prompts to cheaper, open, or third-party-hosted models?

Gaps

  • incompleteEvery quantitative claim (20-30% savings, 60% of work sufficient for open models) is sourced from vendors or a vendor CEO with a direct commercial interest; there is no independent benchmark of either the cost savings or, crucially, the routing quality.
    Read next: Independent routing benchmarks and methodology — e.g., the RouteLLM paper, Not Diamond's / LMSYS routing evaluations, and cost-vs-quality studies that measure whether the 'cheapest capable' choice actually holds up on hard tasks.
  • incompleteThe load-bearing assumption — that 'good enough' can be reliably detected per task so that downgrading to a cheaper model preserves quality — is never examined. The hard part of any routing system is the evaluation of task difficulty and output adequacy, which the piece treats as solved.
    Read next: LLM-eval literature and task-difficulty classification; how routers estimate confidence/complexity; the adjacent argument (e.g. Elena Verna's 'AI Confidence Theater') that the real work is the ongoing eval/iteration, not the first prompt.
  • incompleteBanks are named as the marquee buyers, yet the governance/security dimension is entirely absent: routing sensitive enterprise code and prompts to cheaper/open/third-party-hosted models raises data-residency, IP-leakage, and compliance questions that go unmentioned — a genuine blind spot given the audience.
    Read next: Enterprise AI governance and data-egress controls; self-hosting open-weight models behind the firewall; regulatory constraints on where model inference may run (especially relevant to Nordic/EU public-sector and critical-infrastructure contexts).
  • incompleteThe incentive-conflict problem is flagged only for Cursor but never generalized: any router operated by a party that profits from token usage or from steering to a preferred model is a principal-agent problem, and the reader is left to notice this alone.
    Read next: Frame routing as a two-sided market / principal-agent design problem; evaluate router neutrality (bring-your-own-keys, router-only services like Kilo) versus vendor-locked routers.
  • incompleteThe '60% sufficient by token spend' figure conflates volume with value: 60% of token spend is not 60% of business value, and the hardest 40% may be exactly where frontier-model quality matters most, so the savings story may understate the risk concentrated in high-stakes tasks.
    Read next: Distribution of task criticality and the marginal value of frontier models on the hardest tasks; portfolio thinking about where to spend the expensive tokens rather than average-cost reasoning.
