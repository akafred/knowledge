Your AGENTS.md might be a big problem

Matt Pocock (AI Hero) · email · 2026-07-03 Reader 🖥️

Agent Context Files (CLAUDE.md / AGENTS.md)Coding AgentsContext EngineeringProgressive Disclosure
A reactively bloated AGENTS.md degrades an AI coding agent because it spends the model's finite instruction-following budget on every request, so you should keep the root file to 3-5 essentials and push everything else into nested files the agent loads only when relevant.

Key terms

  • instruction budget Progressive Disclosure
    The finite number of directives (~150-200) a frontier LLM can reliably follow before instruction-following degrades; treated as a scarce resource every rule draws down.
  • progressive disclosure Progressive Disclosure
    A config-architecture pattern where the root AGENTS.md stays minimal and detailed rules live in separate nested files the agent pulls in only when a task needs them.

Arguments

Keep the root AGENTS.md ruthlessly small (3-5 items) and move everything else into nested, on-demand files (progressive disclosure). strength: medium
Explicit premises:
  • LLMs have a limited instruction budget (~150-200 instructions) beyond which they lose focus.
  • Every rule in AGENTS.md is spent from that budget on every single request, relevant or not.
  • Bloated AGENTS.md files therefore hurt performance and waste tokens.
Implicit premises:
  • The whole root AGENTS.md is injected into context on each request (so there is no per-request filtering of its contents).
  • Nested files are retrieved only when relevant, so relocating rules there actually frees budget rather than just hiding it.
  • Instruction-following degradation scales with instruction COUNT, roughly independent of each rule's individual quality.
Inferences:
  • If instruction budget is the true constraint, then ALL persistent context — system prompt, tool definitions, memory, other config — competes for the same budget, so the optimization should be holistic, not AGENTS.md-only.
  • The same reactive rule-accumulation dynamic will simply migrate into the nested files unless the underlying authoring discipline changes; progressive disclosure treats a symptom of doc governance, not its cause.

Narratives

  • A developer who adds one rule every time the coding agent misbehaves, and months later is left with a sprawling AGENTS.md that quietly hurts more than it helps.

Questions addressed

  • ✅ Why might my AGENTS.md be making my coding agent worse rather than better?
    Because it has grown reactively and every rule spends the model's finite instruction budget on every request, diluting focus.
  • ✅ How should I structure AGENTS.md instead?
    Minimal root (3-5 items) plus progressive disclosure into nested on-demand files — though the concrete criteria live in the linked article, not the email.

Gaps

  • incompleteThe '150-200 instructions' ceiling is asserted with no citation and no definition of what counts as one 'instruction' (a bullet? a sentence? a constraint?), so it can't be checked or applied precisely.
    Read next: Anthropic/OpenAI docs and papers on instruction-following degradation with prompt length; the 'context rot' / lost-in-the-middle literature quantifying how many constraints models reliably honor.
  • incompleteThe whole solution depends on agents reliably discovering and loading nested AGENTS.md files only when relevant, but the email never explains that retrieval mechanism — if the agent loads all nested files anyway, progressive disclosure saves nothing.
    Read next: The AGENTS.md spec and Claude Code / Cursor docs on how nested AGENTS.md / CLAUDE.md files are discovered, scoped, and injected (directory-tree vs. explicit-reference loading).
  • illogicalThe piece conflates two distinct costs — dollar/token spend and attentional dilution — treating 'burns tokens' and 'loses focus' as the same problem. A short file can still be badly written, and a long file can be cheap but confusing; the levers are related but not identical.
    Read next: Work separating context-window cost/latency from instruction-following quality; prompt-engineering guidance on signal-to-noise vs. raw length.
  • incompleteIgnores that nested files reintroduce the same reactive-accretion risk and add a governance/discoverability burden across a distributed doc set — the same 'messy docs' failure mode at a smaller granularity.
    Read next: John Cutler on institutionalized overload and messy docs (TBM 411/412); documentation-governance and information-architecture practices for keeping distributed config coherent.
