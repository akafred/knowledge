AI-assisted development is a real step up in abstraction that pays off only when teams pair it with judgment, structure and active learning — speed gains come from agentic tooling, but durable value comes from evals, information architecture and refusing to outsource learning to the model.

practical · mixed — A conversation that is mainly practical (how should developers and organizations adopt AI-assisted development right now?) but interleaves theoretical reflection on abstraction history (assembly → high-level → natural-language instruction) and on the limits of determinism. The hosts use the AI-special framing to push two complementary practitioners — an executive shepherding org-wide adoption and a hands-on senior — to make their working assumptions explicit.

Key terms

  • kodeagent / code agent Coding Agents
    An LLM running in a loop with tool access that works *with* a developer on their local code, gated by the developer's review — distinct from an LLM that runs in production
  • business agent
    An agent that runs in production, exposed to end-users or external systems — different risk profile than a code agent because the LLM itself is in the live request path
  • vibe coding Vibe Coding
    Originally: explicitly not looking at the generated code. As actually used by most people: any LLM-driven coding. Anders distances himself: he uses LLM assistance but reviews tests and structure closely
  • AI-augmented engineering / augmented engineering Agentic Engineering
    Vilde's house term at FINN/Vend for the developer-tooling track of AI adoption (as opposed to product agents or business agents)
  • evals / smak / judgment Evals
    The capacity to look at AI output and tell whether it is good — both as ad-hoc human evaluation ('smak', taste) and as instrumented automated metrics
  • guardrails Guardrails
    Structural constraints (tests, compilers, small commits to prod, type systems, minimal interfaces) that both keep humans and code agents on track — Truls frames them as what makes AI 'behaviorally deterministic' even when it isn't statistically so
  • outsource the learning (du kan ikke outsource læringen) Learning Culture & Slack
    Vilde's slogan for refusing to delegate learning to the model — everyone at every level must try the tools themselves rather than wait for someone else to figure them out
  • sustainable software (Charity Majors) Changeability & Optionality
    Anders borrows Charity Majors' top-of-pyramid category for software that has to live a long time and absorb changes — the regime where AI's speed gains do *not* dominate the work
  • information architecture (in the AI context) Software Design Fundamentals
    Vilde: clear definitions, naming, structure, and consistent patterns — the thing that *still* matters most when LLMs do a lot of generation, because it determines whether the model can navigate the system
  • remote agents / agent-team Multi-Agent Orchestration
    Vilde's near-term vision: not one agent paired with one human, but multiple agents (dependency police, vulnerability watch, code-review, consistency) running in parallel as teammates
  • autonomy levels (Droid / Factory AI) Guardrails
    A model of staged AI permission: planning only → low-impact (reversible) → medium → high (destructive) — Anders endorses this as a useful way to ship LLM access without trusting the LLM
  • prompt injection Agent Security
    An attack where content the LLM reads (web pages, files, mail) contains instructions that hijack the agent — explicitly named as inherent to how LLMs work, not fully solvable
  • kode blir gratis ('code becomes free') Near-Zero-Cost Code
    Vilde's compact phrasing of the shift: some old engineering worries (over-engineering, premature abstraction) lose force when code is cheap to throw away — but information architecture worries do not

Arguments

Information architecture and minimization remain load-bearing engineering skills even when most code is generated. strength: high
Explicit premises:
  • Code agents respond better to small, well-structured changes than to mass rewrites (Anders).
  • Even when prototype-style regeneration works, sustainable software must absorb future changes (Anders).
  • LLM-driven product features still need consistent domain definitions to behave correctly (Vilde).
Implicit premises:
  • The bottleneck on AI-assisted teams is the legibility of the codebase to the next iteration of agents, not human throughput.
  • Codebases that are unstructured today already cost teams more than they admit (Vilde's 'who wrote this idiot code? mostly me' aside).
Inferences:
  • Teams that ditch design discipline in favor of vibe coding will degrade *faster* than before, not slower.
Productivity multipliers from code agents are not uniform; they depend on whether the bottleneck is typing, knowledge, or interdisciplinary judgment. strength: high
Explicit premises:
  • Anders measures ~10–40% uplift on familiar work.
  • Vilde points to people doing Terraform/infra they couldn't do before — effectively unbounded uplift.
  • Anders cites NAV dagpenger work where the bottleneck is jurist-developer alignment over weeks, not typing speed.
Implicit premises:
  • Productivity is a function of where the binding constraint sits, not of the toolchain.
  • Reporting a single 'AI productivity number' is misleading.
Inferences:
  • Org-wide ROI estimates for AI coding are likely wrong unless they segment by software type and developer baseline.
Organizations should treat AI adoption as a competence program with mandatory participation, not as a tooling rollout. strength: high
Explicit premises:
  • Tools change weekly; best practices lag (Vilde).
  • If you wait for someone else to figure it out, you cannot understand their conclusions when they arrive (Vilde).
  • Leaders, CFOs, and summer interns all benefit from at least one 'oh shit' moment of personally building something (Vilde).
Implicit premises:
  • Competence with agents is path-dependent — it does not transfer from reading about it.
  • Organizational change in AI adoption is bottlenecked by individual willingness to be a beginner again.
Inferences:
  • Procurement-led or governance-led AI rollouts are insufficient and probably counterproductive.
The senior/junior story is more about who is willing to retool than about who has more experience. strength: high
Explicit premises:
  • Some seniors are the biggest winners (mastery + agent leverage).
  • Many seniors fail to retool and stay at baseline velocity.
  • Juniors graduating today will have agent fluency baked into reflexes.
Implicit premises:
  • Career-stage proxies are weaker than retraining-rate proxies for predicting who thrives.
  • Hiring/team composition decisions based on 'we need seniors to validate agent output' are likely wrong on average.
Inferences:
  • Within five years, the population of high-leverage developers will be agent-native juniors and retooled seniors, with non-retooled seniors marginalized.
Risk discipline for code agents has to be explicit because the agents inherit the developer's privileges and route content to third-party servers. strength: high
Explicit premises:
  • Claude Code can drive the whole machine (file write, command execution).
  • Prompt injection from anything the agent reads is inherent and not fully solvable (Anders).
  • Code traffic and possibly persondata go to vendor servers; this requires DPA agreements (Vilde).
Implicit premises:
  • Default developer setups understate this attack surface relative to historic IDE/copy-paste behaviour.
  • Sandboxing and autonomy gating (Droid-style) is a stopgap, not a fix.
Inferences:
  • Norwegian public-sector and regulated organizations will need bespoke agent-tool choices, not the default consumer stack.
Generative-AI augmentation at the team level is genuinely unsolved and should be approached as an open question, not a marketing claim. strength: high
Explicit premises:
  • Individual productivity gains are well-attested.
  • Reference to Nils Brede Moe's research line on team-level augmentation.
  • Vilde admits FINN has not 'cracked the code' on team-wide agent use beyond green-field experiments.
Implicit premises:
  • Org-design and team-level coordination patterns have not adapted to multi-agent workflows.
  • Most 'AI in teams' claims in industry conflate individual gains with team gains.
Inferences:
  • There is professional value in being honest about this gap rather than papering over it.

Narratives

  • Anders converts an Azure Bicep IaC project to Terraform with a code agent. He doesn't really know Bicep, knows some Terraform, gets the first conversion + double-check loop done in a way he calls 'helt magisk'.
    → Concrete illustration of capability-unlocking productivity rather than typing-speed productivity.
  • A NAV developer working on dagpenger thinks ahead for three weeks before writing code; uses the agent mainly as a thinking partner, not to type faster, because the work is dominated by interdisciplinary alignment with jurists and political constraints.
    → Counter-example showing that AI does not collapse work where the bottleneck is shared understanding.
  • Amazon agent buys products from third-party sites when customers order things Amazon doesn't stock; vendors are unhappy to find themselves on Amazon.
    → Illustration of business agents (LLM in production) creating real-world consequences outside the org's existing rails.
  • Anders gives Claude/code agent access to OpenTelemetry production data; it analyses an error, finds the root cause, and *automatically opens a pull request* with a fix — full-stack agentic loop from prod telemetry to code change.
    → Anchors the unity's 'real step up in abstraction' claim — closes the production-to-code feedback loop a human would normally orchestrate manually.
  • Vilde steps into a phone booth, voice-dictates a conference pitch as if babbling to the machine; what comes out is very different from what she would write fingertip-first.
    → Shows that the new modality (voice as input + agent loop) changes the *kind* of artifact a senior leader can produce, not just the speed.
  • Vilde's team finds an 'ordinary' team that succeeded with code agents and uses *them* (not the star individuals) as the demo case so other teams can't dismiss the result as exceptional-person dependent.
    → Implicitly argues for selection bias as a core obstacle in tech adoption.
  • Anders' attempt to plan a London trip with an agent: it works hard, takes a long time, doesn't deliver much value — contrasted with the Open Telemetry pull-request flow, which did.
    → Honesty marker — concrete failure case that disciplines the rest of the optimism.
  • Anders and Truls/Audun shared a stage in 2011–12 talking about the importance of *removing* unnecessary complexity — a 15-year continuity that frames Anders' AI stance as 'the new tool for an old goal'.
    → Establishes the implicit premise that AI's value is judged against a long-held simplicity/clarity criterion, not against a productivity-only criterion.

Questions addressed

  • ✅ What happens to software development when AI moves from helper-on-the-side to an integrated part of how teams build, test and deliver?
    Abstraction level rises; some old worries fade (over-engineering, perfect upfront design); judgment, evals and information architecture become the durable engineering work; team-level patterns remain open.
  • ✅ How fast does AI actually make experienced developers?
    Anders' best-effort number is 10–40% for familiar work; Vilde reframes the question to 'effectively unbounded for work you couldn't otherwise do'; both reject the 2x headline.
  • ⭕ Will juniors be able to learn the trade if agents do the early work for them?
    Both express concern; Vilde frames it as a recurring generational worry and notes some fears (e.g. 'they won't learn X') will turn out to be moot when X stops mattering, but neither claims to know how juniors will build judgment.
  • ✅ How do organizations scale AI-coding adoption from a frontier individual to the whole engineering org?
    Vilde gives a concrete playbook (ordinary-team demos, two Slack channels, workshops, top-down 'no opt-out from learning', best tools provided).
  • ✅ How should engineering teams reason about determinism when LLMs are in the system?
    Stop chasing statistical determinism; build behavioral determinism via guardrails (tests, small commits, structured code, evals); preserve domain consistency at the backend.
  • ✅ What are the right risk frames for code agents and computer-using agents?
    Treat machine access as authorization escalation; require DPA agreements; gate autonomy by reversibility; accept that prompt injection cannot be fully eliminated.
  • ⭕ Does AI augment teams the way it augments individuals?
    Both guests explicitly admit this is open; reference to Nils Brede Moe's research line.
  • ⭕ How do we maintain the evaluation skill ('smak / judgment') when daily practice no longer drills it?
    Vilde insists 'don't outsource learning' and Anders flags the concern, but neither proposes a mechanism for sustaining evaluation skill across a workforce.
