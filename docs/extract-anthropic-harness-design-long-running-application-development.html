Harness design for long-running application development

anthropic.com · article · 2026-03-24 Reader 🖥️

Agent HarnessAgent Planning ArtifactsAI SlopContext Rot & Context LimitsEvalsLong-Running & Managed AgentsMulti-Agent OrchestrationRiding Model Improvements
Harness design for long-running autonomous coding should be approached as a moving target: build a multi-agent architecture (planner, generator, evaluator) that compensates for the current model's specific weaknesses on long-horizon, subjective-judgment tasks, then aggressively strip components as model capability advances — because every harness component encodes an assumption about what the model can't do alone, and those assumptions go stale.

practical · mixed — Primarily practical: argues for specific design patterns (GAN-inspired generator/evaluator, three-agent planner/generator/evaluator architecture, stripping components when models improve) in service of building long-running autonomous coding harnesses. Mixed form: expository (defining failure modes like context anxiety, self-evaluation leniency), narrative (the retro-game-maker and DAW case studies), and argumentative (a methodological stance about how to evolve harnesses as models improve). Vendor undertone: every recommendation is anchored to specific Claude model versions (Sonnet 4.5 → Opus 4.5 → Opus 4.6) and Anthropic-built scaffolding (Claude Agent SDK, frontend-design skill, Playwright MCP).

Key terms

  • Harness Agent Harness
    The orchestration scaffolding surrounding one or more Claude agents — agent roles, prompts, tools (MCPs), inter-agent communication mechanisms, and loop logic — that enables the model to perform a complex task that it cannot do in a single inference pass. Implicitly treated as a stack of removable 'components,' each encoding an assumption about a specific model weakness.
  • Long-running Long-Running & Managed Agents
    Multi-hour autonomous coding sessions (concretely: 3.8 to 6 hours in the case studies) that produce 'rich full-stack applications' end-to-end without human intervention. Not formally defined by a time threshold but operationalized in the examples.
  • Context anxiety Context Rot & Context Limits
    A model behavior in which the agent prematurely wraps up work as it approaches what it believes is its context window limit. Distinct from actual coherence loss; it is the model's anticipatory truncation of effort. Author claims Sonnet 4.5 exhibits this strongly; Opus 4.6 largely eliminated it.
  • Context reset Context Rot & Context Limits
    Clearing the context window entirely and starting a fresh agent, combined with a structured handoff artifact that carries the previous agent's state and next steps. Contrasted with compaction.
  • Compaction Context Rot & Context Limits
    Summarizing earlier parts of a conversation in place so the same agent can keep going on a shortened history. Preserves continuity but does not eliminate context anxiety because the agent is still 'the same agent.'
  • Self-evaluation leniency Multi-Agent Orchestration
    The empirical tendency of an agent asked to grade its own output to praise the work confidently even when quality is mediocre. Especially pronounced for subjective tasks lacking binary verification, but also present on verifiable tasks. Not fully eliminated by separating evaluator from generator, but greatly reduced.
  • Generator / Evaluator (GAN-inspired pattern) Multi-Agent Orchestration
    An agent architecture borrowed conceptually from Generative Adversarial Networks: one agent produces, a separate agent grades against explicit criteria and writes critique, feedback flows back to generator for next iteration. Author concedes the evaluator is itself an LLM still 'inclined to be generous,' but separating it makes skepticism tractable.
  • Sprint contract Agent Planning Artifacts
    A negotiated agreement between generator and evaluator agents, before code is written, defining what 'done' means for a chunk of work — specifically: what will be built and how completion will be verified. Bridges the gap between a high-level spec and testable implementation. Communicated via files.
  • Load-bearing (harness component) Riding Model Improvements
    A piece of the harness without which the final output's quality measurably degrades. The methodological test is to remove components one at a time and measure impact. Whether a component is load-bearing depends on the model version.
  • Planner / Generator / Evaluator (three-agent harness) Multi-Agent Orchestration
    The Anthropic-Labs canonical architecture: planner expands a short user prompt into a full product spec; generator builds in sprints (V1) or continuously (V2) against the spec; evaluator drives the running app via Playwright MCP and grades against criteria with hard thresholds, returning detailed feedback.
  • Grading criteria Evals
    Explicit, written prompts shared between generator and evaluator that convert subjective judgments (e.g. 'is this design good?') into concrete graded dimensions (design quality, originality, craft, functionality). Criteria wording itself steers the output — e.g. 'museum quality' pushes designs toward visual convergence.
  • AI slop AI Slop
    Highly generic, telltale AI-generated visual patterns — explicitly: 'purple gradients over white cards,' unmodified stock components, library defaults. The evaluator's originality criterion is explicitly designed to penalize these.

Arguments

A separated generator and evaluator is necessary for long-running autonomous coding tasks. strength: high
Explicit premises:
  • Agents grading their own work exhibit confident leniency, especially on subjective tasks.
  • Even on verifiable tasks, agents show poor judgment that impedes performance.
  • Tuning a standalone evaluator to be skeptical is tractable; tuning a generator to self-critique is not.
Implicit premises:
  • It is desirable that the agent system iterate toward higher quality rather than stop at first-pass-acceptable.
  • The cost of running a second LLM as evaluator is acceptable relative to the quality lift.
  • Two LLMs in different roles will produce sufficiently different judgments — even though both share the same underlying base model's biases — to constitute a meaningful feedback signal.
Inferences:
  • The pattern generalizes beyond coding to any subjective-judgment task where verifiable correctness alone is insufficient.
  • Single-agent self-reflection patterns (e.g. 'review your own work then rewrite') are structurally weaker than multi-agent separation.
Harnesses should be aggressively simplified component-by-component when a more capable model is released. strength: high
Explicit premises:
  • Every harness component encodes an assumption about what the model can't do alone.
  • These assumptions can quickly become stale as models improve.
  • Methodical one-at-a-time component removal is the only reliable way to identify what is still load-bearing.
  • Opus 4.6 eliminated context anxiety, so context resets became unnecessary; sprint structure became unnecessary because Opus 4.6 can sustain multi-hour coherent runs.
Implicit premises:
  • Simpler harnesses are preferable to more complex ones when output quality is held constant — i.e. there is an inherent cost to harness complexity (latency, token cost, brittleness, maintenance).
  • Model improvements between releases will tend to subsume previously load-bearing harness compensations, rather than (e.g.) introduce new failure modes that require new compensations.
  • The engineer building the harness has the time, model access, and tooling to perform the systematic component-removal experiment at each release — i.e. the methodology is feasible for the implied audience.
  • Model versions advance frequently enough that adapting the harness is worth the engineering cost (versus, e.g., choosing model-agnostic patterns).
Inferences:
  • Harness investments depreciate rapidly — any organization building heavy multi-agent scaffolding should treat it as short-lived.
  • Tooling that lets a team A/B harness variants quickly is itself a high-value investment.
  • Vendors with shipping cadence have an advantage over slow-shipping alternatives because harness simplification is a benefit only the next-model customer captures.
Multi-agent harnesses justify their 20x+ cost when tasks lie beyond the model's solo capability.
Explicit premises:
  • On the retro-game-maker task, solo run produced a broken core feature; harness run produced a working one.
  • On the DAW task, the QA evaluator caught real gaps the generator left (stub recording, missing clip drag, etc.) that solo would have missed.
Implicit premises:
  • The buyer of long-running autonomous coding cares about end-of-run usability rather than time-to-first-output.
  • $124-$200 per multi-hour autonomous build is an economically acceptable price point — this is the implicit deployment model.
  • The relevant comparison is harness-vs-solo on the same model, not harness-on-this-model vs. waiting-for-the-next-model.
  • Tasks sit on a definable capability frontier and the engineer can locate where their task sits relative to model capability.
Inferences:
  • There is a frontier-sensitive deployment heuristic: invoke the full harness only when the task is judged beyond solo capability; use solo otherwise.
  • This implies the orchestrator needs a calibration of model capability for each task type — a non-trivial meta-judgment problem the article does not address.
Grading criteria should be made explicit and weighted toward dimensions where the model is weak, not where it is strong. strength: high
Explicit premises:
  • Claude scores well by default on craft and functionality.
  • Claude scores poorly by default on originality and design quality, producing 'AI slop.'
  • Weighting criteria toward weak dimensions pushes the generator to take aesthetic risks.
Implicit premises:
  • What the author considers 'good design' is a sensible target for steering — the criteria are not themselves contested (they are presented as universal).
  • Steering the model toward 'museum-quality' visual convergence is a feature, not a bug, despite the author's own observation that criteria-wording produces stylistic homogenization.
Inferences:
  • The criteria-as-prompt effect means harness designers are also implicit aesthetic curators — the criteria *are* a style.
  • Different criteria authors will produce systematically different default Claude outputs at scale.
File-based inter-agent communication is sufficient infrastructure for a three-agent harness.
Explicit premises:
  • The author implemented planner-generator-evaluator with files as the medium.
  • It worked end-to-end on two case studies.
Implicit premises:
  • Filesystem is a shared, low-latency, cheap, durable medium — therefore appropriate for this orchestration scale.
  • Audit and observability needs are met by file-based logs (this is favorable for regulated/enterprise settings — though the author does not say so).
Inferences:
  • Heavier orchestration infrastructure (message brokers, state machines, dedicated agent frameworks) is unnecessary for this class of problem.
  • For users building in regulated environments, file-based communication has audit-trail advantages.

Narratives

  • The Dutch art museum design: by iteration 9, Claude had produced a clean dark-themed museum landing page that the author found 'visually polished but largely in line with my expectations.' On iteration 10, the model scrapped the approach and reimagined the site as a 3D room with checkered floor in CSS perspective, artwork on walls, doorway-based navigation. The author calls it 'the kind of creative leap that I hadn't seen before from a single-pass generation.'
    → Implicitly argues that the generator-evaluator iteration loop is not merely a quality-floor mechanism but unlocks emergent creative outputs the model would not produce in single-pass. It also implicitly supports the proposition that iteration count matters — quality is non-monotonic and the 10th iteration was qualitatively different from the 9th.
  • Retro game maker, solo vs. harness: a one-sentence prompt produced (solo, $9, 20 min) an app with broken entity wiring and a non-functional play mode; the same prompt produced (full harness, $200, 6 hr) a 16-feature spec across 10 sprints, with a working play mode, richer editors, and an AI-assisted level generator. The author notes that an unjumpable wall in the level revealed the agent's lack of common-sense game-design.
    → Carries the core empirical argument that harness lift is real and justifies 20x cost on Opus 4.5. Also implicitly carries a quieter argument: that the gap between 'looks impressive' and 'actually works when you try to use it' is the killer issue, and that Playwright-driven evaluators close it.
  • DAW build on simplified V2 harness with Opus 4.6: 3h 50m, $124.70, three rounds of build-then-QA. Round 1 produced a beautiful-but-shallow app (timeline clips not draggable, no instrument UI panels, no EQ curves); QA caught it; Round 2 still left audio recording as stub-only and effects as numeric sliders rather than graphical EQ curves.
    → Implicitly argues that even with a simpler harness and a more capable model, the generator-without-evaluator pattern would have shipped a non-functional 'display-only' app. The evaluator's value persists at the capability frontier.
  • Specific evaluator findings: 'Rectangle fill tool allows click-drag to fill a rectangular area' → 'FAIL — Tool only places tiles at drag start/end points instead of filling the region. fillRectangle function exists but isn't triggered properly on mouseUp.' / 'Delete key handler at LevelEditor.tsx:892 requires both selection and selectedEntityId...' / 'PUT /frames/reorder route defined after /{frame_id} routes. FastAPI matches reorder as a frame_id integer.'
    → Carries the implicit claim that LLM evaluators can produce file-and-line-number-precise QA findings — not vague 'the layout feels off' critique. This is doing real argumentative work because it answers an implicit skeptic ('LLM QA can't possibly produce actionable bug reports').
  • Evaluator tuning loop: author repeatedly reads QA logs, finds where the evaluator's judgment diverges from his own, and updates the evaluator's prompt. Multiple rounds. Even then, the evaluator misses deeply nested features and subtle interaction issues.
    → Implicitly supports the claim that harness design is a craft requiring sustained human judgment — there is no auto-generated evaluator. Quietly undercuts vendor narratives of 'just plug in two agents and they bootstrap each other.'
  • DAW QA can't evaluate musical taste because Claude can't hear: the agent could set tempo and key, lay down melody and drums, adjust mixer levels, add reverb — all via tools — but the author cannot use the QA loop to evaluate whether the music sounds good.
    → Implicitly carries a modal-capability boundary argument: the evaluator pattern is bounded by the evaluator's perceptual modality. Where the model lacks a sense (hearing, embodiment, etc.), the iteration loop degrades.

Questions addressed

  • ✅ How do we get Claude to produce high-quality frontend designs that aren't bland defaults?
    Generator + evaluator with explicit weighted grading criteria that explicitly penalize 'AI slop' patterns and reward originality; 5-15 iteration loops; few-shot calibrated evaluator with Playwright MCP for live navigation.
  • ✅ How do we get Claude to build complete full-stack applications without human intervention over multi-hour sessions?
    Three-agent harness (planner expands prompt to spec; generator works in sprints or continuously; evaluator drives the running app via Playwright with hard-threshold criteria); on weaker models, add sprint contracts and context resets; on stronger models, strip those away.
  • ✅ How do we know when a harness component is still earning its place vs. has become obsolete?
    Methodical one-at-a-time removal, measuring impact on final output quality. Treat every component as a hypothesis about a current model weakness.
  • ✅ When is the multi-agent harness worth its cost vs. just running the model solo?
    Worth the cost when the task sits beyond what the current model handles reliably solo; otherwise the evaluator is 'unnecessary overhead.'
  • ⭕ How do we calibrate where 'beyond solo capability' lies for an arbitrary task?
    Not addressed — the author's own calibration is implicit and experience-based ('our builds were at the edge of what the generator could do well solo').
  • ⭕ How do we manage cost, audit, observability, and failure containment for long-running autonomous coding in regulated or production contexts?
    Not addressed. The piece is a Labs / case-study post; no engineering treatment of cost ceilings, audit trails, sandboxing, secret management, IP provenance, or rollback.
  • ⭕ How should harnesses be designed when the generator and evaluator share the same base model's biases?
    Acknowledged ('the evaluator is still an LLM that is inclined to be generous towards LLM-generated outputs') but not solved — the author relies on prompt-engineered skepticism and few-shot calibration without a structural answer.
  • ⭕ What happens when the next model introduces new failure modes rather than just resolving old ones?
    Not addressed — the simplification narrative assumes monotonic capability improvement.
