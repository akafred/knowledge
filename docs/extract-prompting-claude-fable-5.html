Prompting Claude Fable 5

Claude Platform Docs · article · 2026-07-07 source 🖥️

Agent HarnessAgent MemoryContext EngineeringLong-Running & Managed AgentsMulti-Agent OrchestrationPrompting as CommunicationRiding Model Improvements
Prompting Claude Fable 5 well means steering a markedly more capable, longer-horizon, strongly-instruction-following model with brief goal-and-intent-first directives and asynchronous, memory- and subagent-aware scaffolding — and pruning the enumerated rules, guardrails, and hand-holding that prior, weaker models required.

practical · expository — A how-to reference page: it prescribes prompting and scaffolding patterns and supplies copy-paste system-prompt blocks. It asserts rather than argues; rationale (`because the model was trained to...`) is thin and mostly implicit.

Key terms

  • effort
    The primary API control for the intelligence/latency/cost trade-off on Fable 5, with discrete settings (low, medium, high, xhigh); default `high`, `xhigh` for the most capability-sensitive work, `medium`/`low` for routine tasks. Lower effort on Fable 5 is said to often exceed prior models' `xhigh`.
  • adaptive thinking
    Fable 5 uses adaptive thinking only — there are no configurable extended-thinking token budgets; the model decides how much to reason internally.
  • summarized (summarized-only) thinking output
    The model exposes only a summary of its internal reasoning; extracting the full/raw thinking is one of the actions its safety classifiers target.
  • refusal stop reason (`stop_reason: "refusal"`)
    A distinct API stop reason returned when a request trips Fable 5's safety classifiers; the doc pairs it with configuring server- or client-side fallback to Opus 4.8.
  • safety classifiers
    Runtime classifiers that decline requests in offensive-cybersecurity, biology/life-sciences, and summarized-thinking-extraction domains — and that may also trip on benign cybersecurity or beneficial life-sciences work.
  • long-horizon autonomy
    Fable 5's capacity to sustain productive, goal-directed output over extended, multi-day runs with strong instruction retention across long, complex tasks.
  • elicitation language / system reminder
    A short instruction placed in the system prompt that is required to actually trigger a behavior or tool use (e.g., autonomous operation, or calling `send_to_user`); the doc stresses that defining a capability or tool is insufficient on its own without this instruction.

Arguments

Provide the reason and intent behind a request, not just the request. strength: medium
Explicit premises:
  • Fable 5 tends to perform better when it understands the intent behind a request.
  • Context lets the model connect the task to relevant information rather than inferring intent on its own.
Implicit premises:
  • A model given goals and context generalizes to unstated sub-decisions better than one given bare instructions.
Inferences:
  • If understanding intent improves output, then supplying the why (task, audience, what the output enables) should improve output.
You can steer most behaviors with a single brief, high-level instruction instead of enumerating each case. strength: medium
Explicit premises:
  • Instruction-following is improved enough at this model level.
  • A short brevity instruction is as effective as listing each pattern.
Implicit premises:
  • Stronger instruction-following means a high-level directive reliably generalizes across the specific behaviors it implies.
Inferences:
  • Therefore enumeration is redundant work that a brief directive subsumes.
The audit-against-tool-results instruction reliably suppresses fabricated status reports. strength: high
Explicit premises:
  • In Anthropic's testing, this instruction nearly eliminated fabricated status reports even on tasks designed to elicit them.
Implicit premises:
  • Anthropic's internal test set is representative of the reader's own workloads and failure conditions.
Inferences:
  • From near-elimination on adversarial internal tests, infer general reliability of the technique.

Narratives

  • The 'When you have enough information to act, act. Do not re-derive facts already established... give a recommendation, not an exhaustive survey' block.
    → Demonstrates a single brief instruction curbing overplanning/option-surveying on ambiguous tasks.
  • The 'Before reporting progress, audit each claim against a tool result from this session. Only report work you can point to evidence for...' block.
    → Demonstrates the anti-fabrication technique — grounding status claims in tool evidence.
  • The 'You are operating autonomously. The user is not watching in real time... Before ending your turn, check your last paragraph. If it is a plan, an analysis, a question... do that work now with tool calls' block.
    → Demonstrates the autonomous-pipeline system reminder that prevents early stopping and turns end-of-turn promises into tool calls.
  • The intent template 'I'm working on [the larger task] for [who it's for]. They need [what the output enables]. With that in mind: [request].'
    → Demonstrates giving the reason/intent — a fill-in-the-blank pattern operationalizing purpose and audience for an agent.
  • The `send_to_user` tool JSON schema plus the elicitation line 'Between tool calls, when you have content the user must read verbatim... call the send_to_user tool with that content.'
    → Demonstrates that a defined tool needs paired system-prompt elicitation to actually be used, and that tool inputs pass through un-summarized.
  • The 'You have ample context remaining. Do not stop, summarize, or suggest a new session on account of context limits. Continue the work.' block.
    → Demonstrates countering context anxiety triggered by a visible remaining-token countdown.
  • The memory-note format block 'Store one lesson per file with a one-line summary at the top. Record corrections and confirmed approaches alike...' plus 'Reflect on the previous sessions we've had together...'
    → Demonstrates a minimal file-based memory system and how to bootstrap it from past sessions.

Questions addressed

  • ✅ How should I migrate a harness from Opus 4.8 to Claude Fable 5?
    Adjust client timeouts, streaming, and progress indicators and move to async run-checking (runs can last minutes to hours); prune now-unnecessary instructions, tools, and guardrails; and configure fallback to Opus 4.8 for refusals.
  • ✅ How do I stop an agent from fabricating or overstating progress?
    Instruct it to audit each progress claim against a tool result from the session and report only work it can point to evidence for.
  • ✅ How do I trade off intelligence against latency and cost?
    Use the `effort` setting (default `high`, `xhigh` for the hardest work, `medium`/`low` for routine), and reduce effort when a task is slower than needed.
  • ✅ Why does the agent stop early, ask permission mid-run, or suggest a new session, and how do I prevent it?
    Add an autonomous-operation system reminder plus a checkpoint definition, and avoid surfacing remaining-token countdowns (or add a 'you have ample context remaining' reassurance).

Gaps

  • incompleteNo eval methodology to verify that any recommended prompt actually works for the reader's own task. The single near-quantified claim ('nearly eliminated fabricated status reports') is unquantified, drawn from Anthropic's own testing, with no reproducible benchmark, dataset, baseline, or numbers — so a rigorous adopter cannot confirm which system-prompt blocks earn their place.
    Read next: Pair with an evals discipline (e.g. extract-anthropic-harness-design-long-running-application-development, which uses graded evaluators); build an A/B eval harness that measures each system-prompt block's marginal effect before adopting it wholesale.
  • incompleteThe document is undated. It is thoroughly version-specific (Fable 5 vs Opus 4.8, effort tiers, adaptive/summarized-only thinking, the `refusal` stop reason, safety-classifier domains) yet carries no published or last-updated date, so its claims will drift silently as the model and platform update.
    Read next: Cross-check every version-specific claim against the platform changelog and the 'Introducing Claude Fable 5 and Mythos 5' page at time of use; re-verify before betting a migration on any stated default or limit.
  • incompleteThe behavioral-improvement claims (bug-finding recall 'noticeably higher', vision accuracy 'substantially higher', single-pass implementations of multi-day systems, more dependable subagent dispatch) are all qualitative, with no benchmark numbers, task definitions, or independent verification.
    Read next: Seek third-party or benchmark evidence for the specific claimed gains before relying on them; treat the improvement list as vendor signal, not measurement.
  • incompleteNo guidance on interaction effects among the many recommended system-prompt blocks (brevity, anti-over-engineering, anti-overplanning, checkpoint, autonomy, readability, progress-audit, memory). The doc never says whether they conflict, how much instruction budget they consume together, or which takes precedence — awkward given its own advice to prefer brief high-level instructions over enumeration.
    Read next: Consult context-engineering / progressive-disclosure material on instruction budgets; assemble and test the composed system prompt as a whole rather than validating each block in isolation.
  • incompleteEnglish- and software-only. Every example prompt is English and targets agentic/coding/enterprise-document work; there is no guidance for non-English prompting or for non-software domains, even though the guide claims broad capability gains.
    Read next: Look to practitioner reports on multilingual and domain-specific steering; do not assume the English software-agent prompts transfer unchanged.
  • uninformedNo stated boundary for when 'steer with a brief instruction' stops working and explicit enumeration becomes necessary. The doc asserts brevity suffices, yet its own recommended blocks are long, multi-clause instructions — leaving the reader without a rule for where high-level steering degrades.
    Read next: Use eval-driven tuning to find the point where high-level steering fails and enumeration is required for a given behavior and effort level.
