AI Model Context Protocol Adds Centralised Auth for Enterprise

Matt Saunders · rss · 2026-07-06 source ↗ Reader 🖥️

Agent Governance (Internal)Agent SecurityEnterprise AI Adoption & ReadinessMCP (Model Context Protocol)
By promoting its Enterprise-Managed Authorisation extension to stable, MCP moves enterprise agent authorization out of per-user, per-server consent and into the organization's identity provider, removing the friction blocking large-scale MCP adoption while deliberately governing only connection-level access and leaving runtime, per-action control to the enterprise.

practical · expository — An InfoQ news brief reporting that MCP's Enterprise-Managed Authorisation (EMA) extension has reached stable status. It mixes neutral reporting with curated external commentary (Ehsan Hosseini, Jiquan Ngiam, Okta) that frames EMA as the structural fix for enterprise MCP auth friction and implicitly nudges toward adoption. The implicit thesis: enterprise agent authorization is consolidating out of per-user consent and into the organization's identity provider.

Key terms

  • Model Context Protocol (MCP) MCP (Model Context Protocol)
    The open protocol standardizing how AI agents and clients connect to external tools and data through MCP servers; here it is the substrate whose enterprise authorization model is being upgraded.
  • Enterprise-Managed Authorisation (EMA)
    An MCP extension, now stable, that lets an organization control access to MCP servers centrally through its identity provider, so users sign in once and inherit access to the servers their organization has approved.
  • Identity Assertion JWT Authorisation Grant (ID-JAG)
    The OAuth grant (an IETF draft) the identity provider issues to assert a user's identity, which the MCP server's authorization server exchanges for an access token; the token-exchange primitive that underpins EMA.
  • Connection-level vs. runtime authorization
    EMA decides whether a client may connect to a server and at what scope, but explicitly does not inspect or authorize individual actions after the token is issued; a deliberate scope boundary the enterprise must fill with its own runtime controls.
  • MCP server MCP (Model Context Protocol)
    The service that exposes tools and actions to agents, whose access EMA governs at the connection level.
  • Cross App Access (Okta)
    Okta's approach and the first supported identity-provider path implementing EMA in the launch.

Arguments

Making EMA stable removes one of the most obvious operational frictions in enterprise MCP adoption and is significant for teams connecting AI assistants to internal tools at scale. strength: medium
Explicit premises:
  • The community reports repeated authorization and consent prompts as one of the biggest pain points in enterprise MCP deployments.
  • EMA moves authorization into the enterprise identity provider, giving a single-login experience where users inherit access to approved servers.
  • EMA is now stable and adopted across identity, model, IDE and server vendors (Anthropic, Microsoft, Okta, VS Code, Asana, Atlassian, Canva, Figma, Granola, Linear, Supabase, with Slack in progress).
Implicit premises:
  • Authorization friction, not model capability, was the binding constraint on scaling enterprise MCP.
  • A stable spec plus broad vendor adoption is sufficient to make centralized enterprise auth a de-facto default rather than a niche integration.
  • Enterprises trust identity-provider-mediated, admin-defined policy more than per-employee, per-server consent.
Inferences:
  • From 'stable and broadly adopted' it infers 'default rather than special integration', an adoption-breadth-implies-standardization step.
  • From 'removes a named friction' it infers 'significant for adoption at scale' without any outcome data.
EMA is not a complete governance solution; organizations still need their own controls for what an agent does once it is inside a system. strength: high
Explicit premises:
  • The guide explicitly warns EMA is not runtime authorization for individual actions.
  • The enterprise-managed layer decides connection and scope but does not inspect MCP traffic after the token is issued.
  • Ehsan Hosseini independently distinguishes connection-level control from per-action control and argues EMA does not replace runtime policy enforcement.
Implicit premises:
  • Connection-level authorization and per-action authorization are genuinely separate governance problems (defense-in-depth).
  • Centralizing connection authorization can create a false sense of complete control if the runtime layer is neglected.
Inferences:
  • From 'authorization stops at token issuance' it infers 'a separate runtime control layer remains the enterprise's responsibility'.

Questions addressed

  • ✅ How can an enterprise control access to MCP servers without per-server consent prompts for every employee?
    Via EMA: the organization's identity provider becomes the authority; users sign in once (an ID-JAG is exchanged for an access token) and inherit access to org-approved servers.
  • ✅ Does EMA solve runtime, per-action authorization of what an agent does inside a system?
    No. The specification explicitly scopes EMA to connection-level authorization; organizations must still supply their own runtime controls.
  • ✅ Is EMA a universal solution today?
    Only partially. It requires both the identity provider and the MCP server to support the extension, remains additive, and leaves non-supporting organizations needing a fallback path.

    closed byGoverning AI in the Cloud: A Practical Guide for Architects — Partial, cross-layer closure. The sibling left 'what per-action controls actually look like once an agent holds a token' entirely to the reader; this guide gives a concrete pattern at the cloud data-path layer — deny-by-default IAM conditions evaluated per S3 request (classification/AIApproved tags, VPC-endpoint binding, SCP guards) and OPA/Rego policy-as-code evaluated at request time against contextual inputs, plus Kubernetes admission controllers (Kyverno / OPA-Gatekeeper). It answers the shape of runtime, per-action data-access control but at the infrastructure layer rather than MCP tool-call authorization, so the MCP-protocol-specific per-action gap remains only partially addressed.
  • incompleteNo security analysis of the EMA/ID-JAG mechanism itself. Centralizing authorization in the identity provider concentrates risk, yet token theft, identity-provider compromise, and the blast radius of a single control plane are unexamined, especially against the corpus finding that 43% of MCP servers already have authentication flaws.
    Read next: Threat models for OAuth identity-assertion grants and token exchange; MCP auth vulnerability reports (mcp-api-sprawl); non-human/workload identity standards such as SPIFFE.
  • incompleteThe EU / Norwegian regulated-enterprise angle is absent. Whether identity-provider-mediated connection authorization satisfies EU AI Act human-oversight and audit-chain obligations, and how it interacts with data-residency requirements, is not addressed, which are the deciding questions for a telco or bank.
    Read next: EU AI Act high-risk documentation obligations; NSM/Datatilsynet guidance; data-sovereignty sources in the corpus.
  • incompleteMigration and adoption cost is glossed. The 'fallback path' for organizations on a non-supporting identity provider is named but not examined, and no account is given of what it takes to move an existing per-user MCP deployment onto EMA.
    Read next: Enterprise-AI-adoption last-mile literature; identity-federation migration experience reports.
  • incompleteThe claim that auth friction was the binding constraint rests on vendor adoption and community sentiment, not evidence. No data shows that removing consent prompts actually increases successful enterprise MCP deployment rather than merely removing one visible annoyance.
    Read next: Independent enterprise-AI adoption studies distinguishing tooling friction from organizational readiness as the true constraint.
