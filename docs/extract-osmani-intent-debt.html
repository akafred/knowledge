The Intent Debt

Addy Osmani (addyosmani.com) · rss · 2026-06-06 source 🖥️

Architecture Decision RecordsAgent Context Files (CLAUDE.md / AGENTS.md)Cognitive DebtIntent DebtOrchestration Tax / Human Review BottleneckTechnical Debt
Intent debt — the absence of externalized rationale, goals and constraints in artifacts an agent can actually read — is the one of Storey's three debts (technical, cognitive, intent) that AI agents structurally cannot pay down, because intent is the one input that must originate with a human; and because agents start every session cold with no tacit memory, every gap in the externalized why is now paid every session by every agent, making the previously-affordable habit of carrying intent in your head the single most expensive workflow choice in agent-assisted development.

practical · mixed — Argumentative-with-prescriptions. Osmani imports a taxonomic model (Storey's triple debt) to do theoretical work — distinguishing intent debt from technical and cognitive debt — and then makes the case that agents alter the economics of one specific cell in that taxonomy. The closing section is operational: write specs for intent, treat AGENTS.md as an intent ledger, log decisions, run a self-improving learnings loop. Practical because every claim is ultimately in service of 'externalize the why'.

Key terms

  • Intent debt Intent Debt
    The absence or erosion of the EXTERNALIZED rationale, goals and constraints that explain why the system is the way it is — recorded in artifacts (specs, AGENTS.md, ADRs, learnings files, commit messages) that a teammate, a future you, or an agent can actually read. Crucially distinct from understanding-in-your-head: intent that exists only inside the senior engineer's skull is enormous intent debt for everyone (and every agent) else.
  • Technical debt (Osmani's framing within the triple model) Technical Debt
    Implementation choices that make the system harder to change later — tangled modules, leaked abstractions, deadline shortcuts. Lives in the code. Announces itself via slow builds, fragile tests, dread of touching a file. Increasingly cheap to take on AND increasingly cheap to pay down because an agent can refactor.
  • Cognitive debt (Osmani's framing within the triple model) Cognitive Debt
    Erosion of shared understanding — the gap between how much code exists and how much any human actually understands. Lives in people. Accumulates when the system grows faster than the team's mental model of it. Recoverable in a way underrated: you can ask the agent to explain the code back to you. Osmani equates it with what he previously called comprehension debt.
  • Independence (of the three debts)
    The technical/cognitive/intent debts are orthogonal axes: you can have pristine code AND nobody understands it AND no externalized rationale, in any combination. From the inside they 'feel similar' but 'are not the same bill'. The claim is operationally important — it means cleaning the code or rebuilding mental models will not reduce intent debt at all.
  • Externalize (intent) Intent Debt
    Move rationale OUT of human heads and INTO machine-readable artifacts. The verb does the work of the whole prescription section. 'External' means readable by an agent, not just by a teammate.
  • Tacit intent Intent Debt
    The why-of-the-system that humans absorb slowly through hallway conversations, code review comments, incident lore ('we don't do it that way because of an incident in 2023'), and senior-engineer mentorship. Worked for years because humans accumulated it; breaks under agents because agents do not.
  • Cold start (of an agent session) Intent Debt
    Every agent session begins without the long-term memory humans build over years; whatever isn't in a readable artifact, the agent doesn't have. Multiplied by every agent you run, every session. The economic engine of the post.
  • Plausible guess (about intent) Intent Debt
    What an agent produces when intent is missing from artifacts: a confident-sounding inference from the code that is structurally indistinguishable from real rationale, and worse than the agent admitting it doesn't know. Frames the failure mode as fabrication rather than absence.
  • Load-bearing rationale Intent Debt
    The subset of why-decisions whose loss would be expensive to reconstruct or dangerous to get wrong. The triage criterion that resolves the 'you can't write down everything' objection: capture the load-bearing why, not the totality.
  • AGENTS.md (as intent ledger, not config) Agent Context Files (CLAUDE.md / AGENTS.md)
    A file that describes what the TEAM means — conventions, 'we don't do it this way because…', constraints invisible in any single file — as opposed to /init-style auto-generated files that describe what the code IS. The distinction is what makes the artifact useful to agents.
  • Decision log / lightweight ADR Architecture Decision Records
    Capture-at-the-moment-of-decision artifacts. Pure intent-debt paydown: trivial cost at write time, enormous cost to reconstruct eight months later after the decider has left.
  • Self-improving agent / learnings file
    An agent loop that writes intent back into the repo at end of session — 'we tried X and it didn't work because Y' — converting each mistake's root cause from one person's bad-afternoon memory into a durable artifact. An 'intent-debt pump running in reverse'.
  • Helplessness (as the felt experience of intent debt)
    Intent debt is not noticed as friction (the way technical debt is) — it is noticed as 'a particular kind of helplessness': defending a design choice you can't reconstruct, or accepting an agent's refactor because nobody can say if the guard clause was load-bearing. The operational test for detecting intent debt.
  • Orchestration tax (as intent-debt tax) Orchestration Tax / Human Review Bottleneck
    Cross-reference to Osmani's previous post: the cognitive bandwidth cost of managing many agents is 'partly an intent-debt tax' — a lot of what makes orchestration exhausting is re-supplying the intent you never wrote down, once per agent per session.

Arguments

Intent debt is qualitatively distinct from technical and cognitive debt — not a subset of either. strength: high
Explicit premises:
  • Technical debt lives in code and announces itself via friction (slow builds, fragile tests).
  • Cognitive debt lives in heads and is recoverable by asking the agent to explain the code back.
  • Intent debt lives in artifacts you never wrote, is felt as helplessness rather than friction, and the rationale cannot be reconstructed from code alone — only fabricated.
  • The three debts are independent: clean code does not imply low intent debt, and personal understanding does not imply low intent debt for the team.
Implicit premises:
  • Storey's Triple Debt taxonomy is exhaustive of the debt classes that matter for agent-assisted development.
  • Rationale is not derivable from code by inference, even in principle — the code under-determines its own intent.
Inferences:
  • Effort spent on refactoring (technical paydown) or on rebuilding mental models (cognitive paydown) will not reduce intent debt by itself.
  • Teams measuring debt by code-quality metrics or by knowledge-bus-factor are missing the third axis entirely.
Of the three debts, intent debt is the one agents structurally cannot pay down. strength: high
Explicit premises:
  • Agents can refactor tangled code (technical paydown).
  • Agents can explain code back to you (partial cognitive paydown).
  • Intent must originate with a human, because the WHY is not in the code; an agent can only infer a plausible-sounding why.
  • A fabricated plausible why is worse than the agent admitting it doesn't know.
Implicit premises:
  • Distinguishing real intent from fabricated plausible-sounding intent is hard enough at merge time that the failure mode is not self-correcting.
  • Even very powerful future models will not solve this, because the limit is informational, not capability-based — the why is not encoded in the code that the model has access to.
Inferences:
  • The capability gap between model generations will not close intent debt; even a stronger model has the same informational deficit.
  • Any productivity-claim that 'better agents will solve documentation' is misframed — better agents make the fabrication more convincing.
The cost of un-externalized intent has compounded by orders of magnitude under agents, even though the underlying habit (carrying intent in your head) is unchanged. strength: high
Explicit premises:
  • Humans absorbed tacit intent slowly via shared context and accumulated it over years.
  • Agents start every session cold with no long-term memory.
  • Whatever isn't externalized, the agent simply doesn't have — every session, per agent.
  • Teams now run many agents in parallel.
Implicit premises:
  • The cost of re-supplying intent in-session is non-trivial — it shows up as 'orchestration tax' / cognitive bandwidth load on the developer.
  • Agents cannot share intent peer-to-peer; each session is an island.
Inferences:
  • The optimal frequency of writing rationale down is dramatically higher in agent-assisted workflows than in pre-agent workflows.
  • Teams whose senior-engineer-as-intent-documentation strategy was working at 4 humans / 0 agents will silently fail at 4 humans / 20 agents — without anyone making a decision to change.
  • The 'orchestration tax' Osmani named previously is partly intent-debt servicing in disguise.
The objection 'you can't capture all intent in a spec' does not defeat the paydown program — it specifies it. strength: medium
Explicit premises:
  • A spec detailed enough to BE the program is just the program in a slower language (Osmani's prior position).
  • But not capturing all intent does not entail capturing none of it.
  • Some decisions are load-bearing — expensive to get wrong and nobody can reconstruct them later.
Implicit premises:
  • Developers can identify load-bearing decisions at the moment of decision (before knowing which ones will matter).
Inferences:
  • Paydown is a triage problem (which rationale is worth recording) not a totality problem (record everything).
  • The relevant skill is recognizing load-bearing decisions in the moment — not previously framed as a discipline but now central.
The diagnostic test for intent debt is felt-helplessness, not code-friction. strength: high
Explicit premises:
  • Technical debt is felt as friction (slow builds, fragile tests, dread).
  • Intent debt is felt as helplessness — inability to defend or reconstruct a design choice.
  • Three vignettes operationalize this: nobody can defend a removed guard clause; behavior changes pass green tests because the tests encoded behavior not intent; an architectural choice is justified as 'an agent suggested it and it seemed fine'.
Implicit premises:
  • The phenomenology of helplessness is reliable enough that developers can use it as a real-time signal.
  • Code review at agent speed is where this helplessness becomes detectable, because that's where the question 'why was this here?' is asked.
Inferences:
  • Code review against agent output is the diagnostic surface for intent debt — and is therefore where the metric should live.
  • A repo could be instrumented for intent debt by recording how often reviewers cannot defend a piece of code under question.
The remedy is personal: spec for intent, AGENTS.md as intent ledger, decision logs, self-improving learnings files — all individual-workflow disciplines. strength: low
Explicit premises:
  • Externalized intent is the only paydown mechanism.
  • Four artifact formats already exist that can carry externalized intent.
  • Agents make logging cheaper than it has ever been.
Implicit premises:
  • Developers will choose to perform these disciplines if they understand the diagnosis.
  • Organizations will not actively pressure against externalization (e.g., by measuring throughput in merged code).
  • Tooling vendors will not actively work against externalization (e.g., by shipping /init flows that generate descriptive AGENTS.md rather than intent-bearing AGENTS.md).
Inferences:
  • If organizational and vendor pressures DO work against externalization (and they often do), the individual-discipline frame will systematically fail.

Questions addressed

  • ✅ What is intent debt and how does it differ precisely from technical debt and cognitive debt?
    Intent debt is the absence/erosion of EXTERNALIZED rationale in artifacts. It is independent of code quality (technical) and of personal understanding (cognitive). The three are orthogonal; they 'feel similar from the inside but are not the same bill'. The defining feature is artifact-locus: rationale that exists only in human heads is high intent debt regardless of how well-understood the system is.
  • ✅ Why can't AI agents pay down intent debt the way they can pay down technical (and partially cognitive) debt?
    Because intent is the one input that has to originate with a human — it is informational, not capability-limited. An agent can fabricate plausible-sounding intent from the code, but a fabrication is structurally worse than the agent admitting ignorance, because at merge time the two are indistinguishable. Stronger future models do not close this gap.
  • ✅ Why is intent debt urgent now, when teams have managed it informally for decades?
    Agents start cold every session with no long-term memory; un-externalized intent used to be paid occasionally (onboarding, departures) and now is paid every session, per agent. The senior-engineer-as-intent-documentation strategy collapses when half the team is agents.
  • ✅ How can the reader detect intent debt in their own system?
    By the felt experience of helplessness rather than friction. Three vignettes operationalize: inability to defend a guard clause's existence, behavior change that passes review because tests encoded behavior not intent, architectural choices justified only as 'an agent suggested it'. The diagnostic surface is code review against agent output.
  • ✅ How does intent debt relate to the earlier critique that specs cannot fully capture implementation?
    Both are true simultaneously: a spec detailed enough to BE the program is just the program in a slower language AND the fact that you cannot capture all intent does not license capturing none of it. The reframe is triage: identify load-bearing decisions and externalize their rationale, not totality.
  • ✅ What are the operational levers for paying intent debt down?
    Four: (1) write specs for intent not implementation, (2) treat AGENTS.md as intent ledger not /init-generated config, (3) capture decisions as lightweight ADRs at decision-time, (4) self-improving learnings loops that write intent back into the repo at session-end.
  • ⭕ Whose problem is intent debt — individual developers, team leads, tooling vendors, or organizations?
    Osmani addresses the individual-developer level exclusively. He gestures at vendor-level pressure (the '/init' default, AGENTS.md-as-config) but does not propose vendor-level remedies. He does not address organizational incentives that might reward visible agent activity over externalized intent — even though his own diagnosis implies they would. This is the third post in his series where the diagnosis is structural and the remedy stays at the individual-practice level.
  • ⭕ How does the load-bearing-rationale triage actually work — how do you identify in the moment which rationale will matter later?
    Asserted as the resolution to the 'can't capture everything' objection but not operationalized. No criterion is offered for distinguishing load-bearing from incidental decisions at the moment of decision. The implicit assumption is that developers can do this reliably — an unobvious assumption.
  • ⭕ Are AGENTS.md, ADRs, learnings files and specs actually adequate substrates for externalized intent, or just the artifacts that happen to exist?
    Unaddressed. The prescriptions assume the format question is settled. But intent has rich structure (constraints with priorities, non-negotiables vs preferences, historical contingencies vs principled stances, decisions whose conditions might change vs immutable) and current artifact formats are flat. Whether they can actually carry the load is not interrogated.
  • ⭕ What about tests that encode intent, not just behavior — is there a fourth paydown lever Osmani hints at but does not develop?
    The second vignette ('the tests never encoded the intent, only the previous behavior') implies intent-specifying tests are possible and would catch a failure class neither prose nor ADRs would catch. But the prescription section moves on to prose-form artifacts (specs, AGENTS.md, ADRs, learnings) without developing the test angle.
  • ⭕ What happens to intent debt at the codebase / multi-team scale — are AGENTS.md and ADRs sufficient when intent is distributed across services, teams, and time?
    Unaddressed. The prescriptions are local-to-repo; cross-cutting intent (architectural principles that span services, security postures that span teams, business invariants that span codebases) is not engaged.

  • illogicalThe 'agents cannot generate intent because intent is the one input that has to come from a human' premise is treated as structural / informational, but it is asserted rather than argued. A strong-form counterargument exists: agents COULD acquire intent by observing the human in the moment (commit-time prompting: 'what's the rationale for this change?'), by behavioral telemetry (what does the developer test, what do they NOT test, what do users complain about), or by negotiated dialogue (the agent proposes possible intents and the human selects). Osmani's claim that agents can only 'fabricate' intent presupposes a no-interaction model. The interaction-rich model is exactly where the future is heading. Without engaging it, the structural claim is weaker than it sounds.
    Read next: Recent harness-engineering writing (Birgitta Bockeler, already in your library) on interactive intent elicitation. The 'plan first' / 'specs first' agent patterns in Claude Code / Cursor / Codex. Recent multi-agent-systems literature on negotiated specification. Greg Brockman / OpenAI on commit-time prompting flows.
  • incompleteThe series-level pattern is itself worth naming. Comprehension debt → orchestration tax → intent debt is now a tightly linked triptych, and Osmani treats them as cumulatively framing one diagnosis. But he never names the underlying disease. The candidate name — and the missing post — is something like 'externalization debt' or 'legibility debt': all three debts are downstream of a single failure to make the relevant signal (code, mental model, rationale) readable by something other than the senior engineer's head. The post that names that underlying disease, distinguishes it from the symptom-class, and asks what an externalization-first development culture looks like, has not been written. For Kjetil tracking the series, that is the post to ask Osmani for next.

Don Norman 'Things That Make Us Smart' on cognitive artifacts. Andy Clark on extended mind. Possibly David Weinberger 'Everything is Miscellaneous' on the conditions under which knowledge externalizes well.
