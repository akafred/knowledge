Podcast: Formal Methods for Every Engineer in an AI-Powered Future

Gabriela Moreira (guest); host Shane Hastie (InfoQ) · podcast · 2026-07-10 Reader 🖥️

Formal methods are a practical tool for reasoning about the edge cases of complex distributed and concurrent systems, and because AI in 2026 collapses the historic barriers to writing specifications and the glue code that binds them to real implementations, the profession's enduring human work shifts to defining which system behaviours are correct — the judgment AI cannot take over.

Practical · Mixed — an expository interview that is also argumentative (defending formal methods as a practical, human-essential discipline) and carries two embedded narratives (Moreira's Byzantine-fault-tolerance learning story and the AWS DNS-outage case). — Conversational podcast: claims are stated informally, restated and drawn out through Shane Hastie's questions. Almost all propositions are Moreira's; Shane contributes one load-bearing framing move (spec-code drift is an age-old problem). Moreira explicitly rejects the framing of formal methods as an academic responsibility in favour of 'a solution to a problem you already have', which fixes the classification as practical.

Key terms

  • Formal methods Formal Methods
    A practical family of techniques (specification, model checking, model-based testing, trace validation) for precisely reasoning about, and checking, the behaviours of complex systems — framed as a solution to an edge-case problem, not an academic obligation.
  • Quint Formal Methods
    A new specification language Moreira has built for four years, based on TLA+ but stripped of the parts that were too hard for newcomers — TLA+'s power made accessible; now a spin-out she leads as CEO.
  • TLA+ Formal Methods
    A mathematical formal-specification language for defining what a system can do and the properties it should hold; expressive but hard to learn, and the ancestor of Quint.
  • Specification / model
    A state machine describing what a system or solution can do; you define initial states, transitions, and properties, then run it to observe behaviours — deliberately more abstract than the code.
  • Model checking Formal Methods
    Automatically exploring a specification's state space to find behaviours, verify properties, and surface corner cases — the act of 'running' the spec where most of the value lives.
  • Model-based testing (MBT) Formal Methods
    Deriving tests from the model: replaying behaviours from the spec against the real implementation and checking that the resulting states match step-by-step; can use random or targeted (witness) behaviours.
  • Glue code / harness code
    The replay code that links a model to an implementation for model-based testing — tedious and annoying to write and maintain, historically the main reason MBT never caught on, and exactly what AI is good at generating.
  • Trace validation Formal Methods
    Checking real production execution traces (via logging/instrumentation) against the spec to ask 'is this compliant with what you said?' — the frontier for connecting formal specs to runtime monitoring.
  • Witness
    A property you deliberately expect to break in order to witness an interesting or rare behaviour; used to challenge the tools ('the most cornery corner case I can come up with') and to gain confidence that the tools find scenarios at least as well as you can.
  • Failure modes
    Explicitly modelled external failures (and the system's response to them) — needed because a system's whole point may be to recover from failure; modelling them reveals 'which ways does your software break' when failures exceed tolerance.
  • Reproducible behaviours
    Specific modelled scenarios (like 'process B accesses X while process A holds the lock') written down executably — whiteboard reasoning made concrete without sacrificing abstraction, usable as documentation, in PR descriptions and on CI.
  • Corner case / edge case
    The rare event sequences in complex systems that developers are expected to anticipate 'in their heads' but cannot — the core problem formal methods exist to address.

Arguments

AI does not threaten the core of the software profession, because defining which behaviours are correct remains essential, non-delegable human work. strength: high
Explicit premises:
  • AI may write much or even all of the code, and non-technical product people can already build software where safety and complex behaviours are not major concerns.
  • But a large class of software has complex behaviours (concurrency, event orderings) that require technical judgment.
  • Defining what we want and which behaviours are correct or incorrect is what formal methods let engineers capture precisely.
Implicit premises:
  • Correctness is defined relative to human intent, which an AI cannot originate on its own — it can generate a system but not the criterion by which the system is judged good.
  • Understanding a system is a human requirement that persists even when the code is machine-generated.
Inferences:
  • Therefore the value and the 'fun part' of the profession migrate to specifying and verifying correctness, and are preserved rather than removed.
Model-based testing (and formal methods generally) is now practically feasible in a way it never was before. strength: medium
Explicit premises:
  • Model-based testing requires glue code that links a spec to an implementation.
  • That glue code is tedious and annoying to write and maintain, which is the main reason MBT never caught on.
  • AI is excellent at generating exactly this kind of trivial, boring connective code — plus instrumentation and log parsing.
Implicit premises:
  • The AI-generated glue code (and the AI-generated spec) is faithful and correct enough to be trusted as the link between model and reality — i.e. the interface guardrails plus the triviality of the code make its correctness a non-issue.
Inferences:
  • Therefore the historical barrier to model-based testing is removed and the technique can be integrated into the ordinary developer lifecycle.
Formal specifications must be kept continuously connected to the running code or they are worse than useless. strength: high
Explicit premises:
  • Code drifts away from a correct original design over time (the AWS DNS-outage case: correct design, correct initial code, later divergence and a production bug).
  • A spec left to rot in a folder does not catch the drift-introduced bug and wastes the effort of writing it.
Implicit premises:
  • A specification delivers value only as an active, checked artifact, not as static documentation.
Inferences:
  • Therefore spec-code checking must be wired into testing, CI and even production monitoring (via model-based testing and trace validation), which AI now makes cheap enough to do.

Narratives

  • At Informal Systems, surrounded by PhDs who discussed Byzantine fault tolerance and consensus fluently, Moreira struggled to understand even basic distributed protocols. Writing a formal specification for one algorithm was not enough; only when she ran it and produced witnesses for its corner cases — finally writing one reproducible example for the algorithm's most extreme case — did she feel she truly understood it, an understanding that has stuck.
    → Grounds the claim that understanding comes from running a spec and producing witnesses, not from writing it (p11), and that confidence scales with genuine understanding rather than a check mark (p10).
  • In October (last year), AWS suffered a DNS outage. The original design was correct and the initial code was correct; the bug arose because the code diverged from the design over time. As presented at AWS re:Invent, they made their (previously informal) documentation into a formal model and used an AI tool to compare the new model against the code — the AI read both and found a divergence, which they then fixed.
    → Supports both the spec-code-drift danger (p14) and the claim that AI is a new, better solution to the age-old sync problem because it can compare model to code and surface divergence (p16).
  • Moreira's own arc: introduced to compilers and type systems during her bachelor's, pointed at TLA+ by a colleague ('I think you will like it'), she fell in love with it, built TLA+ tooling through her master's while working on event-driven distributed systems, was found and hired by Informal Systems at a TLA+ conference, spent four years building Quint (an accessible TLA+), and is now spinning Quint out as CEO.
    → Establishes credibility and motivates the accessibility mission — the conviction that formal methods 'should be used more' and made easy enough for newcomers (frames p1 and p3).

Questions addressed

  • ✅ Why should software engineers care about formal methods?
    They already do, implicitly: formal methods are reframed as the solution to an edge-case problem engineers already face, not a responsibility to take on.
  • ✅ How is formal methods different from just good test design?
    Formal specs work at a higher abstraction that can reason about concurrency/timing/scheduling tests cannot control, and the tools synthesize scenarios for you instead of requiring manual enumeration.
  • ✅ How do I get started and integrate this into my workflow?
    In 2026: ask an AI to generate a first Quint/TLA+ spec, run and refine it, then wire it to the code via model-based testing (AI-generated glue code) and, at the frontier, trace validation against production.
  • ✅ What is the relationship between the model and my codebase, and how do you keep them in sync?
    The model is deliberately more abstract than the code; the two are kept in sync by model-based testing (replaying model traces against the code) and by trace validation (checking production traces against the model).
  • ✅ What does this mean for our profession if AI is writing the code?
    Defining which behaviours are correct or incorrect remains essential human work AI cannot take; formal methods preserve both understanding/confidence and professional value.
  • ⭕ Who verifies that the AI-generated specification and glue code are themselves faithful and correct?
    Left unaddressed: the advice is to proceed 'when you're happy with your specification', but how you gain warranted confidence in an AI-authored spec and its AI-authored glue code — when understanding was the original barrier — is never explained.

Gaps

  • incompleteThe spec-verification circularity. The whole 2026 pitch is that AI removes the barrier to WRITING the spec and the glue code. But Moreira also insists there is 'no easy check mark' and that confidence requires 'a lot of understanding'. Judging that an AI-generated spec is a faithful representation of the intended system, and that the 'trivial' AI-generated glue code is correct, still requires exactly the deep understanding that was the original barrier. The talk papers over this with 'when you're happy with your specification' and 'it's very easy for AI to generate the code without inventing anything' — never saying how a newcomer reaches warranted confidence in artifacts they did not author.
    Read next: Read against the corpus's LLM-fidelity material: llm-limitations (silent corruption, non-determinism, over-attribution) and verifiability, plus the finding that AI code which 'works' is not thereby correct or secure (extract-founders-playbook-building-ai-native-startup#p9) and that agents generating lots of software says nothing about whether it is correct (extract-ai-funhouse-mirror-velocity-debt#p3). Also extract-ai-first-software-delivery#p8 on tests fitting the code rather than the intent — the same fidelity risk one level up, at the spec.
  • incompleteThe boundary of 'essential human work' is asserted but under-drawn. Moreira says product people can safely leave AI to build software 'when safety and big concerns' are absent, while complex concurrent systems still need technical judgment — but she gives no criteria for classifying which systems warrant formal specification versus which can be left to unspecified AI generation. Without that decision procedure, 'defining correct behaviour is essential human work' cannot be operationalised: a team cannot tell where the line falls for their system.
    Read next: Read alongside risk-tiering / not-all-software-is-equal material — the autonomy-boundary and classification gap in extract-ai-funhouse-mirror-velocity-debt#g4, and the risk-tiering concept — which frame exactly the 'which changes get which rigor' judgment this talk leaves implicit.
  • incompleteTrace validation for runtime monitoring is named as the frontier ('we are going to focus on developing infrastructure for that') and simultaneously conceded to be hard and historically infeasible ('you have to orchestrate your code, there's a bunch of challenges'). How you instrument production, faithfully parse logs into behaviours, and orchestrate real code for trace validation at scale is left as future work, so the strongest version of the pitch (specs connected to production behaviour, not just tested behaviour) is promissory rather than demonstrated.
    Read next: Read against the AWS re:Invent case in full detail, and against agent/observability material (agent-observability) on capturing and interpreting production telemetry — the practical substrate trace validation would need.

Closes gaps elsewhere

  • What Survived Contact · g3 — g3 asks how to build a self-generating oracle for genuinely NEW behaviour, where no running reference implementation exists to reference an expected answer from live. Formal methods supply exactly that: instead of computing the expected answer from a pre-existing running system, the engineer authors an abstract formal model of the intended behaviour and uses model-based testing to replay the model's behaviours against the implementation — the authored model itself becomes the oracle, constructed from understanding before any reference implementation exists. (Note the mechanism difference from the referenced-oracle case: the answer is authored, not referenced live.)
