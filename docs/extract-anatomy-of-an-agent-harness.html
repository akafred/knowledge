The Anatomy of an Agent Harness

Akshay · tweet · 2026-07-08 Reader 🖥️

An agent harness is not monolithic prompt-glue but a decomposable system of roughly twelve components (orchestration loop, tools, memory, context management, prompt construction, output parsing, state, error handling, guardrails, verification, subagents) plus seven design-decision axes, and naming the parts is what lets you reason about, debug, and — because the parts are treated as separable and independently improvable — evolve agent behavior; yet as delivered the list reads as a convergent practitioner checklist organized by mechanism and vendor vocabulary rather than an orthogonal, coupling-minimizing decomposition with defined interfaces.

practical · expository — Expository/taxonomic: it defines a term (the harness), enumerates ~12 components and 7 design-decision axes, traces one execution cycle, and surveys five frameworks. It largely ASSERTS a synthesized consensus rather than argues; the two genuine mini-arguments (separate permission from reasoning; name the parts so you can debug/evolve them) are light. Prescriptive intent throughout ('the next time your agent fails, look at the harness'), so practical.

Key terms

  • agent harness Agent Harness
    The complete non-model software infrastructure wrapping an LLM — orchestration loop, tools, memory, context management, state persistence, error handling, guardrails, verification. 'If you're not the model, you're the harness.' The harness is the machinery; the 'agent' is the emergent goal-directed behavior it produces. Von Neumann framing: the harness is the operating system around the model-as-CPU.
  • orchestration loop (TAO / ReAct / 'dumb loop')
    The heartbeat: assemble prompt, call LLM, parse output, execute tool calls, feed results back, repeat until done. Mechanically 'often just a while loop'; Anthropic's is a 'dumb loop' where all intelligence lives in the model — 'the complexity lives in everything the loop manages, not the loop itself.'
  • context management & context rot
    The component that keeps the working set small and high-signal; 'where many agents fail silently.' Context rot = performance degrading 30%+ when key content lands mid-window (Chroma / 'Lost in the Middle'), even in million-token windows. Countermeasures: compaction, observation masking, just-in-time retrieval, subagent delegation; goal is the 'smallest possible set of high-signal tokens.'
  • memory (multi-timescale) Agent Memory
    State kept across turns and sessions at multiple timescales: short-term (session history), long-term (CLAUDE.md/MEMORY.md, LangGraph Stores, OpenAI Sessions). Claude Code's three-tier hierarchy (lightweight always-loaded index -> on-demand topic files -> search-only raw transcripts) with the design principle that the agent treats its own memory as a 'hint' and verifies against actual state before acting.
  • verification loops
    The component that 'separates toy demos from production agents': rules-based feedback (tests, linters, type checkers), visual feedback (screenshots via Playwright), and LLM-as-judge (a separate subagent). Cherny: giving the model a way to verify its work improves quality 2-3x. Framed (via Fowler/Thoughtworks) as guides (feedforward) vs sensors (feedback).
  • guardrails & the permission/reasoning separation
    Safety enforced architecturally separate from model reasoning: the model decides what to ATTEMPT, the tool system decides what is ALLOWED. Claude Code gates ~40 capabilities across trust/permission/confirmation stages; OpenAI runs input, output, and per-tool guardrails with a 'tripwire' that halts the agent.
  • subagent orchestration
    Spawning bounded sub-runs: Claude Code's Fork (byte-identical context copy), Teammate (separate pane, file-mailbox comms), Worktree (own git branch); OpenAI's agents-as-tools and handoffs; LangGraph's nested state graphs. Each subagent explores extensively but returns a condensed 1-2k-token summary (a context-management move as much as a topology one).
  • harness thickness / scaffolding
    How much logic lives in the harness vs the model. Scaffolding is TEMPORARY construction infrastructure, removed when the building is complete; as models improve, harness complexity should shrink (Anthropic deletes planning steps as models internalize them; Manus rebuilt 5x). The 'future-proofing test': a design is sound if performance scales with stronger models WITHOUT added complexity.
  • three levels of engineering (prompt ⊂ context ⊂ harness)
    Concentric scopes: prompt engineering crafts the instructions; context engineering manages what the model sees and when; harness engineering encompasses both plus tool orchestration, state, error recovery, verification, safety, lifecycle. 'The harness is not a wrapper around a prompt.'
  • the seven design decisions
    The axes every harness architect chooses along: single- vs multi-agent, ReAct vs plan-and-execute, context-window strategy, verification-loop design, permission/safety architecture, tool-scoping, harness thickness. A decision space layered on top of (and partly restating) the component list.

Arguments

Permission enforcement should be architecturally separated from model reasoning — the tool system, not the model, decides what is allowed. strength: high
Explicit premises:
  • The model decides what to attempt; the tool system decides what's allowed (Anthropic).
  • Claude Code gates ~40 discrete tool capabilities independently, with trust/permission/confirmation stages.
  • OpenAI's tripwire halts the agent immediately when a guardrail fires.
Implicit premises:
  • Deciding an action and authorizing it are different concerns that must not be coupled in the same nondeterministic component.
  • A model that both proposes and authorizes its own actions has no independent check, so safety cannot be a model property.
Inferences:
  • Therefore route every tool call through an out-of-model permission layer rather than trusting prompt-level instruction.
Decompose the harness into named components (and decision axes) because naming the parts is what lets you reason about, debug, and improve agent behavior.
Explicit premises:
  • Changing only the harness moved agents 20+ ranking positions on TerminalBench (identical model).
  • Two products on identical models can differ wildly based solely on harness design.
  • Named failure modes map to named components (context rot -> context management; silent tool failures -> error handling; forgetting -> memory).
Implicit premises:
  • You can only debug and independently improve a part you can name and isolate.
  • The components are actually separable — the load-bearing and undischarged assumption (see gaps).
Inferences:
  • Therefore treat the harness as a decomposable engineering artifact, and locate each failure at a specific component.

Narratives

  • Beren Millidge's 2023 'Scaffolded LLMs as Natural Language Computers': a raw LLM is a CPU with no RAM, disk, or I/O; the context window is RAM (fast, limited), external databases are disk (large, slow), tool integrations are device drivers, and the harness is the operating system — 'we have reinvented the von Neumann architecture.'
    → The conceptual anchor — makes the harness's role precise by mapping each component onto a known computing-architecture part, and justifies why a harness is a natural, not incidental, abstraction.
  • LangChain changed only the infrastructure wrapping their LLM (same model, same weights) and jumped from outside the top 30 to rank 5 on TerminalBench 2.0; a separate research project reached 76.4% pass rate by having an LLM optimize the infrastructure itself, surpassing hand-designed systems.
    → The empirical hook that the harness — not the model — drives benchmark rank; motivates naming and engineering the harness. (Also relayed in the sibling as corroboration.)
  • Claude Code's memory is a three-tier hierarchy — a lightweight ~150-char-per-entry index always loaded, detailed topic files pulled in on demand, and raw transcripts reachable only via search — and a critical principle: the agent treats its own memory as a 'hint' and verifies against actual state before acting.
    → A component done well: shows memory as a progressive-disclosure structure plus the anti-hallucination discipline (verify the hint), i.e. memory and context management working together.
  • Manus was rebuilt five times in six months, each rewrite removing complexity: complex tool definitions became general shell execution; 'management agents' became simple structured handoffs.
    → Demonstrates that evolving a harness includes SHRINKING it as models absorb capability — the concrete basis for the scaffolding-removal / future-proofing stance.

Questions addressed

  • ✅ Concretely, what is an agent harness — what is it made of?
    The complete non-model infrastructure, decomposed into ~12 named components and framed via the von Neumann analogy (harness = OS) and the three concentric engineering levels (prompt ⊂ context ⊂ harness).
  • ✅ Why do agents that work in demos fall apart in production?
    The failures live in the harness, not the model — forgetting (memory), silent tool failures (error handling), context filling with garbage (context management), compounding step errors — each mapped to a specific component you can name and fix.
  • ✅ How do the components actually work together in a single cycle?
    The seven-step walkthrough — prompt assembly, LLM inference, output classification, tool execution (validate/permission/sandbox/capture), result packaging, context update (triggering compaction near limits), loop — with layered termination conditions and a Ralph Loop for multi-window tasks.
  • ✅ Should you build a thick harness or a thin one, and how should it change as models improve?
    Trend thin and shrink over time: scaffolding is temporary, removed as models internalize capabilities; pass the future-proofing test (performance scales with stronger models without added complexity).

Gaps

  • incompleteThe decomposition is not orthogonal and specifies no module boundaries. Four of the named components — memory (3), context management (4), prompt construction (5), and state management (7) — are views of a single 'what the model sees / what persists' concern (Claude Code's git-commits-as-checkpoints sits under state, MEMORY.md under memory, compaction under context, prompt assembly under construction, but they are one persistence-and-window problem wearing four hats). Guardrails/permission (9) is not disjoint from tools (2) — the walkthrough runs the permission check INSIDE tool execution (step 4). Output parsing (6) is near-vestigial under native tool calling (a sub-step of the loop, not a peer module). And the orchestration loop (1) is a LEVEL above the contents it manages, not a coequal component ('the complexity lives in everything the loop manages'). The prose enumerates eleven items under a 'twelve components' heading. So the list reads as a practitioner checklist organized by mechanism and vendor vocabulary, not a coupling-minimizing partition into high-cohesion modules — it cannot tell you where to actually cut the system.
    Read next: Read Parnas ('On the Criteria To Be Used in Decomposing Systems into Modules'), Yourdon & Constantine on coupling/cohesion, and Ousterhout on deep modules; re-derive the harness as narrow-interface deep modules and test which of the eleven are genuinely separable versus one persistence/context responsibility split four ways — the structured-design question the piece's naming step presupposes but never answers.
  • incompleteAnatomy without failure-mode interactions. The seven-step walkthrough gives happy-path dynamics only; the piece never addresses how components interfere when things go wrong — compaction (context management) discarding state the verifier still needed, a guardrail tripwire firing mid-tool-call and leaving state half-written, or a memory 'hint' that contradicts actual state after a context reset. The hardest engineering lives in these cross-component failure interactions, which the static list and the clean walkthrough elide.
    Read next: Read extract-anthropic-harness-design-long-running-application-development and extract-anthropics-three-agent-harness on context-reset handoff artifacts and evaluator calibration; look for how components must expose contracts to survive each other's failures rather than only cooperating on the happy path.
  • incompleteNo interface or contract between components is specified. The piece names components but never states what each EXPOSES to the others — what memory hands to context management, what guarantees the loop makes to the tool layer, what verification consumes and returns. Without interface contracts, the 'independently improvable' premise (p13/p14) that the sibling's fitness-selection strategy depends on is unverified: 'add one mechanism' only stays local if the components are actually decoupled, which the piece asserts by layout rather than by design.
    Read next: Read on module interfaces/APIs (information hiding, contracts) and re-examine the sibling's automated 'add one mechanism, keep if better' loop to check whether a change to one component ripples across the coupled memory/context/state cluster.
  • incompleteTwo overlapping taxonomies whose relationship is left unstated. The twelve components and the seven decisions substantially overlap: decisions 3/4/5/6 (context-window strategy, verification design, permission architecture, tool scoping) simply re-open components 4/10/9/2 as 'choices,' while only decisions 1/2/7 (single-vs-multi-agent, ReAct-vs-plan-execute, harness thickness) are architectural choices not already a component. The piece presents both frameworks without saying whether the seven decisions are just the tunable parameters of a subset of the twelve components.
    Read next: Consolidate into one model — components as nouns (what the harness contains) versus configuration axes as verbs (how each is set) — and test whether 'seven decisions' collapses into 'the tunable dials on four of the components,' leaving only single-vs-multi-agent, ReAct-vs-plan, and thickness as genuinely first-order architecture.
  • incompleteSecond-hand synthesis with no first-party build and no eval of the taxonomy itself. Every component and framework claim is assembled from vendor docs, and the empirical figures (30%+ context rot, 2-3x verification, 3.6x LLMCompiler, TerminalBench 20-rank jump, 76.4%, 80% tool removal, 26-54% token reduction) are all relayed. There is no worked minimal harness, no ablation showing these components are necessary or sufficient, and no test of the '12' as a complete or minimal set.
    Read next: Build a minimal harness against the eleven components and ablate each to test necessity/sufficiency; go to the primary sources (Chroma context-rot study, the LangChain TerminalBench writeup, the LLMCompiler and ACON papers) before treating the relayed figures as load-bearing.

Closes gaps elsewhere

  • Agent Harness Engineering · g1 — PARTIAL. Osmani g1 asks when harness complexity itself becomes a liability — the scaling problem of accumulating ratchet rules, and at what point the constraint set is too large. The anatomy supplies a directional criterion: the 'future-proofing test' (a design is sound only if performance scales up with more powerful models WITHOUT added harness complexity) plus the scaffolding-shrinks stance (Manus rebuilt 5x removing complexity; Anthropic deletes planning steps as models internalize them). So complexity that a stronger model would render redundant is, by this test, a liability. It does NOT answer the quantitative half of g1 — no threshold for 'too large for the model to follow / the engineer to maintain,' no maintenance-cost model — so this is a design principle, not the boundary the gap asks for.
