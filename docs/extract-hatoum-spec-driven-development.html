Spec-driven development with AI is not waterfall because waterfall's fatal flaw was slow feedback, not specifications themselves, and AI collapses the feedback loop from months to minutes — making specs more essential, not less.

practical · argumentative — A short LinkedIn post rebutting two common objections to spec-driven development with AI. Primarily argumentative — identifies the objections, diagnoses their shared error, and makes a case for spec-first workflows under new AI-driven feedback dynamics.

Key terms

  • spec-driven development Spec-Driven Development
    A workflow where human-authored specifications are the primary artifact, and AI handles implementation and test verification against those specs in a rapid feedback loop
  • feedback speed / feedback loop Feedback Loops
    The elapsed time between expressing intent (writing a spec) and learning whether the resulting implementation is correct
  • amnesiac AI Spec-Driven Development
    An AI system that does not retain decisions, context, or rationale across sessions — so without specs, accumulated design intent is lost

Propositions

  • explicitWaterfall's fundamental problem was slow feedback, not the use of specifications.conf: high
  • explicitBig upfront design fails for the same reason as waterfall: the loop between intention and reality is too long to allow timely correction.conf: high
  • explicitcontestedSpec-driven development with AI collapses the feedback loop from months to minutes.conf: high
    contradicts ←It Ain't Broke: Why Software Fundamentals Matter More Than Ever · p2 — Hatoum claims the fast spec-to-implementation loop is the win; Pocock claims that same loop produces progressively worse code each iteration due to software entropy.
  • explicitTDD, BDD, and acceptance criteria are spec-first practices that are accepted as agile because their feedback loops are tight.conf: high
  • explicitIf implementation became truly instant, 100% of human effort would go to specification/design.conf: medium
    extendsPresentation: AI-First Software Delivery: Balancing Innovation with Proven Practices · p6 — Generalizes the empirical 10:1 spec-to-code effort ratio to its limiting case: with instant implementation, 100% of human effort goes to specification and design.
  • explicitThe constraint on batch size has moved from implementation speed to verification capacity — how much spec you can meaningfully verify, review, and correct in one cycle.conf: high
  • explicitWithout explicit specs, you are trusting an AI that cannot retain decisions across sessions to hold your entire design intent.conf: high
  • implicitWhether a development practice is 'waterfall' or 'agile' is determined by feedback-loop speed, not by whether specifications exist.conf: high
  • implicitSpecs serve as durable external memory for design intent — a function that becomes more critical when the implementer (AI) has no persistent memory.conf: high
  • implicitcontestedAI implementation quality is good enough that the spec-implement-verify cycle can run in minutes without human coding intervention.conf: medium
    contradicts ←[FULL WORKSHOP] AI Coding For Real Engineers - Matt Pocock, AI Hero · p5 — Pocock insists the code is the developer's battleground and specs-to-code does not work, directly denying Hatoum's claim that the spec-implement-verify cycle can run without human coding intervention.

  • Waterfall's problem was feedback speed, not specifications (P1)

Implicit premises:
  • The defining characteristic of waterfall is slow feedback, not the presence of upfront design artifacts (P8)
  • AI implementation is reliable enough that the rapid loop actually works (P10)

Narratives


  • → Reductio ad absurdum to show that as implementation cost approaches zero, spec work is all that remains — and calling that 'waterfall' would be absurd.

Questions addressed

  • ⭕ Is spec-driven development with AI just a return to waterfall?
  • ⭕ Does big upfront design work when AI does the implementation?
  • ⭕ Why should developers write specs when AI can just generate code?

Gaps

  • incompleteThe post assumes AI implementation is reliable enough to close the loop in minutes but provides no evidence for this. Real-world AI coding involves debugging, hallucination, and rework cycles that could lengthen feedback well beyond minutes.
  • incompleteNo discussion of what a 'spec' actually looks like in this workflow — the granularity, formalism, and format of specs are left entirely unspecified. The practical difficulty of writing good specs is not addressed.
    closed by[FULL WORKSHOP] AI Coding For Real Engineers - Matt Pocock, AI Hero — Gives a concrete stance on spec granularity and format: the PRD is a disposable alignment artifact ('just a hint'), not worth optimizing past the alignment point, and should be closed in an issue tracker rather than kept in-repo.
  • incompleteThe 'amnesiac AI' framing assumes no progress on persistent AI memory (RAG, project context, long-term memory). If AI memory improves, the strongest argument for specs (external memory) weakens — though the argument from verifiability would remain.
  • illogicalThe thought experiment ('if coding became instant') proves too much — it assumes zero implementation cost, but actual AI implementation is not zero-cost. The gap between 'minutes' and 'instant' may matter more than the post acknowledges.
