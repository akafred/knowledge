Presentation: The Multi-Agent Approach: Building Reliable and Controllable Software Development Automation

Itamar Friedman · rss · 2026-07-08 source ↗ Reader 🖥️

Agent Governance (Internal)Code Was Never the BottleneckContext EngineeringEvalsGuardrailsMulti-Agent OrchestrationPull Requests & Async Code Review
To break through the AI 'productivity ceiling' in software development you must stop pouring investment into faster code generation and redirect it toward verification and context — building an adaptive, quality-grounded multi-agent system with an arbiter to resolve agent disagreements, human-observable inter-agent communication, first-class governance, and continuous learning of what quality means for your organization.

practical · Practitioner conference talk (QCon AI / InfoQ presentation) with audience Q&A — a how-to argument for architecting AI software automation, grounded in vendor and third-party survey data plus product demos. — Spoken transcript of a live talk by Qodo's founder/CEO; propositions are stated informally and restated across the arc, then normalized here. The speaker is also selling a product (Qodo), so many statistics are vendor-sourced — treated as evidence with an interest caveat, not as neutral measurement.

Key terms

  • Arbiter / arbitration
    A designated authority (agent or human) that decides which output is correct when agents disagree — rival implementations of the same prompt, a planning agent vs a coding agent, or a testing agent vs a coding agent. Friedman insists it can only adjudicate if the plan/spec carries enough context, so the arbiter first checks that the ticket is complete enough to arbitrate.
  • Context engine
    No longer a RAG lookup but an agent in its own right (ingestion, queue and retrieval sub-agents) specialized in fetching the right multi-source context — code versions, PR history, in-PR review knowledge, org rules, docs and logs — with a meaningful share (~8%) coming from non-code sources like ARCHITECTURE.md.
  • Productivity ceiling / glass ceiling
    Successive limits on how much value AI dev tooling delivers (plotted as gain on the y-axis vs investment on the x-axis); each is broken only by investing in the next capability layer, and even code-review/quality workflows hit a further ceiling that demands an adaptive, quality-grounded multi-agent system.
  • Multi-agent vs workflow
    Friedman deliberately declines to distinguish 'agentic' from 'workflow' — both use LLMs; his concern is orthogonal: making whichever you use reliable and controllable.
  • Gateway and gatekeeper
    His dual definition of code review's purpose: a gateway where the team shares information, learns and collectively owns the code, and a gatekeeper that governs and enforces quality at both code and process level.
  • Viable coding / responsible vibe coding (Vibe-DD)
    The target state that keeps developer flow at the center (unlike TDD's correctness focus or SpecDD's behavior focus) but adds proactive rigor, correctness and ownership through always-on quality workflows — moving from 'vibe coding' to 'viable coding'.
  • Agentic quality gate ('agentic CI/CD')
    Putting LLMs/agents into CI/CD quality gates — which he concedes is partly rebranded CI/CD, but argues the real opportunity is to break the one-way left-to-right flow so review and tests can apply to the plan, not just the code.
  • Software DB
    His name for the combined RAG/context-engine plus rules/best-practices store in the reference architecture — the place where the system's learned standards and specs live.

Arguments

Redirect the majority of AI-SDLC investment from code generation to verification (testing + review) and the context engine. strength: medium
Explicit premises:
  • AI code generation roughly triples the amount of code produced.
  • AI generates approximately the same rate of defects per line, so issues scale with code volume (METR/Sonar/Faros/Qodo reports: 42% more developer time on bug-fixing, ~35% of projects delayed, 3x security incidents).
  • Teams that use AI for testing and for code review report markedly higher productivity (up to 47% more) and ~2x better code quality.
Implicit premises:
  • The value of code is realized only after it is verified and shipped, not at the moment it is written, so relocating the bottleneck to review yields no end-to-end gain unless verification also scales.
  • You cannot make a non-deterministic generator reliable by instructing it better, so reliability has to be manufactured in a separate verification layer.
  • Marginal investment produces more end-to-end velocity in verification and context than in additional generation capacity.
Inferences:
  • If output triples but defect-rate-per-line holds, total defects triple; if human review capacity is fixed, the binding constraint moves downstream to verification; therefore the highest-leverage investment is downstream, not upstream.
A production multi-agent SDLC structurally requires a designated arbiter, plus investment in complete specs. strength: medium
Explicit premises:
  • Agents disagree in practice: seven runs of one prompt give seven implementations; two coding agents can touch the same file; the coding agent can disagree with the planning agent and the testing agent with the coding agent.
  • Shifting review and testing left ('doubling down' earlier) improved quality but did not resolve who decides — the Cursor-vs-Qodo 'who is right?' dispute remains open.
  • In the demo, the arbiter had to first confirm the ticket contained enough information (e.g. which repo) before it could arbitrate.
Implicit premises:
  • A multi-agent system without a conflict-resolution authority produces unresolved contradictions that stall autonomous progress.
  • Correct arbitration requires a shared ground truth to check against, so absent a sufficiently complete spec the arbiter cannot function.
Inferences:
  • Because disagreement is inherent and unresolved disagreement blocks automation, a decision authority is necessary; because that authority needs ground truth, spec completeness is a precondition for it.
Rule and quality enforcement must move to a deterministic 'must-check' review agent. strength: medium
Explicit premises:
  • A Qodo survey found Cursor follows written rules only inconsistently (no respondent said 'completely followed'); the same prompt yields different implementations each run.
  • Humans cannot review at AI volume — ~80% of PRs get no human review, and reviewers give thorough feedback only on small diffs (the LGTM-by-size pattern).
  • 70% of PRs carry a high-severity issue, a rate that grows as AI adds more PRs.
Implicit premises:
  • If the generator will not reliably obey rules and humans cannot scale review, the only remaining enforcement point is an agent that treats each rule as mandatory.
  • A guardrail that is followed only probabilistically is not a guardrail.
Inferences:
  • Given unreliable generation and collapsed human review, deterministic per-rule checking in a review agent is the sole enforcement mechanism left.
Inter-agent communication should be designed to be human-observable. strength: medium
Explicit premises:
  • If agents communicate only via MCP or pub/sub without commits, you must dig through BigQuery/logs to reconstruct which agent did what.
  • Recording each agent's turn in GitHub PRs and commits lets a human watch the exchange (Qodo suggests in a draft PR; Cursor reacts and commits via a defined workflow).
Implicit premises:
  • Human oversight and accountability are non-negotiable and require legible traces.
  • Opacity of agent-to-agent channels erodes exactly the governance that makes autonomy safe.
Inferences:
  • Since oversight requires legible traces and opaque protocols hide them, the communication design should route agent exchanges into a human-reviewable surface.

Narratives

  • Friedman opens by asking the audience to raise hands for using one, two, three, four AI dev tools — counting Figma emitting code, Jira planning help, an SRE agent — then asks whether they will use more than three code-generating tools within a year.
    → Grounds the whole talk in the audience's lived reality that tool proliferation is already here, so coordination and control are the real problem.
  • More than three cloud outages occurred in recent months; some of the affected companies say 10%, 30%, even 50% of their code is AI-generated — he explicitly disclaims a direct causal link but asks the audience to 'reflect on that'.
    → Motivates the reliability stakes of AI-generated code without overclaiming causation.
  • Anthropic's published security-reviewer prompt essentially says 'from now on you are a senior security engineer, good luck', with a very long exclusion list that includes DDoS.
    → Illustrates naive single-agent 'vibe reviewing' and the inadequacy of prompt-only quality control.
  • A Qodo survey asked whether Cursor follows written Markdown rules; no one answered 'completely followed'. The familiar experience: give a proper prompt with rules, run it seven times, get seven different implementations.
    → Evidence that generation is probabilistic about rules — the setup for deterministic 'must-check' review agents.
  • A Qodo developer used Cursor's new web-app feature to change a ChatGPT-like app (move a button, edit text, remove a field) with no code written by hand — done well and quickly.
    → Concedes that code generation is genuinely magnificent, even on heavy-duty software.
  • For that same 'simple' change, a computer-use testing agent running on staging caught that one of the chat options had actually been ruined — a regression in an element Cursor was not supposed to touch.
    → Shows autonomous/computer-use testing surfacing regressions that a quick human review would miss.
  • Across millions of PRs, Qodo found 70% contained a high-severity issue, and a large share (~80%) had no human review — partly because only ~60% of teams discuss reviews on GitHub/GitLab/Bitbucket, partly the LGTM-by-diff-size meme.
    → Quantifies the collapse of human review and the resulting quality exposure, motivating AI review + arbitration.
  • A Google DeepMind paper tested MAS architectures (centralized, decentralized, independent, hybrid) across financial and software scenarios and concluded there was 'no advantage'; Friedman jokes 'you just need one agent, prompt it differently — I'm kidding, at least partially', then mines the paper for its decision dimensions.
    → Pre-empts the strongest skeptic's evidence and converts it into his framework of considerations (parallelizability, interfaces, arbiter, cost).
  • A tweet where 'Cursor didn't know what to do and Qodo butted in' — and the author asked ChatGPT who was right, Cursor or Qodo.
    → Dramatizes the unsolved arbitration problem: when two agents conflict, nothing yet decides.
  • In a live product demo, a developer writes a ticket and the arbiter refuses to proceed because the ticket doesn't say which repo to change; the developer pastes the repo and the checkboxes turn green.
    → Makes concrete that arbitration depends on spec completeness — the arbiter first ensures there is enough information to adjudicate later.
  • Some teams report that reaching 80% test coverage, which used to take three months, now takes one month — roughly 3x — after adopting agentic testing/review.
    → Evidence that incremental, bottleneck-first adoption of quality agents pays off.
  • Q&A: asked how the survey was built, Friedman explains it combines three surveys (Qodo's own plus Sonar and Faros); they reached out to a variety of Qodo's 1,000+ clients, sought diversity and control, and stopped collecting at ~617 respondents once they saw enough diversity.
    → Discloses the provenance and sampling of the headline statistics — revealing both their basis and their vendor/self-selected limits.

Questions addressed

  • ✅ Why does AI code generation not translate into end-to-end delivery velocity?
    Because it triples code while defect-rate-per-line holds, so it multiplies issues and moves the bottleneck to review and verification rather than removing it.
  • ✅ If AI writes both code and tests, how can AI-written tests help — wouldn't a wrong model be wrong in both?
    The token and reasoning spend on testing forces thinking that improves both artifacts, just as writing tests helps a human reason, even when individual tests fail.
  • ✅ Are multi-agent systems worthless, as the DeepMind 'no advantage' paper suggests?
    No; the paper's finding is reframed as a set of design decisions (parallelizability, number of interfaces, need for an arbiter, cost) rather than a refutation — though he never actually engages the finding that architecture choice showed no advantage.
  • ⭕ Who decides when two agents disagree?
    He prescribes a designated arbiter that needs a sufficiently complete spec, but explicitly calls it 'one of the hardest things' and 'not solving the problem yet' — a direction, not a solution.
  • ✅ How should agents communicate — via protocols (MCP/A2A) or traditional pub/sub?
    Use whatever mechanism, but design it so the exchange is recorded in a human-observable UI (e.g. GitHub PRs/commits) rather than opaque channels.
  • ✅ Is the 'agentic quality gate' just rebranded CI/CD?
    Partly yes, but the opportunity is to break the one-way left-to-right CI/CD flow so review and tests can apply to the plan, not only the code.
  • ⭕ How do you mature AI adoption from small companies (where it dominates) to the enterprise?
    Raised as the key question but answered only indirectly — via quality workflows, governance and incremental bottleneck-first adoption — without an enterprise-specific playbook.

Gaps

  • incompleteThe arbiter is asserted as necessary but its decision procedure is never specified: when the planning agent and coding agent (or two rival implementations) conflict, how does the arbiter actually choose — confidence scores, voting, running the tests, escalation to a human? Beyond 'have a complete enough spec' and 'have enough context', the adjudication mechanism is a blank.
    Read next: Read multi-agent orchestration and LLM-judge/evaluator literature (e.g. Anthropic's three-agent harness and its 'what if the evaluator is wrong' failure mode) and consensus/adversarial-review patterns.
  • incompleteNearly every quantitative claim (2x code quality, 47% more productivity from AI review, 70% of PRs high-severity, ~80% no human review, 88% context-attribution) comes from vendor-run or vendor-adjacent surveys — Qodo's own plus Sonar and Faros — sampled from 'at least half of our [1,000+] clients', i.e. self-selected AI-tool users, n≈617, with no independent baseline or controlled non-AI comparison. The figures are correlational and commercially interested, not causal measurement.
    Read next: Read independent longitudinal/controlled measurement (METR's randomized study, full Faros methodology, DORA) and treat the vendor figures as directional, not established.
  • incompleteThe rebuttal of the DeepMind 'multi-agent shows no advantage' paper is promised and then sidestepped: he reframes it into his four considerations without ever engaging the actual finding that architecture choice (centralized/decentralized/independent/hybrid) produced no advantage. Whether his adaptive quality-grounded MAS would beat a single well-prompted sequential agent is left unproven.
    Read next: Read the DeepMind MAS paper directly and the multi-agent-skeptic case (Beck's 'nobody wants agents') to judge the claim on its own evidence.
  • incompleteCost/economics of running the full parallel quality stack (multiple testing, review, context and arbiter agents on every change) is named as 'very important' and then explicitly deferred to last and never quantified — no token-cost figure, no comparison of the multi-agent quality stack's cost against the productivity it recovers.
    Read next: Read AI inference unit-economics and the token-cost gaps flagged for adversarial multi-agent review workflows.
  • incomplete'Continuous learning' of your quality standards/specs is asserted as the moat, but the mechanism is unspecified: who curates the learned rules and best practices, how is drift or poisoning of the learned standards prevented, and how does the system avoid ossifying bad conventions it observed in old PRs?
    Read next: Read on knowledge curation for agent memory and organizational-legibility of standards (who owns the rule set, how it is reviewed and versioned).
