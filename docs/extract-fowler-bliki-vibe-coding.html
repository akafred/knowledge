Bliki: Vibe Coding

Martin Fowler · rss · 2026-05-21 source 🖥️

Agent HarnessAgentic EngineeringAgile & XPCode QualityLethal TrifectaLLM LimitationsMalleable SoftwareRiding Model ImprovementsVibe Coding
Vibe coding — building software by prompting an LLM without ever looking at the generated code — is a distinct practice from agentic programming, useful only for disposable software, because forgetting the code exists produces serious risks in security, maintainability, and correctness.

practical · expository — A bliki definitional entry. Practical because its ultimate purpose is to guide action: how readers should use vibe coding (only for disposable software) and how they should talk about it (don't conflate with agentic programming). Expository in form — Fowler defines a term, distinguishes it from a neighbouring concept, and enumerates risks — rather than constructing a sustained argumentative case.

Key terms

  • Vibe coding Vibe Coding
    Building software by prompting an LLM and iterating on the result *without ever reading the generated code*. The defining feature is 'forgetting the code exists.'
  • Agentic Programming Agentic Engineering
    Letting an LLM (often an agent) write most or all of the code, but the human still cares about the code: reviews it, attends to its internal structure.
  • Semantic Diffusion Agile & XP
    Fowler's own coined process by which a term's precise original meaning is diluted as it spreads through wider usage.
  • November Inflection Riding Model Improvements
    Fowler's label for the moment (late 2024) when LLM-driven coding crossed a usefulness threshold for serious programmers.
  • Lethal Trifecta Lethal Trifecta
    A specific class of agentic-AI security vulnerability (linked, not redefined here); invoked as the canonical risk non-programmers must understand.
  • Harnesses Agent Harness
    The surrounding tooling/scaffolding around an LLM that shapes what it can do (linked to his harness-engineering article).
  • Disposable software Malleable Software
    Software built for a limited audience (often just the author or a small group) where low quality, weak maintainability, and accepted risk are tolerable.

Arguments

Vibe coding is a distinct practice from agentic programming and the distinction is worth preserving. strength: high
Explicit premises:
  • Vibe coding's defining feature is forgetting the code exists.
  • Many programmers now have LLMs write all their code but still review it and attend to its structure.
  • Those two practices are different to use and different in their consequences.
Implicit premises:
  • Distinctions in practice that produce different consequences deserve distinct terms.
  • Semantic Diffusion is a real and corrosive force on technical vocabulary.
Inferences:
  • Discourse about 'AI coding' that fails to make this distinction will produce muddled empirical claims (e.g. about productivity, quality, risk).
  • Studies of 'vibe coding' productivity that include code-reviewing developers are not measuring vibe coding.
Vibe coding should be confined to disposable software. strength: high
Explicit premises:
  • Vibe coding produces security risks (Lethal Trifecta, credential exposure).
  • It produces large volumes of low-quality code that is hard to evolve.
  • LLM hallucination produces subtly incorrect behaviour.
  • LLM non-determinism causes regressions even in unchanged areas.
Implicit premises:
  • The cost of these risks is acceptable when the software's blast radius is limited to its author or a close group.
  • Authors of vibe-coded software can correctly anticipate their software's eventual blast radius.
Inferences:
  • Vibe coding is a poor fit for any software that touches credentials, customer data, or regulated environments.
  • Tooling or social practices that prevent vibe-coded software from escaping its disposable context would meaningfully reduce its risks.
Well-structured software remains valuable in an LLM-heavy world. strength: medium
Explicit premises:
  • Vibed code is hard for LLMs to modify and enhance later.
  • Empirically, well-structured code currently makes life easier for LLMs too.
Implicit premises:
  • Human and LLM consumers of code share enough cognitive constraints that structures helpful to humans are also helpful to LLMs.
  • Today's empirical pattern is a reasonable predictor of the near-term future.
Inferences:
  • Investments in code structure and design discipline are not made obsolete by capable LLMs.
  • Engineering leadership that defends design quality has a continued, possibly amplified, role.

Narratives

  • Karpathy's X post: he talks to Cursor Composer via SuperWhisper, 'Accepts All' without reading diffs, copy-pastes error messages back to the model, lets the code grow beyond his comprehension, and treats the whole thing as amusing weekend play.
    → Both attributes the term and instantiates its defining attitude — a working programmer demonstrating that vibe coding is play, not professional practice. It silently supports the proposition that vibe coding is only fit for throwaway work, because the originator himself frames it that way.

Questions addressed

  • ✅ What does 'vibe coding' actually mean?
    It is building software by prompting an LLM without looking at the generated code; the operative phrase is 'forget that the code even exists.'
  • ✅ How is vibe coding different from other forms of AI-assisted coding?
    Agentic programming retains human attention to the code; vibe coding abandons it. The consequences differ accordingly.
  • ✅ When is vibe coding appropriate?
    For disposable software used by the author or a close group who understand and accept the risks.
  • ✅ What are the dominant risks?
    Security (esp. Lethal Trifecta), maintainability collapse, hallucinated incorrect behaviour, non-deterministic regressions on edits.
  • ⭕ How does an author know up front that their software will remain disposable?
  • ⭕ What organisational practices or guardrails should constrain vibe coding inside a real engineering org?
  • ⭕ What happens when vibe coding is done by someone with no programming knowledge but the resulting tool quietly becomes load-bearing for a team or business?

Gaps

  • incompleteFowler offers no mechanism for detecting when 'disposable' software is silently graduating into consequential software. The disposable/consequential boundary is treated as legible to the author, but in practice it is one of the most common failure modes in software organisations — internal scripts becoming critical infrastructure.
    Read next: Cutler on institutional overload and shadow systems; classic discussions of 'tactical' vs 'strategic' programming (Ousterhout, Philosophy of Software Design); literature on shadow IT and end-user computing.
  • incompleteThe piece is silent on team and organisational implications. It treats vibe coding as a relationship between one author and one LLM. But the most consequential question for an engineering leader is what vibe coding does to teams that mix it with agentic programming: shared codebases, code review norms, ownership.
    Read next: Kent Beck's recent writing on AI and design discipline; Gene Kim's vibe coding series; Steve Yegge's manifesto; DORA's 2025 AI findings.
    closed byTrust Factory — Beck addresses the team/organizational dimension Fowler leaves out: single-player augmented development erodes trust across four vectors, and trust is the binding agent current AI-coding methodology debates ignore.
  • incompleteNo discussion of what 'forgetting the code exists' does to the author's mental model and learning trajectory. For non-programmers it bypasses learning entirely; for experienced programmers used routinely it may atrophy the skills that allow agentic programming to be done responsibly.
    Read next: Literature on automation complacency, skill decay (e.g. aviation HCI); also writings on AI's effect on programmer cognition (Marcus, Karpathy himself in other settings).
    closed byPresentation: The Ironies of A^2 I^2 (Ironies of Automation and Artificial Intelligence in Incidents) — Supplies the automation-complacency and skill-decay evidence the gap's reading direction requests: manual and diagnostic skills deteriorate when unused, capability induces complacency, and mental models erode faster than AI generates code.
  • incompleteThe security treatment is gestural — pointing at the Lethal Trifecta — and does not address the asymmetry that non-programmers, the very population vibe coding most empowers, are least equipped to recognise security trifectas in their own systems.
    Read next: Simon Willison on prompt injection and lethal trifectas; agentic-AI security taxonomies; end-user security literature.
  • incompleteNo engagement with the cultural/professional question of whether vibe coding deserves the name 'coding' at all, or whether legitimising it as a coding practice (rather than, say, 'spec-driven prototyping') invites exactly the diffusion problem Fowler is trying to push back against.
    Read next: Naur on programming as theory building; the broader 'what is programming' debate that has resurfaced post-LLM.
