Claude Code's harness should be decoupled from Anthropic's models: by running a local proxy that intercepts its API requests and routes each sub-task — background, thinking, long-context, web-search — to the cheapest capable model across many providers, with per-provider transformers normalizing each vendor's API dialect into Claude Code's Anthropic-shaped protocol, you keep Anthropic's coding harness and its updates while freely choosing, mixing, and cost-optimizing the model behind it.

practical · mixed — A tool README: how-to exposition (installation, config.json walkthroughs, transformer tables, CI recipes) wrapped around an argumentative spine — you SHOULD run Claude Code through a router that swaps the model per task. Its propositions serve 'you can/should do X', so it reads as a practical text with an implicit thesis rather than a neutral manual.

Key terms

  • Router
    The config object that maps task scenarios (default, background, think, longContext, webSearch) to a concrete 'provider,model' target; the policy layer deciding which model handles each request.
  • Provider
    A configured model endpoint — name, api_base_url, api_key, a list of models, and an optional transformer — representing one upstream vendor or local server (OpenRouter, DeepSeek, Ollama, Gemini, etc.).
  • Transformer
    A plugin that rewrites request/response payloads so a given provider's API dialect is compatible with Claude Code's Anthropic-shaped protocol (e.g. deepseek, gemini, openrouter, maxtoken, tooluse, reasoning, enhancetool); the per-provider adaptation layer.
  • Model Routing Model Routing
    Directing each request to a different model based on the kind of task — background, reasoning, or long context — rather than sending everything to one model.
  • Dynamic Model Switching
    Changing the active model mid-session on the fly via the '/model provider,model' command inside Claude Code.
  • background / think / longContext routing
    The built-in scenario categories: 'background' for cheap/local bulk work, 'think' for reasoning-heavy tasks like Plan Mode, 'longContext' for requests past a token threshold (default 60000).
  • Subagent Routing
    Pinning an individual subagent's task to a specific provider/model by placing a <CCR-SUBAGENT-MODEL>provider,model</CCR-SUBAGENT-MODEL> tag at the start of the subagent's prompt.
  • Custom Router
    A user-supplied JavaScript module (CUSTOM_ROUTER_PATH) that inspects the request and returns 'provider,model' (or null to fall back), enabling arbitrary content-based routing beyond the fixed scenarios.
  • ccr
    The command-line wrapper that launches Claude Code through the router ('ccr code'), restarts the service after config changes ('ccr restart'), and opens the config UI ('ccr ui').

Arguments

You should run Claude Code through a router that dispatches each request to the cheapest capable model per task type across many providers. strength: medium
Explicit premises:
  • Task categories within a session differ (background vs. reasoning vs. long-context vs. web-search).
  • Providers offer different price/capability points, and a smaller or local model suffices for background work.
  • Transformers make any provider's API compatible with Claude Code's protocol, so the choice of model is unconstrained.
  • The router runs headless in CI, letting you shift work to cheaper off-peak runs.
Implicit premises:
  • Anthropic's harness is decoupled from — and more durable than — the specific model it was built around.
  • No single model is optimal across all sub-tasks, so per-task routing dominates single-model selection.
  • Quality need only be 'good enough' per task, so downgrading cheaper tasks preserves acceptable output.
  • The value Anthropic delivers is the harness; the model is a swappable commodity.
Inferences:
  • If sub-tasks have heterogeneous requirements and models have heterogeneous price/capability, then routing each task to its cheapest adequate model lowers cost at equal quality versus using one model for everything.

Narratives

  • The comprehensive config.json example: default = deepseek-chat, background = a local Ollama qwen2.5-coder, think = deepseek-reasoner, longContext = Gemini 2.5 Pro via OpenRouter past a 60k threshold, webSearch = gemini-2.5-flash.
    → Concretely instantiates the 'different model per sub-task on a cost gradient' thesis — from free local to premium frontier.
  • The custom-router.js example that detects 'explain this code' in the user message and routes it to a powerful model (claude-3.5-sonnet via OpenRouter), else falls back.
    → Shows routing can key on request content, not just built-in scenarios — supporting per-task model selection driven by arbitrary rules.
  • The 'UI mode is 100% vibe coding' aside: the author built the entire web UI with ccr + qwen3-coder + gemini(webSearch), creating only a folder and a project.md.
    → A self-demonstration that a routed mix of open/cheaper models is capable of real project work — supports 'open/local models are good enough for a meaningful share of work'.
  • The GitHub Actions workflow pointing ANTHROPIC_BASE_URL at a locally-started router with NON_INTERACTIVE_MODE and a deepseek backend.
    → Extends routing into CI and names the payoff — off-peak runs to reduce API costs — supporting the cost-optimization motive.

Questions addressed

  • ✅ How can I use a non-Anthropic (open, local, or cheaper) model behind Claude Code without giving up its harness?
    Run ccr as a proxy, define Providers, and use transformers to adapt each provider's API to Claude Code's protocol.
  • ✅ How do I cut Claude Code's model spend without switching tools?
    Route background/bulk tasks to a small local or cheap model, reserve frontier models for think/longContext, and run heavy jobs off-peak in CI.
  • ✅ How do I use the best-suited model for each kind of task within one session?
    Configure the Router scenarios, switch with /model, add a custom router, or pin subagents with the CCR-SUBAGENT-MODEL tag.
  • ⭕ How is a task's difficulty or output adequacy actually measured so that downgrading to a cheaper model preserves quality?
    Not addressed — routing is by static scenario category (background/think/longContext by token count), never by measured difficulty or evaluated output quality.
  • ⭕ What are the data-residency, IP-leakage, and compliance implications of routing proprietary code and prompts to third-party or local model endpoints?
    Unaddressed; the README covers an APIKEY/HOST security note for the proxy itself but is silent on where routed code and prompts end up.

Gaps

  • incompleteThe tool's implicit promise is model-agnostic 'swap freely', yet the very existence of per-provider and per-model transformers (tooluse, enhancetool, maxtoken, reasoning, deepseek, gemini) proves the harness is NOT model-agnostic — different models need bespoke request/response rewriting and tool-call error tolerance just to work behind Claude Code. The README never reconciles 'any model' with 'each model needs its own adapter', so the residual model-harness coupling — and how much reliability you lose swapping to a model the harness wasn't tuned for — is unquantified.
    Read next: Read against the harness-engineering thread on model-harness co-training and overfitting: extract-agent-harness-engineering (models get post-trained coupled to their harness) and its open gap on co-training risks; and extract-experiences-with-local-models-for-coding on how much viability is model vs. harness.
  • incompleteNo quality evaluation of routing. Routing decisions are made by static scenario (background/think/longContext by token count), never by measured task difficulty or output adequacy — but 'route the cheaper tasks to a cheaper model without hurting quality' is the load-bearing claim of any routing system, and the hard part (detecting 'good enough' per task) is treated as solved. No benchmarks show that a routed coding session preserves output quality.
    Read next: Pair with extract-pragmatic-engineer-pulse-smart-model-routing (its open gaps on unexamined 'good enough' detection and vendor-only quality claims) and with eval discipline for measuring per-task adequacy.
  • incompleteThe governance/security dimension is absent. Routing proprietary source and prompts to third-party-hosted or local model endpoints raises data-residency, IP-leakage, and compliance questions; the README addresses only proxy-level auth (APIKEY/HOST) and says nothing about where routed code lands or whether pointing Claude Code at non-Anthropic models respects the harness's terms of use.
    Read next: Read alongside extract-pragmatic-engineer-pulse-smart-model-routing's governance gap (routing sensitive enterprise code to cheaper/open/third-party models) and enterprise agent-governance material on data classification and residency.
  • incompleteNo cost or capability data. The economic case (background to local, off-peak CI, cheaper defaults) is asserted with a config gradient but no numbers — no measured savings, no tokens/sec, no quality-at-price comparison versus running everything on a frontier model.
