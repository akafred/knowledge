The keyboard was never the bottleneck

Mike Amundsen · article · 2026-04-16 source 🖥️

Changeability & OptionalityCode Was Never the BottleneckNear-Zero-Cost CodeSociotechnical ArchitectureSoftware Design Fundamentals
Generative AI commoditizes the lowest levels of programming work — producing code for known problems under fixed assumptions — but the harder, higher-level challenges of designing systems that survive change remain untouched and are now more visibly important.

theoretical · argumentative — The article advances a thesis about the nature of programming difficulty and where AI fits within it. It is structured as an argument with a taxonomy (the four levels) serving as supporting evidence. No practical how-to advice is offered.

Key terms

  • the keyboard Code Was Never the Bottleneck
    The act of producing code — typing, syntax, implementation — as opposed to the intellectual work of design and judgment
  • solution-specific computing
    Level 1: code that solves one problem under fixed assumptions and has no further role once the task is done
  • socio-technical systems Sociotechnical Architecture
    Level 4: systems operating within human institutions, shaped by policy, incentives, interpretation, and accountability — the 'system' includes people and organizations
  • anticipation Changeability & Optionality
    The design skill of building systems that support a range of possible futures, including unanticipated ones
  • scaling design Changeability & Optionality
    Making design itself — not just runtime behavior — able to absorb change, support extension, and remain coherent across teams over time

Arguments

AI does not replace programming; it shifts where difficulty and attention are required. strength: high
Explicit premises:
  • The bottleneck in programming was never code production but understanding problems, defining boundaries, and anticipating change.
  • AI is effective at producing code for known problems under fixed assumptions (level 1).
  • Higher-level programming challenges (shared systems, socio-technical systems) involve evolving constraints, multiple stakeholders, and anticipation of change.
  • AI struggles with maintaining coherence across systems as they grow and evolve.
Implicit premises:
  • The distribution of programming work across the four levels is such that higher levels represent the bulk of long-term value.
  • The skills required at higher levels (judgment, anticipation, design for change) are not readily automatable by current generative AI approaches.
Inferences:
  • Since AI addresses the historically non-bottleneck part of programming, it cannot eliminate the need for programmers.
  • Since higher-level work becomes more visible when lower-level work is automated, the role of the programmer shifts upward rather than disappearing.
APIs and shared contracts become more important, not less, in an AI-assisted development environment. strength: medium
Explicit premises:
  • When multiple actors depend on a system, expectations must be made explicit.
  • AI enables code to be produced more quickly and by more contributors with less shared context.
Implicit premises:
  • More contributors with less shared context increases the coordination burden.
  • Shared contracts (APIs) are the primary mechanism for managing coordination across contributors.
Inferences:
  • More producers plus less shared context equals greater need for explicit coordination mechanisms.

Narratives

  • Book-shopping agent example across four levels: from a single-purpose agent finding one out-of-print book, to a repeatable multi-source search system, to a multi-user e-commerce platform, to a full ecosystem handling currencies, tax, fraud, and external dependencies.
    → Makes the abstract four-level taxonomy concrete by showing how the same domain escalates in complexity at each level
  • Reference to Douglas Engelbart's 'getting better at getting better'
    → Frames the opportunity positively — AI could enable meta-improvement in how we build systems, not just faster production of code

Questions addressed

  • ✅ Is AI replacing programmers?
    No — AI is shifting where the difficulty resides, commoditizing lower-level code production while making higher-level design and judgment more visibly important.
  • ✅ Are APIs dead?
    No — the need for shared contracts becomes more pronounced, not less, when code production is cheaper and contributors have less shared context.
  • ✅ Where exactly does AI's effectiveness in programming break down?
    At the boundary between solving known problems under fixed assumptions and generalizing under changing constraints — roughly between levels 2 and 3.
  • ⭕ Will the opportunity created by cheaper code production actually be realized?
    Acknowledged as uncertain — lower costs could equally lead to more throwaway code without design gains.

Gaps

  • incompleteThe four-level taxonomy is presented as self-evident without empirical grounding, prior art references (beyond Engelbart), or criteria for distinguishing the levels in practice. It is unclear whether the levels are discrete, overlapping, or a continuum, and what determines when a system transitions from one level to another.
    Read next: Compare with established taxonomies: Lehman's laws of software evolution, the Cynefin framework, or Conway's law literature for more rigorous level-distinction criteria.
  • incompleteThe claim that AI-generated code characteristically fails at higher levels is asserted as an observed pattern but supported only by a brief characterization ('works in isolation, fails when inputs vary'). No specific examples, studies, or failure cases are cited.
    Read next: Look for empirical studies on AI-generated code in production systems, e.g., GitHub Copilot studies, or case analyses of AI-generated code failures in integration contexts.
  • incompleteThe article does not address the possibility that AI capabilities at higher levels could improve significantly. The boundary between levels 2 and 3 is treated as if it were a stable, inherent limitation rather than a current snapshot of AI capability.
    Read next: Follow research on AI agents for multi-component system design, autonomous software engineering agents, and LLM-based architecture reasoning.
  • incompleteAmundsen argues the opportunity to 'get better at getting better' exists but immediately hedges that it may not be realized. No mechanisms, incentives, or conditions for realizing it are discussed.
    Read next: Engelbart's original 'Augmenting Human Intellect' framework, and organizational learning literature (Argyris, Senge) for conditions under which meta-improvement actually occurs.
