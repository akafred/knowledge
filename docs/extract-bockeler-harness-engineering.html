Harness engineering for coding agent users

Birgitta Böckeler · article · 2026-04-20 source ↗ Reader 🖥️

Agent HarnessAgentic EngineeringCoding AgentsFeedback LoopsOrchestration Tax / Human Review Bottleneck
Coding agent users should engineer an outer harness of feedforward guides and feedback sensors — distributed across computational and inferential execution types and across the change lifecycle — to reduce review toil and increase system quality, but the behavioural harness (ensuring functional correctness) remains the hardest unsolved problem.

practical · expository — A practitioner framework article that organizes emerging harness-engineering practice into a structured mental model using control-theory concepts (feedforward/feedback). Primarily expository with prescriptive elements — taxonomizes rather than argues a contested point.

Key terms

  • harness (outer / user harness) Agent Harness
    The user-configurable layer of guides, sensors, and feedback loops that a coding agent user builds around the agent, on top of the agent builder's built-in harness
  • guides (feedforward controls) Feedback Loops
    Controls that anticipate the agent's behaviour and steer it before it acts, increasing the probability of good first-attempt results. Examples: AGENTS.md, skills, code mods, LSP
  • sensors (feedback controls) Feedback Loops
    Controls that observe after the agent acts and help it self-correct. Particularly powerful when signals are optimized for LLM consumption (e.g. custom linter messages with correction instructions)
  • computational controls Feedback Loops
    Deterministic, fast, CPU-run controls: tests, linters, type checkers, structural analysis. Milliseconds to seconds; reliable results
  • inferential controls Feedback Loops
    Semantic, probabilistic, GPU/NPU-run controls: AI code review, LLM-as-judge. Slower, more expensive, non-deterministic, but capable of richer semantic judgment
  • steering loop Feedback Loops
    The human practice of iterating on the harness by converting recurring agent failures into improved feedforward and feedback controls
  • harnessability Agent Harness
    The degree to which a codebase's properties (type system, module boundaries, framework abstractions) make it amenable to effective harness controls
  • harness templates Agent Harness
    Pre-defined bundles of guides and sensors matched to common service topologies (API services, event processors, dashboards), justified by Ashby's Law of Requisite Variety as a variety-reduction move
  • regulation categories
    Three domains the harness regulates: maintainability (internal code quality), architecture fitness (non-functional characteristics), and behaviour (functional correctness)

Arguments

Both feedforward and feedback controls are necessary for effective agent harnessing strength: high
Explicit premises:
  • Guides (feedforward) anticipate behaviour and steer before the agent acts
  • Sensors (feedback) observe after the agent acts and enable self-correction
  • Feedback-only leads to repeated mistakes; feedforward-only means rules are never validated
Implicit premises:
  • Agent behaviour is sufficiently unpredictable that anticipation alone cannot prevent all errors
  • Self-correction is only valuable if the initial guidance constrains the space of possible errors
Inferences:
  • By elimination — each direction alone is shown to be insufficient, therefore both are needed.
The behavioural harness is the most important unsolved problem in harness engineering strength: medium
Explicit premises:
  • Maintainability harness has abundant pre-existing tooling
  • Architecture fitness harness can leverage fitness functions
  • Behavioural harness currently relies on AI-generated tests, which are not trustworthy enough
  • Correctness is outside any sensor's remit if the human didn't clearly specify requirements
Implicit premises:
  • Functional correctness is ultimately more important than maintainability or architecture fitness
  • The quality of AI-generated tests cannot be assumed to match human-written specification tests
Inferences:
  • Argument from difficulty gradient: easiest problems have solutions, hardest problem (behaviour) does not yet.
Pre-defined service topologies make comprehensive harnessing more achievable strength: high
Explicit premises:
  • Ashby's Law states a regulator must have at least as much variety as the system it governs
  • An LLM-based coding agent can produce almost anything
  • Committing to a topology narrows the output space
Implicit premises:
  • The current variety of possible agent outputs exceeds what any practical harness can regulate
  • Topology constraints are acceptable trade-offs for the governance gains
Inferences:
  • Deductive application of Ashby's Law: reduce system variety to make regulation feasible.
Harnesses can only partially externalize human developer experience strength: high
Explicit premises:
  • Humans bring implicit knowledge: conventions, aesthetic judgment, organizational context, social accountability
  • Agents lack all of these qualities
  • Externalizing and making explicit what experience provides is expensive and must be prioritized
Implicit premises:
  • Some human knowledge is tacit and resists formalization (Polanyi's paradox)
  • Social accountability and aesthetic judgment cannot be replicated by rules or sensors
Inferences:
  • From the gap between tacit and explicit knowledge: if not all can be formalized, externalization must be partial.
Controls should be shifted as far left in the lifecycle as possible strength: medium
Explicit premises:
  • Earlier detection means cheaper fixes (CI/CD principle)
  • Fast/cheap controls (linters, fast tests) can run pre-commit
  • Expensive controls (mutation testing, broad review) should run post-integration
Implicit premises:
  • The cost of fixing agent errors scales with how far they propagate before detection
  • This CI/CD principle transfers directly from human-authored to agent-authored code
Inferences:
  • Argument by analogy: shift-left economics apply equally to agent-produced code.

Narratives

  • OpenAI team documented their harness: layered architecture enforced by custom linters, structural tests, and recurring 'garbage collection' drift scans with agent-suggested fixes. Their conclusion: 'Our most difficult challenges now center on designing environments, feedback loops, and control systems.'
    → Primary external validation that harness engineering is the central challenge, not model capability. Supports the lifecycle distribution argument (P6) and the ongoing-practice thesis (P15).
  • Stripe's minions system uses pre-push hooks with heuristic-based linter selection, emphasizes shift-left, and uses 'blueprints' to integrate feedback sensors directly into agent workflows.
    → Illustrates computational feedback sensors integrated into the agent loop and the shift-left principle in production at scale. Supports P6 and P4.
  • Thoughtworks teams tackle architecture drift with both computational and inferential sensors — improving API quality with agents + custom linters, improving code quality with a 'janitor army'.
    → Shows the computational/inferential distinction playing out in real team practice. Supports P5 and P15.
  • Colleagues using the 'approved fixtures' pattern selectively for behavioural verification, but acknowledging it is not a wholesale answer to the test quality problem.
    → Illustrates both a partial solution and the incompleteness of the behavioural harness. Supports P9 and P7.

Questions addressed

  • ✅ How should coding agent users think about and structure the controls they build around their agents?
    Use a 2x2 framework: feedforward guides vs feedback sensors, crossed with computational vs inferential execution types. Both directions are necessary.
  • ✅ What mental model from control theory applies to agent harnessing?
    The cybernetic governor model: feedforward anticipation combined with feedback self-correction, regulated by the human steering loop.
  • ✅ Where in the software change lifecycle should different types of controls be placed?
    Shift left: cheap/fast controls pre-commit, expensive controls post-integration, continuous drift detection outside the change lifecycle.
  • ✅ What can and cannot be effectively harnessed today?
    Maintainability is easiest (pre-existing tooling). Architecture fitness is feasible (fitness functions). Behavioural correctness is the hardest and least solved.
  • ✅ What is the role of the human developer when agents are harnessed?
    The human steers the harness by iterating on guides and sensors. The goal is not to eliminate human input but to direct it where it matters most.
  • ✅ How do codebase properties affect harness effectiveness?
    Harnessability varies: strong typing, clear module boundaries, and framework abstractions increase it. Legacy codebases face a paradox — harness is most needed where hardest to build.

Gaps

  • incompleteThe behavioural harness is acknowledged as 'the elephant in the room' but no concrete framework or methodology is proposed for solving it beyond mentioning approved fixtures as a partial pattern.
    Read next: Read the approved fixtures pattern (lexler.github.io/augmented-coding-patterns) and explore specification-based testing approaches (property-based testing, contract testing) as potential behavioural harness components.
  • incompleteThe article does not address how to evaluate harness quality or coverage — it raises the question ('If sensors never fire, is that high quality or inadequate detection?') but offers no answer.
    Read next: Explore meta-testing approaches: mutation testing for harnesses, harness coverage metrics, or adversarial testing of the harness itself.
  • incompleteThe cybernetic framing (feedforward/feedback, Ashby's Law) is adopted as the organizing metaphor but not critically examined. Control theory assumes a well-defined desired state and measurable error signal — software development often has neither.
    Read next: Read Stafford Beer's work on viable systems and cybernetic management to assess how well control-theory concepts transfer to inherently creative/ambiguous engineering tasks.
  • incompleteNo discussion of the cost-benefit analysis of harness engineering itself — when is the investment in building and maintaining a harness worthwhile vs. simply relying on human review?
    Read next: Look for empirical studies or experience reports quantifying harness ROI in agent-assisted development.
  • incompleteThe interaction between guides and sensors — coherence, contradiction, priority resolution — is raised as an open question but not explored.
    Read next: Explore policy conflict resolution in cybernetic and multi-agent systems literature.
