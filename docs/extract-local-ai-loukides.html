Local AI

Mike Loukides and Claude · article · 2026-05-09 source ↗ Reader 🖥️

Data SovereigntyEvalsInference OptimizationLocal AIOpen-Weight Models
Open-weight AI models running on local hardware have become viable alternatives to cloud-based frontier models for a growing majority of use cases, driven by converging forces of cost, privacy, regulatory compliance, fine-tuning flexibility, and the needs of a global developer community that US cloud providers have largely ignored.

practical · survey-argument — A practitioner-oriented survey that argues for the viability of local AI deployment. Combines market analysis, cost modeling, security assessment, and a model landscape overview into a sustained case for local over cloud AI, with particular attention to non-US developers.

Key terms

  • open weight Open-Weight Models
    Models that release trained numerical parameters but not training data or training code — usable and fine-tunable but not reproducible or auditable
  • openwashing Open-Weight Models
    The practice of claiming openness while releasing only weights, without training data or code that would allow reproduction or audit
  • mixture-of-experts (MoE) Inference Optimization
    An architecture where only a fraction of total parameters are activated per token, enabling large models to run on smaller hardware
  • time to first token Inference Optimization
    Latency before the model begins producing output — the key metric for interactive applications, where local models have a structural advantage (no network hop, no queuing)
  • golden dataset Evals
    A curated set of a few hundred real prompts with known-good answers, used to evaluate candidate models against your actual use case rather than published benchmarks
  • data sovereignty Data Sovereignty
    Legal requirements that data remain within a jurisdiction's borders, making cloud APIs legally awkward or impossible

Questions addressed

  • ✅ Why would you run AI models locally instead of using cloud APIs?
    Cost, privacy, regulatory compliance, fine-tuning flexibility, and performance for agentic workflows
  • ✅ What does local AI cost compared to cloud API usage?
    Hardware $500–$1,500 upfront + $20–$40/month electricity vs $500–$1,000/month API costs; breakeven in months
  • ✅ How do local models perform relative to frontier cloud models?
    Local wins on time-to-first-token and agentic latency; cloud wins on throughput and concurrency
  • ✅ Why is local AI adoption strongest outside the United States?
    European data sovereignty, Chinese hardware-constraint-driven efficiency, Global South cost and language barriers
  • ✅ Which open-weight models are currently most capable?
    Gemma 4 as top recommendation; GLM-5.1, DeepSeek V4/R1, Qwen 3.6, Kimi K2.6 also surveyed
  • ✅ What is the difference between open-weight and open-source AI?
    Open-weight releases only parameters; OSI open-source requires training data, code, and evaluation code
  • ✅ What security trade-offs does local deployment involve?
    Eliminates cloud exfiltration risk but introduces supply-chain, alignment, and prompt-injection responsibilities
  • ✅ Can you fine-tune models on consumer hardware?
    Yes — QLoRA enables fine-tuning 7B–8B models on 12GB VRAM; Unsloth reduces requirements further
  • ✅ How should you evaluate whether a local model is good enough for your use case?
    Build a golden dataset of a few hundred real prompts with known-good answers rather than trusting published benchmarks
