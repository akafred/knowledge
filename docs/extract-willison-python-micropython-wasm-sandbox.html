Running Python code in a sandbox with MicroPython and WASM

Simon Willison · email · 2026-06-06 Reader 🖥️

Agent SecurityCoding AgentsLethal TrifectaSandboxingVibe Coding
WebAssembly-embedded MicroPython is the first Python sandbox that meets all the criteria Willison thinks a trustworthy in-process sandbox must satisfy — clean PyPI install, memory/CPU limits, controlled file and network access, host-function interop, and active upstream maintenance — making it the right substrate for safely running plugin and agent-generated Python code.

Key terms

  • sandbox Sandboxing
    An in-process execution environment for untrusted code that enforces memory/CPU limits, mediates all file and network access, and lets the host selectively expose functions — embeddable as a Python package
  • vibe-coded Vibe Coding
    Built primarily via prompting coding agents (here Codex Desktop + GPT-5.5) with the human reviewing rather than authoring; used self-deprecatingly when applied to security-critical code
  • fuel
    wasmtime's unit for bounding WebAssembly compute; Willison uses it as his CPU limit but is unsure 20M is the right default
  • Lethal Trifecta Lethal Trifecta
    His prior framing (invoked in the Lockdown Mode item): private data + untrusted content + exfiltration channel; cut any one leg to neutralize prompt-injection risk
  • host functions
    Functions defined in the outer Python application that sandboxed MicroPython code may call — the controlled interface through which all privileged operations must pass

Arguments

MicroPython+WASM is the right substrate for a Python sandbox embeddable in Python applications. strength: high
Explicit premises:
  • A trustworthy sandbox must satisfy six criteria (clean install, memory limits, CPU limits, file mediation, network mediation, host functions, active maintenance).
  • JavaScript-engine embeds for Python are poorly maintained and warn against untrusted input.
  • Pyodide is browser/Node only and not supported server-side.
  • MicroPython is designed for constrained environments and compiles to WebAssembly via WASI.
  • wasmtime provides memory limits natively and a fuel mechanism for CPU bounding.
Implicit premises:
  • The six criteria are jointly sufficient, not merely necessary, for a sandbox to be trustworthy.
  • Bounding blast radius is the safety property that matters most for plugin/agent-executed code.
  • MicroPython's smaller standard library is acceptable for the workloads Willison cares about (Datasette enrichments, plugin glue, agent-generated data shaping).
Inferences:
  • If MicroPython+WASM meets the criteria, every comparable substrate (Lua-in-WASM, QuickJS, gVisor for in-process Python) should be evaluated against the same six-point rubric.
  • Sandbox maturity is bottlenecked on documentation and institutional adoption more than on technical novelty.
It's defensible to ship an alpha sandbox built largely with coding agents because the WASM boundary contains failures. strength: medium
Explicit premises:
  • If the C code has bugs, the worst outcome is a WASM exception, not host compromise.
  • Two LLMs have explained the code to Willison and it has been tested.
  • GPT-5.5 xhigh has been challenged to break out and has not.
Implicit premises:
  • wasmtime itself is trustworthy enough that its boundary will hold even when the in-sandbox C is wrong.
  • Adversarial testing by a frontier model is a meaningful proxy for adversarial testing by humans + automated tools.
  • Risk from a buggy sandbox is borne by the operator who chose to install it, not by third parties.
Inferences:
  • The argument transfers cleanly to anyone building sandbox tooling on top of mature isolation primitives: the substrate carries most of the safety budget.
  • If the substrate ever fails, all of the downstream 'vibe-coded' sandboxes inherit a coordinated vulnerability.
Cutting exfiltration is the right industry-wide defense posture for LLM systems exposed to untrusted content. strength: high
Explicit premises:
  • The Lethal Trifecta requires all three legs; removing any one neutralizes it.
  • Private data access and untrusted-content exposure are hard to remove without making products useless.
  • Deterministic egress controls (Lockdown Mode, sandbox network mediation) are not subvertable by prompt injection.
Implicit premises:
  • Most damaging outcomes require exfiltration; in-place damage (corruption, denial of service inside the sandbox) is tolerable.
  • Egress channels are enumerable and closeable — the API surface is small enough to lock down.
Inferences:
  • Default-on egress restriction should become baseline product behavior; the existence of Lockdown Mode as opt-in is itself an admission that defaults are inadequate.
  • Sandboxes that meet Willison's network criterion are doing the same work as Lockdown Mode at a finer grain.

Questions addressed

  • ✅ What does a sandbox need to support to be trustworthy enough to embed in a real Python application?
    Six explicit criteria: PyPI-installable, memory limits, CPU limits, file mediation, network mediation, host functions, active maintenance.
  • ✅ Can a single developer, working with coding agents, produce a viable Python-in-WASM sandbox?
    Yes — micropython-wasm 0.1a0 exists, with persistent sessions and host-function interop in 78 lines of C.
  • ✅ Should you trust this specific implementation in production?
    No — alpha, recommended only for users willing to take significant risk; hopes professional security teams will pick up the approach.
  • ⭕ Does sandboxing solve the problem of agents executing destructive code, or only bound its reach?
    Not addressed — Willison's framing equates 'cannot escape' with 'safe', collapsing sanctioned-but-wrong actions into background noise.
  • ⭕ What's the right CPU bound (fuel value) for general-purpose sandboxed Python?
    Explicitly open — 20M is a guess; units 'are hard to reason about'.
  • ⭕ How should an organization manage proliferation of cheap-to-build, easy-to-abandon AI-generated projects?
    Raised in the roundup (David Wilson, Charity Majors) but not connected to the sandbox piece, even though the sandbox makes proliferation easier.
