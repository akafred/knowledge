Claude Code: Best practices for agentic coding

Because Claude Code is deliberately low-level and unopinionated, its effectiveness on your problem is set by the harness you engineer around the model — the context, tools, and workflow you supply — not by the raw model alone.

practical · expository prescriptive guide — A best-practices guide: it argues 'you should do X' for agentic coding with Claude Code, grounding each recommendation in claims about how the tool and model behave. Organized as six numbered sections (customize setup, give more tools, common workflows, optimize workflow, headless mode, multi-Claude) of experience-derived patterns rather than a single sustained argument.

Key terms

  • CLAUDE.md Agent Context Files (CLAUDE.md / AGENTS.md)
    A special markdown file Claude Code automatically pulls into context at the start of a conversation, holding persistent project context (bash commands, core files, code style, testing instructions, repo etiquette, environment setup, warnings); placeable at repo root, parent dirs, child dirs, or home folder, and meant to be treated and iterated on like a prompt
  • agentic coding
    Software development in which an assistant like Claude Code autonomously reads, searches, plans, edits, runs, and iterates on code across multiple steps with tool access, rather than emitting a single completion
  • extended thinking Extended Thinking / Reasoning
    A mode giving Claude additional computation time to evaluate alternatives, triggered by escalating keywords 'think' < 'think hard' < 'think harder' < 'ultrathink' that map directly to progressively larger thinking-budget allocations
  • explore, plan, code, commit
    A versatile workflow in which the user first has Claude read/research relevant material (explicitly not coding yet), then make a plan (optionally with extended thinking), then implement, then commit — because skipping the first two steps makes Claude jump straight to code
  • test-driven development Test-Driven Development
    An Anthropic-favorite agent workflow: have Claude write tests from expected input/output pairs first (avoiding mock implementations), confirm they fail, commit them, then write code that iterates until all tests pass without modifying the tests — the tests are the verifiable target Claude iterates against
  • allowed tools / allowlist
    The curated set of tools Claude Code may run without asking; by default it requests permission for anything that could modify the system (file writes, many bash commands, MCP tools), and the user loosens this deliberately conservative deny-by-default posture only for actions that are safe or easily undone
  • subagents Multi-Agent Orchestration
    Secondary Claude instances the main agent spawns to verify details or investigate specific questions — used especially early in a task to preserve the main context window without much efficiency loss, and to check that an implementation isn't overfitting to its tests
  • headless mode
    Non-interactive invocation of Claude Code via the -p flag for CI, pre-commit hooks, build scripts, and automation — with --output-format stream-json for structured output; does not persist between sessions and must be triggered each time
  • course correct
    Actively steering Claude mid-task rather than letting it run autonomously — asking for a plan before code, pressing Escape to interrupt while preserving context, double-tapping Escape to rewind history and edit a prior prompt, or asking Claude to undo changes
  • safe YOLO mode
    Running claude --dangerously-skip-permissions to bypass all permission checks and let Claude work uninterrupted — recommended only inside a container without internet access to bound the risk of data loss or exfiltration via prompt injection
  • git worktrees
    Multiple branches of one repo checked out into separate directories sharing history, letting several Claude sessions run simultaneously on independent, non-overlapping tasks without merge conflicts — a lighter-weight alternative to multiple full checkouts
  • MCP MCP (Model Context Protocol)
    The Model Context Protocol, which Claude Code speaks as both server and client; as a client it connects to any number of MCP servers (via project config, global config, or a checked-in .mcp.json) to gain their tools

Arguments

You should invest effort in engineering the context and tools around Claude Code (CLAUDE.md, allow-list, MCP, custom scripts and slash commands) rather than just prompting the raw model. strength: high
Explicit premises:
  • Claude Code automatically pulls context into prompts, which consumes time and tokens but can be optimized through environment tuning.
  • CLAUDE.md is auto-loaded and its content becomes part of Claude's prompts.
  • Claude inherits your bash environment and can use any tools and scripts you expose to it.
Implicit premises:
  • The model's effective capability on your codebase is bounded by the context and tool surface it is given, not by its weights alone.
  • Configuration effort amortizes because CLAUDE.md and tool setup are reused every session and shareable across the team.
  • A generic setup underperforms one shaped to a specific project's conventions and history.
Inferences:
  • If output quality tracks the assembled context and tools, then improving that assembly is the highest-leverage action available to the user.
Give Claude a clear, checkable target (tests, a visual mock, or expected output) and let it iterate against that target. strength: high
Explicit premises:
  • Claude's outputs improve significantly with iteration — typically much better after 2-3 passes.
  • Claude performs best when it has a clear target to iterate against.
  • With expected outputs like tests, Claude can make changes, evaluate results, and incrementally improve until it succeeds.
Implicit premises:
  • Without an external, machine-checkable signal the agent has no reliable gradient to climb and cannot distinguish real improvement from drift.
  • The human's role shifts from writing the solution to specifying the acceptance signal.
  • A verification signal the agent can run itself (tests, screenshots) is what lets the correction loop close autonomously.
Inferences:
  • Therefore the quality ceiling of agentic coding is set by the quality and checkability of the feedback signal, not by prompt wording.
Intervene early — force a plan before code and correct course continuously — rather than letting Claude run to completion. strength: high
Explicit premises:
  • Without an explicit research-and-plan step, Claude tends to jump straight to coding a solution.
  • Asking Claude to research and plan first significantly improves performance on problems requiring deeper thinking.
  • Escape and double-Escape preserve context and let you redirect or rewind at any phase.
Implicit premises:
  • Errors introduced early compound as the agent builds on them, so early correction is disproportionately cheaper than late correction.
  • The human retains judgment the model lacks about whether the chosen approach is the right one.
Inferences:
  • Therefore steering at the plan stage yields better and faster results than reviewing a finished but misdirected implementation.

Narratives

  • The explore-plan-code-commit workflow: ask Claude to read relevant files/images/URLs without coding (using subagents for hard sub-questions), have it make a plan with extended thinking and optionally save it as a doc/GitHub issue to reset to, then implement, then commit and open a PR.
    → The guide's flagship worked workflow, demonstrating that inserting explicit research and planning phases before coding is what improves results on hard problems.
  • The agentic TDD workflow: Claude writes tests from input/output pairs (no mocks), runs them to confirm they fail, commits the tests, then writes code that loops — run, adjust, re-run — until all tests pass without touching the tests, with a subagent checking it isn't overfitting.
    → Concretely instruments the 'clear target to iterate against' principle, showing tests as the verifiable signal the agent's self-correction loop converges on.
  • The screenshot-iterate workflow: give Claude a way to take browser screenshots (Puppeteer/iOS-simulator MCP) plus a visual mock, ask it to implement the design, screenshot the result, and iterate 2-3 times until it matches the mock.
    → Shows the same iterate-against-a-target principle in a non-test modality, generalizing the feedback signal from tests to visual output.
  • The lint-fix checklist workflow: tell Claude to run the lint command and write every error (file + line) to a Markdown checklist, then work through them one by one, verifying and checking off each before moving on.
    → Demonstrates externalizing state into a scratchpad to preserve performance across a long, exhaustive multi-step task — an antidote to context degradation.
  • The headless fan-out: have Claude write a script to list, say, 2,000 files to migrate from React to Vue, then loop calling claude -p per file with a scoped toolset, each returning the string OK or FAIL, refining the prompt across runs.
    → Illustrates using headless mode to turn the agent into a programmatic batch worker for large migrations, scaling beyond interactive use.
  • The multi-Claude write/verify loop: one Claude writes code, then /clear or a second terminal Claude reviews it, then a third reads both code and review and edits accordingly — optionally with the instances passing notes via separate scratchpad files.
    → Concretely realizes the generation/verification separation, showing separate context windows acting like separate engineers.

Questions addressed

  • ✅ How should I configure Claude Code so it performs well on my codebase?
    Create and iteratively tune CLAUDE.md files, curate the tool allow-list, install gh, and expose bash tools, MCP servers, and custom slash commands.
  • ✅ When and how much should the model think before acting?
    Trigger extended thinking with escalating keywords (think < think hard < think harder < ultrathink) mapped to larger thinking budgets, chosen by problem difficulty.
  • ✅ How do I keep a long session from degrading in quality?
    Use /clear between tasks, offload investigation to subagents, and externalize multi-step state into Markdown checklists/scratchpads.
  • ✅ How do I keep the agent from jumping to a wrong or premature solution?
    Use explore-plan-code-commit, be specific, and course-correct early with plan-first, Escape/double-Escape, and undo.
  • ✅ How do I scale beyond a single interactive session?
    Use headless mode (fan-out and pipelining) and multi-Claude workflows across separate terminals, checkouts, or git worktrees.
  • ⭕ Which model, or which effort/thinking level, should I use for which step — and can this be routed automatically?
    The guide treats 'Claude' as a single model; it never addresses model selection, per-step model allocation, or automatic routing.
  • ⭕ How much do these practices cost (tokens, rate limits) and when is the harness overhead worth it?
    Cost, rate limits, and cost/benefit thresholds for multi-Claude and headless fan-out are not discussed.
