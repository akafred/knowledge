GitHub Enhances CodeQL with Declarative Security Modeling for Faster, More Flexible Analysis

Craig Risi · rss · 2026-04-20 source 🖥️

GitHub's new declarative models-as-data approach to CodeQL security analysis lowers the expertise barrier for custom security rules by replacing query-language programming with YAML configuration, reflecting a broader industry shift toward data-driven, developer-accessible security tooling.

Key terms

  • models-as-data
    A CodeQL feature where security behavior of functions (sources, sinks, sanitizers, validators) is declared in structured YAML data rather than written as CodeQL query code
  • sanitizers
    Functions that clean or neutralize untrusted data, modeled as 'barriers' that stop taint propagation in CodeQL analysis
  • validators
    Functions that check data safety conditions, modeled as 'barrier guards' that halt taint propagation when a boolean condition is met
  • taint tracking
    Static analysis technique tracing how untrusted data flows through an application to detect security vulnerabilities
  • declarative security modeling
    Defining security analysis rules via structured data (YAML) rather than imperative query code, reducing the need for specialized expertise

Arguments

The models-as-data approach to security analysis is better than custom query development strength: medium
Explicit premises:
  • Custom CodeQL queries require deep expertise in query languages
  • YAML-based data extensions can express the same sanitizer/validator logic declaratively
  • Declarative configuration is easier to version, share, and scale
Implicit premises:
  • The expressiveness lost by moving from a full query language to YAML declarations is negligible for the sanitizer/validator use case
  • Teams that currently lack CodeQL expertise would adopt declarative modeling if available
  • The security rules most teams need to add fall within the sanitizer/validator pattern
Inferences:
  • If declarative modeling covers only sanitizers and validators, teams still need query expertise for other customization needs — the barrier is lowered but not eliminated
The security tooling industry is converging on developer-accessible, friction-reducing approaches strength: medium
Explicit premises:
  • GitLab emphasizes prebuilt rules and pipeline integration
  • Snyk focuses on developer-first inline remediation
  • Semgrep uses code-like patterns avoiding full query languages
  • GitHub's CodeQL now supports declarative modeling
Implicit premises:
  • These four platforms are representative of the industry
  • Developer accessibility and precision of analysis are on a single trade-off spectrum
  • The direction of this convergence is desirable and inevitable
Inferences:
  • If all vendors converge on accessibility, deep-customization use cases may become underserved as tooling optimizes for the common case
