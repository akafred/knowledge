Coding Models Are Doing Too Much

nrehiew.github.io · article · 2026-04-20 source ↗ Reader 🖥️

Code QualityCoding AgentsEvalsLegacy & Organizational ForgettingScope Discipline
Current coding models systematically rewrite code beyond the minimal fix required — a widespread, measurable failure mode called over-editing — and reinforcement learning can train models to produce faithful minimal edits without degrading their general coding ability.

mixed · argumentative — An empirical research report structured as an argument. The author identifies a practical problem (over-editing by coding models), constructs a benchmark to measure it, evaluates frontier models, and proposes a training method (RL) to fix it. Argumentative in that the entire piece advances a thesis about a specific failure mode and its remedy, but grounded in experimental evidence.

Structure

PartRole
Problem definition: the Over-Editing problemNames and illustrates the phenomenon — models modify code far beyond what is necessary to fix a bug, making diffs unrecognizable and code review harder
Green-field vs brown-field framingExplains why over-editing matters: in brown-field work the existing code is deliberate, and the model's job is to fix only the issue
Benchmark construction and metricsCreates a dataset of 400 programmatically corrupted problems with known minimal fixes, and introduces token-level Levenshtein Distance and Added Cognitive Complexity as metrics
Evaluation of frontier modelsShows that all frontier models over-edit, with GPT-5.4 worst and Claude Opus 4.6 best; demonstrates that high Pass@1 does not imply minimal editing
Prompting experimentsShows that an explicit instruction to preserve original code reduces over-editing across all models, especially reasoning models
Reasoning vs non-reasoning comparisonDemonstrates that reasoning models over-edit more by default but respond better to explicit constraints

LoRA and scaling experimentsShows RL with LoRA nearly matches full fine-tuning, and the recipe scales from 4B to 14B parameters

Key terms

  • Over-Editing Scope Discipline
    A model modifying code beyond what is strictly necessary to fix the problem at hand — the output is functionally correct but structurally diverges from the original more than the minimal fix requires
  • Token-level Levenshtein Distance
    Edit distance computed over Python token sequences (not characters), normalized by total token count, measuring how much a model's patch diverges from the minimal fix
  • Added Cognitive Complexity
    The absolute difference in Cognitive Complexity between the model's output and the original uncorrupted code; any nonzero value means the model introduced unnecessary structural complexity
  • brown-field Legacy & Organizational Forgetting
    Working within an existing codebase where code has been deliberately written and understood by the team, as opposed to green-field (building from scratch)
  • relative patch score
    The ratio comparing the model's edit distance (from corrupted input) to the ground-truth minimal edit distance, where values closer to zero indicate the model's patch resembles the true minimal fix

Questions addressed

  • ⭕ Do current coding models over-edit, and can this be measured?
  • ⭕ Can prompting alone fix over-editing?
  • ⭕ Does reasoning make over-editing worse?
  • ⭕ Can models be trained to produce minimal edits?
  • ⭕ Is full fine-tuning necessary for this behavioral change?

Gaps

  • incompleteThe benchmark uses only single-function, single-bug corruptions. Real-world brown-field edits often span multiple files, involve multi-step reasoning, and require understanding broader architectural context. The author acknowledges this but does not address how the metrics or training approach would scale to more realistic settings like SWE-Bench Pro.
  • incompleteThe corruptions are all value-level (operator flips, boolean swaps) rather than structural (control flow, API misuse, concurrency bugs). It is unclear whether the over-editing phenomenon and the RL remedy hold for structural bugs where the minimal fix may itself be structurally complex.
  • incompleteThe paper does not investigate whether over-editing correlates with RLHF training that optimizes for helpfulness. Models may be trained to be maximally helpful, which in a coding context manifests as 'improving' code proactively. This would be a root-cause analysis the paper does not attempt.
  • incompleteNo analysis of how the system prompt, context length, or conversation history affects over-editing. In practice, coding agents operate with long contexts and multi-turn interactions — does over-editing get worse with more context?
  • incompleteThe RL reward function combines correctness and Levenshtein distance with a fixed weighting (pass bonus +0.1, fail penalty -0.2). No sensitivity analysis on these hyperparameters is provided, and the Pareto frontier between correctness and minimality is not explored.

Closes gaps elsewhere

  • LLMs Corrupt Your Documents When You Delegate · g2 — Supplies the missing mitigation strategies for unrequested-modification corruption: explicit preserve-the-original prompt instructions and RL training that generalizes minimal-editing behavior.
