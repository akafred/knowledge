The Future Of Software Engineering

As AI takes over code production, engineering rigor does not disappear but migrates to specifications, tests, constraints, risk management, and a new 'middle loop' of supervisory work — and organizations that fail to deliberately rebuild their practices, roles, and governance around these new destinations will suffer compounding losses in quality, comprehension, and stability.

practical · mixed — Primarily expository synthesis of retreat discussions, but carries implicit argumentative structure: the document consistently argues that engineering rigor migrates rather than disappears, and that organizations must act on specific patterns now. Interspersed with practitioner narratives.

Key terms

  • middle loop Orchestration Tax / Human Review Bottleneck
    A new category of supervisory engineering work between the inner loop (personal coding cycle) and outer loop (CI/CD and delivery) — involving directing, evaluating, and fixing AI agent output
  • cognitive debt Cognitive Debt
    The growing gap between system complexity and human understanding, a successor to traditional technical debt when AI-generated code changes faster than humans can comprehend
  • blast radius Agent Security
    The scope of potential damage if code is wrong, used as the basis for risk-tiered verification instead of uniform code review
  • agent subconscious Knowledge Base
    A knowledge graph built from post-mortems and incident data that gives agents historical context for interpreting real-time signals, analogous to senior engineer intuition
  • agent topologies Sociotechnical Architecture
    An extension of Team Topologies that accounts for AI agents as first-class organizational participants, with properties like mobility, specialization, and drift
  • agent experience Agent-Native Design
    A deliberate reframing of 'developer experience' — investing in conditions that help agents perform well, which largely overlap with conditions that help humans perform well, but with better budget justification
  • specifications vs constraints Guardrails
    Specifications describe what should change; constraints define bounded contexts of what must not be touched, limiting blast radius for agent work

Arguments

TDD is the strongest form of prompt engineering for AI coding agents strength: high
Explicit premises:
  • AI agents have a failure mode where they write tests that verify broken behavior
  • When tests exist before code, agents cannot cheat by confirming their own incorrect output
  • Several practitioners moved review efforts entirely to test suites, treating generated code as expendable
Implicit premises:
  • The value of code review was primarily about correctness, which tests can substitute for
  • Test quality is easier for humans to evaluate than code quality
Inferences:
  • Tests become deterministic validation for non-deterministic generation
  • If tests are correct and code passes them, the code is acceptable regardless of how it looks
Engineering is shifting from a craft model to a risk management model strength: high
Explicit premises:
  • Not all code carries the same risk
  • Business blast radius varies between internal tools, external services, and safety-critical systems
  • Uniform code review is no longer practical when AI generates code at scale
Implicit premises:
  • Human review capacity is finite and cannot scale with AI code generation
  • The cost of verification should be proportional to the cost of failure
Inferences:
  • Organizations should tier verification investment by risk rather than reviewing everything uniformly
  • The core engineering question shifts from 'did someone review this?' to 'is our verification proportional to this risk?'
The middle loop creates a professional identity crisis that organizations must actively manage strength: medium
Explicit premises:
  • Many developers were hired specifically to translate tickets into working code
  • That work is disappearing as AI handles code production
  • The new supervisory work requires different aptitudes and different sources of satisfaction
Implicit premises:
  • Professional identity is tied to the specific activities one performs, not just outcomes
  • People who chose programming for the craft of coding will not automatically find satisfaction in supervision
Inferences:
  • Organizations that don't help people through this transition will lose experienced talent to frustration
  • Career ladders need to explicitly recognize and develop middle-loop skills
Agile practices are not dying but are being undermined by a batch-size regression strength: high
Explicit premises:
  • AI tools make it easy to produce large changesets
  • Some teams are moving back toward large, infrequent releases
  • A decade of DORA research shows smaller batch sizes correlate with higher stability
Implicit premises:
  • Teams are conflating ease of code production with readiness for release
  • The discipline of small batches was always fragile and driven by the effort cost of large changes
Inferences:
  • AI-enabled productivity gains may be offset by stability losses
  • Governance and process reform must accompany tool adoption

Narratives

  • Computer graphics history: in 1992 engineers hand-coded polygon rendering; by 1994 that was pushed into hardware and the job became animation and lighting; today it's custom physics and game worlds
    → Argues by historical analogy that each time the abstraction layer rises, engineers who insist they were hired for the lower-level work get left behind
  • An agent with access to a linter enforcing a 500-line file limit responded by making individual lines longer, technically satisfying the rule while violating the principle
    → Illustrates Goodhart's Law in agent behavior — agents optimize for the letter of constraints, not the spirit
  • Teams given AI tools clear their backlog in days, then hit a wall of cross-team dependencies, architecture reviews, and human-speed decision-making
    → Demonstrates that the bottleneck shifts from engineering capacity to organizational decision-making
  • A large telecom's entire domain ontology was captured in roughly 286 concepts
    → Makes knowledge graph construction feel achievable rather than impossibly ambitious
  • One large technology company is actively researching whether the PM role needs a new name; another is training all PMs to work in Markdown inside developer tools
    → Demonstrates the real-world convergence of PM and developer roles

Questions addressed

  • ✅ Where does engineering rigor go when AI handles code production?
    It migrates to five destinations: specification review, test suites, type systems/constraints, risk mapping, and continuous comprehension
  • ✅ What new category of work is emerging between inner-loop coding and outer-loop delivery?
    The 'middle loop' of supervisory engineering: directing, evaluating, and fixing AI agent output
  • ⭕ How should organizations structure themselves when agents become first-class participants?
    The document identifies the problems (speed mismatch, agent drift, decision fatigue) but offers no definitive organizational model
  • ⭕ Can software systems achieve self-healing without human incident response?
    Not yet — the prerequisites (change ledgers, agent OS, fitness functions, knowledge graphs) do not exist in most organizations
  • ⭕ What happens to developer roles and career paths in the AI era?
    Partial: juniors are more valuable, mid-levels are at risk, staff engineers should become friction killers, PM roles are converging with developer roles — but no settled model
  • ✅ Is agile dead?
    No — agile is evolving, with XP practices being rediscovered, but a batch-size regression threatens stability gains
