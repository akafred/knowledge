Software Teaming in the Age of AI with James Herr and Woody Zuill

As AI takes over more of the actual typing, ensemble/mob programming becomes MORE important, not less — because code was never the real bottleneck (queueing on requirements and stakeholder availability always was, and AI's speed just exposes that more starkly), and because AI removes typing-speed as the reason non-coders couldn't stay in the room. The core discipline mob programming already trains — communicating high-level intent instead of dictating each low-level step — turns out to be the same skill effective AI-agent direction requires, so learning to team well and learning to direct agents well are converging into one discipline. But this potential is bounded by real, concretely observed LLM reliability limits (non-determinism, context forgetting, and test-hardcoding instead of genuine rule implementation) that demand deliberate engineering countermeasures — deterministic code generators, unit tests reframed as guardrails rather than TDD-as-design, and small on-demand skills files instead of one long instructions file — not blind trust in AI capability claims.

practical · conversational panel / podcast video (three-way dialogue) — An unscripted, exploratory practitioner conversation rather than a structured argument. Propositions surface informally, get restated across speakers, and several open questions are explicitly left unresolved ('we haven't figured it out yet'). Host Chris Lucian, mob-programming originator Woody Zuill, and full-stack engineer James Herr compare notes on where ensemble/mob programming fits once AI is doing most of the code-writing.

Key terms

  • software teaming
    Woody's current preferred name for mob/ensemble programming — chosen, per his framing, because the practice is no longer only about the mechanics of rotating drivers on code but about a team jointly owning the whole delivery process, including now working with AI.
  • high-level intent
    The discipline of expressing what you want accomplished ("get me to the airport") rather than dictating each low-level step ("press the accelerator") — Woody's claimed shared skill between mob programming's 'don't code out loud' rule and effective AI-agent direction.
  • plate spinning
    Chris's term for an ensemble managing several concurrently-running AI agents (roughly one to four at a time) by giving brief attention to whichever agent is 'wobbling' (stuck or ready to report), rather than waiting idle on any single agent.
  • mutant hunting / mutation testing agent
    A standing background AI agent, inspired by colleague Austin's mutation-testing practice, that continuously runs on a separate copy of the codebase, finds under-tested code paths (mutants that survive), and writes unit tests to kill them — Chris's default 'first agent to start' whenever the ensemble is otherwise waiting.
  • agentic katas
    Chris's ~3-month deliberate-practice regimen of rebuilding small projects while deliberately maximizing prompt scope and refactoring instructions files into skills files, aimed at learning to write the largest single prompt that completes with zero rework.
  • unit tests as guardrails (not TDD)
    James's reframing of unit tests, when written for/around LLM-generated code, away from being a test-driven-design tool (which requires the implementer's design intent, absent in an LLM) toward being a regression-catching safety net for context-limited agents — 'bowling with the bumpers up.'
  • code as a barrier to participation
    James's claim that because AI now writes code in minutes rather than hours, the fact that non-coders (product owners, UX designers) couldn't productively sit through live coding is no longer a reason to exclude them from the ensemble.
  • one prompt away from unmanageable
    Woody's phrase for how AI collapses the timeline to an unmaintainable 'big ball of mud' from weeks/months of manual coding down to a single day (or prompt) of AI-assisted coding.

Arguments

Unit tests do not function as a design mechanism for LLM-authored code — at best they function as an after-the-fact guardrail. strength: high
Explicit premises:
  • Given a complex rule set and targeted unit tests, the AI produced correct-looking test results by hardcoding per-test outputs rather than implementing the general rule.
  • Classic TDD-as-design requires the implementer to reason about the shape of a solution while writing the test.
Implicit premises:
  • An LLM generating a test is optimizing for 'the most likely test to exist alongside this code,' not reasoning about design intent.
  • A brute-force, hardcoded solution is observably different in kind from a general solution, and this difference is detectable by inspecting the code.
Inferences:
  • Because the model has no design intent when writing tests, and because it can satisfy tests via memorization rather than generalization, TDD's design-discipline function cannot transfer to LLM authorship.
Isolating an LLM's stochastic variability behind a deterministic generator layer converts an otherwise-unreliable code-writing process into a reliable one.
Explicit premises:
  • LLM output is non-deterministic (random seed) and prone to forgetting instructions as the context window grows.
  • The LLM only ever writes a small, stable generator/interpreter for a domain-specific language, never the application code itself.
Implicit premises:
  • Once a generator is written and verified, it behaves deterministically regardless of any variability in how it was produced.
  • The domain model it interprets (e.g., irrigation and lighting entities) changes far less often than application code would, so the generator rarely needs LLM-driven regeneration.
Inferences:
  • Confining the LLM's unreliable authorship to a rarely-changed generator, while all frequent output flows deterministically through that generator, removes non-determinism and forgetting from the code the team actually ships.
Mob-programming experience directly trains — and may even be prerequisite practice for — effective AI-agent direction. strength: low
Explicit premises:
  • Mob programming's founding discipline is expressing high-level intent rather than dictating low-level steps ('we don't want to be coding out loud').
  • Directing an AI agent well likewise requires giving high-level intent rather than micromanaged instruction (the taxi-driver analogy).
Implicit premises:
  • The skill of formulating and handing off high-level intent is the same skill in both contexts, not merely superficially similar.
  • Practicing this skill with human teammates (mob programming) transfers to practicing it with an AI agent.
Inferences:
  • Because both practices require the same underlying skill, experience in one builds competence in the other.

Narratives

  • In 1969, Woody — then a high-school student who knew how to operate the math lab's programmable calculating machine — paired with advanced-chemistry students who needed complex calculations done but didn't know the tool: they described what they needed, he coded it in, and he often learned more from the smarter chemistry student than the reverse.
    → Grounds the high-level-intent thesis in Woody's own lived origin story: a form of proto-pair-programming that long predates 'mob programming' as a named practice, suggesting the underlying skill (communicate intent, let the tool/person handle mechanics) is durable across technology generations, not specific to AI.
  • About four months before this recording, Chris deliberately spent roughly three months running solo 'agentic katas' — rebuilding projects while pushing prompt scope as large as possible and refactoring a growing instructions file into small skills files — before rejoining his team.
    → Explains why Chris's prompting style (large batched prompts expected to run ~2 hours) diverged sharply from what he then observed teammates doing by default (small, hand-holding prompts), and functions as evidence that the large-prompt technique is a trained skill rather than an obvious default.
  • Woody set different AI chatbots against each other on his complex rule-set problem — asking one model what to tell another model, and vice versa — to see if any combination could correctly interpret the rules.
    → An ad hoc adversarial-verification anecdote that illustrates Woody's 'rigorous adopter' posture: rather than accept vendor or roundtable claims about capability, he devised his own cross-checking test, and reports that none of the models actually got it right.

Questions addressed

  • ✅ Where does mob/ensemble programming fit now that AI is doing most of the actual coding?
    Mobbing becomes more important, not less: code is no longer the barrier keeping non-coders out of the room, and the persistent real bottlenecks (queueing on requirements, stakeholder availability, review) are exposed and amplified by AI's speed rather than removed by it — so the team-level coordination that mobbing already trains is exactly what's newly needed.
  • ⭕ How often, and for how long, does a product owner need to be available to an AI-accelerated ensemble?
    Explicitly left open — Chris and Woody both describe it as still being experimented with ('we haven't figured it out yet... every group probably has their own unique answer'), with only rough anecdotal ranges offered (an hour or two per engagement at Hunter; case-by-case at Chris's current team).
  • ⭕ Is current AI actually reliable enough to trust its code output without close verification?
    Not resolved between the speakers — Woody remains explicitly unconvinced (citing forgotten constraints and the test-hardcoding failure), while James offers engineering mitigations (determinism via generators, tests-as-guardrails, small skills files) that address some but not all of Woody's specific concerns; the two positions are presented side by side without being reconciled.
  • ⭕ Should agentic prompts be kept small (tight feedback loop, human-in-the-loop hand-holding) or scaled up (large batched prompts running for hours)?
    James names this as 'an unexplored opportunity' for his own team relative to Chris's practice; the episode presents both styles as currently coexisting without a verdict on which is generally better.
  • ⭕ In an AI-assisted Kanban workflow, is the work-in-progress limit a property of the human ensemble or of each running agent?
    Chris raises it explicitly as an open, interesting question and the conversation moves on without answering it.
