The Pulse: AI load breaks GitHub – why not other vendors?

The Pragmatic Engineer · email · 2026-05-07 source ↗ Reader 🖥️

Organizational DebtTechnical Debt
GitHub's ongoing reliability crisis is not adequately explained by AI-driven load increases alone; it reflects deeper organizational and technical failures that competitors have avoided, making GitHub increasingly unfit for professional software development.

practical · argumentative — The article argues that GitHub's reliability crisis is partly self-inflicted due to poor planning, not solely caused by AI load, and implicitly argues professionals should consider alternatives. It uses evidence (outage data, quotes, competitor comparisons) to build a case.

Key terms

  • zero nines
    Reliability below 90% uptime — a deliberate inversion of the industry standard 'number of nines' metric (99.9%, 99.99%) to emphasize how far GitHub has fallen
  • data integrity incident
    A failure where GitHub's merge queue silently produced incorrect merge commits, losing code — distinguished from mere availability outages as a categorically worse class of failure
  • innovator's dilemma
    GitHub's pre-AI workflows and capacity planning, which made it successful, now prevent it from adapting to the AI-agent-driven usage pattern shift
  • organizational debt Organizational Debt
    The coordination overhead of 4,000 employees (1,000 engineers) with cross-team dependencies that slow response to crises
  • self-hosted forge
    Open-source, locally hosted alternatives to GitHub (e.g., Forgejo) that give teams full control over availability

Arguments

GitHub's reliability crisis is not primarily caused by AI load but by organizational and technical failures. strength: high
Explicit premises:
  • GitHub's load increased 3.5x over two years — a manageable increase for a well-resourced organization
  • Competitors handling similar load spikes are not experiencing comparable reliability problems
  • Google prepared for 10x load increases a year before GitHub even began planning for it
  • GitHub has 18 years of tech debt and 4,000 employees creating organizational overhead
Implicit premises:
  • A well-run engineering organization backed by the world's largest tech company should be able to handle a 3.5x load increase over two years
  • The comparison with competitors is valid despite differences in scale and architectural complexity
  • The ability to forecast and plan for load changes is a basic competency expected of infrastructure companies
Inferences:
  • Microsoft's management of GitHub may be a root cause — the organizational bloat and slow planning are symptoms of large-company dysfunction
  • GitHub's monopoly position in developer tools reduced its incentive to invest in resilience pre-crisis
GitHub is no longer suitable for professional software development work. strength: high
Explicit premises:
  • GitHub has been experiencing daily outages lasting 2-3 hours on average for 90 days
  • A data integrity bug silently corrupted merge results for 2,092 pull requests
  • GitHub's leadership downplayed the severity rather than issuing sincere apologies
  • Reliability is at 'zero nines' (85.51% uptime)
Implicit premises:
  • Professional software development requires tools with at least 99.9% availability
  • A platform that can silently corrupt your code cannot be trusted regardless of availability metrics
  • A vendor that downplays data integrity incidents cannot be trusted to prioritize fixing them
Inferences:
  • The reputational damage may be irreversible for some users even if GitHub eventually fixes reliability
GitHub's CTO explanation (AI load) is partially a deflection from organizational accountability. strength: medium
Explicit premises:
  • The CTO's chart had no Y-axis, obscuring the actual magnitude
  • GitHub only started capacity planning for major growth in October 2025, months after the industry recognized the trend
  • The actual load increase was 3.5x, not the dramatic spike the chart visually implied
Implicit premises:
  • Publishing a chart without a Y-axis is a deliberate choice to create a more alarming visual impression
  • Blaming external forces (AI agents) is more palatable than admitting planning failures
  • Engineering leadership is accountable for forecasting, not just reacting

Narratives

  • Mitchell Hashimoto, founder of HashiCorp and creator of Ghostty, kept a daily journal marking days with GitHub outages; almost every day got an X. After 18 years on GitHub, he announced he was leaving.
    → Provides a high-credibility, emotionally resonant case for the proposition that GitHub is unfit for professional work. Hashimoto's stature in the developer community makes his departure a signal, not just a complaint.
  • Can Duruk at Modal discovered GitHub had silently corrupted their repository's merge history and had to manually recover lost commits; GitHub's COO used inflated denominators to minimize the impact.
    → Supports the proposition that GitHub's leadership is downplaying severity. The contrast between the user's emergency (digging into git history to recover code) and the executive's PR framing ('huge denominator') is the argument.
  • Google's SRE teams had been preparing for 10x code volume increase a year before GitHub even started planning for 10x capacity.
    → Supports the proposition that GitHub's failure to forecast was avoidable — the signals were industry-wide.

Questions addressed

  • ✅ Why has GitHub's reliability been so poor recently?
    GitHub CTO attributes it to a 3.5x load increase from AI agents, but the author argues the real causes are tech/organizational debt, a simultaneous Azure migration, and failure to plan ahead.
  • ⭕ Why aren't other vendors suffering the same AI-load-driven outages?
    The author notes that competitors are handling it but does not deeply analyze why — he gestures at GitHub's unique tech debt and organizational size but does not compare architectures.
  • ✅ What should developers do about GitHub's reliability problems?
    Consider alternatives: GitLab, Bitbucket, self-hosted forges like Forgejo, or wait for new startups.
  • ⭕ Is Microsoft's ownership of GitHub a factor in these failures?
    The author implies it strongly but does not make the argument explicitly — noting Microsoft has 'more resources than any competitor' yet failed.

The comparison needs to account for architectural complexity, not just load volume.
