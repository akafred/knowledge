Key terms

  • skill Agent Skills
    A reusable, markdown-packaged, slash-invoked workflow (e.g. /wayfinder, /to-spec, /code-review) shipped in his Claude Code 'skills' repo that encodes a step of his development process as a portable, agent-runnable procedure.
  • /wayfinder
    The new headline skill that takes an oversized plan and breaks it into 'agent-sized chunks with blocking relationships', mapping them as GitHub issues with blocking edges so work can be parallelized across sessions.
  • grilling
    An adversarial interrogation step in which the agent interviews the human to reach alignment before building; v1.1 fixes it so the agent no longer 'self-grills' and no longer jumps straight to implementation.
  • agent-sized chunks
    Units of work scoped small enough for a single agent/session to execute coherently — the right-sized pieces /wayfinder decomposes a large plan into.
  • blocking relationships / blocking edges
    Declared dependency ordering between the decomposed issues (issue B is blocked by issue A), forming a dependency graph that tells the orchestration which chunks may run in parallel and which must wait.
  • spec vs PRD (/to-spec, /to-tickets)
    A deliberate rename: '/to-prd' becomes '/to-spec' because 'the name now matches what we're actually building', and '/to-issues' becomes '/to-tickets' as 'unified terminology that's not GitHub-biased' — favouring a tool-agnostic vocabulary over PRD/GitHub-specific labels.

Arguments

A plan too big for one session should be decomposed into agent-sized issues with blocking edges and parallelized, rather than run as a single sequential plan. strength: low
Explicit premises:
  • A plan can be too big for one session ('When your plan is too big for one session...').
Implicit premises:
  • A single sequential plan can only be executed by one agent at a time, so it caps throughput.
  • Independent, agent-sized chunks can be worked in parallel across sessions and agents.
  • Blocking edges preserve the necessary ordering constraints, so parallelization does not sacrifice correctness of dependency order.
Inferences:
  • Therefore mapping the plan to a dependency graph of issues (some parallelizable, some blocked) lets more work proceed at once while respecting order.
