LLM Knowledge Bases

Andrej Karpathy · tweet · 2026-04-10 Reader 🖥️

Knowledge BaseRetrieval & RAGToken Throughput
LLMs can serve as compilers and operators of personal knowledge bases — ingesting raw sources into structured markdown wikis, then answering queries, generating outputs, and self-improving the wiki — shifting the human role from writer to curator and questioner.

practical · informal essay / social media post — A practitioner's workflow description with implicit theoretical claims about knowledge management and LLM capabilities. Structured as a how-to with commentary on why it works.

Key terms

  • compile Knowledge Base
    Transform raw, heterogeneous source documents into a structured, interlinked markdown wiki using an LLM — analogous to compiling source code into an executable.
  • wiki Knowledge Base
    A local directory of interlinked markdown files maintained entirely by an LLM, serving as a structured knowledge base.
  • token throughput Token Throughput
    The volume of LLM tokens consumed per unit time, here shifting from code generation to knowledge manipulation.
  • knowledge base Knowledge Base
    A personal, LLM-compiled and LLM-queryable repository of structured information on a research topic.
  • RAG Retrieval & RAG
    Retrieval-Augmented Generation — a technique Karpathy expected to need but found unnecessary at this scale thanks to LLM-maintained indexes.

Arguments

LLM-compiled personal wikis are a superior way to manage research knowledge compared to manual note-taking or traditional tools. strength: medium
Explicit premises:
  • LLMs can incrementally compile raw documents into structured markdown wikis.
  • The resulting wiki supports complex Q&A without fancy RAG at moderate scale.
  • LLM health checks can maintain data integrity and discover new connections.
  • Query outputs can be filed back, making the knowledge base accretive.
Implicit premises:
  • Manual knowledge organization does not scale as well as LLM-driven organization.
  • Markdown is a sufficiently expressive format for structured knowledge.
  • The quality of LLM-generated summaries and categorizations is adequate for research use.
Inferences:
  • Therefore, practitioners should shift from writing knowledge artifacts to curating LLM-generated ones.
  • Therefore, this is a viable product category.
RAG is unnecessary at moderate knowledge base scale if the LLM maintains good indexes. strength: low
Explicit premises:
  • The wiki is ~100 articles and ~400K words.
  • The LLM auto-maintains index files and brief summaries.
  • The LLM reads all the important related data fairly easily at this scale.
Implicit premises:
  • Context window sizes of current LLMs are large enough to hold indexes plus relevant articles simultaneously.
  • The quality of LLM-curated indexes is comparable to or better than vector similarity search for this use case.
Inferences:
  • At small scale, well-structured text indexes outperform or match vector-search RAG.

Questions addressed

  • ✅ How can LLMs be used for personal knowledge management beyond code generation?
    By compiling raw sources into a structured markdown wiki, then querying and enhancing it iteratively.
  • ✅ Is RAG necessary for querying a personal knowledge base?
    Not at moderate scale (~400K words) if the LLM maintains good index files and summaries.
  • ⭕ What is the natural evolution of a context-window knowledge base?
    Fine-tuning is mentioned as a direction but not yet implemented or evaluated.
