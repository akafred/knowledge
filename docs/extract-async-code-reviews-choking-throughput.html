From Async Code Reviews to Co-Creation Patterns

Dragan Stepanović (published via Ben Linders / InfoQ) · article · 2026-06-01 source ↗ Reader 🖥️

DORA & Engineering MetricsLean & Toyota WayPair & Mob ProgrammingPull Requests & Async Code Review
Pull-Request-based async code review forces teams to trade off quality against throughput — small PRs incur exponentially worse wait-time-per-size and large PRs degrade reviewer engagement — and the only way out of the trade-off is to make code review continuous and synchronous via co-creation patterns (Pair and Mob Programming), which drive the transaction cost toward zero and let teams have both throughput and quality.

practical · argumentative — Stepanović argues for adopting Pair/Mob programming over PR-based async review. Uses theoretical scaffolding (Lean, Theory of Constraints, Little's Law, DORA) and empirical data (40,000+ PRs across 40+ repos) to back a prescriptive recommendation. The ultimate aim is action: change how teams work.

Key terms

  • Async code review Pull Requests & Async Code Review
    Code review where the author finishes a batch of work, raises a PR, and waits for a separate reviewer to provide feedback at some later time — a workflow whose defining property is the time delay between authoring and feedback
  • Co-creation patterns Pair & Mob Programming
    Pair and Mob Programming — ways of working where multiple people start together, work together, and finish together on the same problem at the same time; the defining property is continuous, in-context, synchronous review folded into authorship
  • PR batch transaction cost Pull Requests & Async Code Review
    The cost (real or perceived) of pushing a batch of work through the code review stage — when high, it incentivizes batching more inventory in front of the stage (bigger PRs)
  • Wait time per size Pull Requests & Async Code Review
    The PR queue time normalized by lines-of-code changed — empirically observed to grow exponentially as PR size shrinks under async review
  • Flow efficiency DORA & Engineering Metrics
    Time spent actively working on an item as a percentage of its total lead time; under async with small PRs the wait component balloons and flow efficiency takes a nosedive
  • LGTM syndrome Pull Requests & Async Code Review
    The pattern where reviewers, facing a large PR they cannot meaningfully steer, capitulate and approve with 'Looks Good To Me' rather than do the costly work of substantive review
  • WIP (Work In Process) DORA & Engineering Metrics
    Items started but not finished; the author notes that under async, real WIP is typically 2x or more the visible WIP because each waiting author pulls in additional work
  • Little's Law DORA & Engineering Metrics
    Average cycle time = average WIP / average throughput; invoked to explain why the WIP inflation caused by async review structurally degrades cycle time
  • Quality-enabled speed Lean & Toyota Way
    The state where building in quality is not a cost paid against speed but a precondition for it — achievable when course correction happens at the smallest possible unit of change
  • DORA metrics DORA & Engineering Metrics
    Deployment Frequency, Lead Time for Changes, Change Failure Rate, Time to Restore Service — the author cites DORA's stability-AND-throughput finding as parallel to his own throughput-AND-quality claim
  • You Burn, I'll Scrape Lean & Toyota Way
    Deming's metaphor for make-and-inspect systems where authorship and inspection are separated; the author uses it to characterize the incentive structure of async review and contrast with co-creation
  • Flow resiliency
    The team's ability to keep work flowing despite individual unavailability — when pairing breaks down because one person is unavailable, the answer is to add more people to the session (mob), not to revert to async

Arguments

Async code review forces an unwinnable trade-off between throughput and quality. strength: high
Explicit premises:
  • Big PRs degrade reviewer engagement per size and produce LGTM-syndrome merges (quality loss).
  • Small PRs under async review incur exponentially growing wait-time-per-size and collapsed flow efficiency (throughput loss).
  • Both observations are persistent across 40+ repositories and 40,000+ PRs.
Implicit premises:
  • Engagement-per-size is a valid proxy for review quality.
  • Wait-time-per-size is a valid proxy for throughput loss.
  • The two failure modes are the only available regions on the async curve — there is no goldilocks PR size that escapes both.
Inferences:
  • Optimization within the async paradigm is structurally limited; the paradigm itself must change.
  • Teams reporting 'we're doing PRs well' are likely operating at the throughput-loss end or the quality-loss end without noticing.
Reducing PR review transaction cost toward zero (via Pair/Mob) is the only way to recover small batches without losing throughput. strength: high
Explicit premises:
  • Inventory expands to match transaction cost (Lean / Reinertsen).
  • Pair/Mob make reviewer availability immediate and review cost effectively free.
  • Therefore optimal batch size shrinks to ~1 line of code.
Implicit premises:
  • Reducing transaction cost via tooling/automation alone (e.g. faster CI, AI-assisted review) cannot match the cost reduction achieved by physical co-presence — this is not argued but assumed.
  • Human availability is the dominant cost driver in review, not artifact comprehension time.
Inferences:
  • Any intervention that reduces review wait time without changing the work model (e.g. SLAs on PR response) is a partial fix that will plateau before reaching the co-creation regime.
  • AI-assisted review, if it provides immediate in-context feedback, may be a partial substitute for human co-creation on the transaction-cost axis.
Co-creation delivers BOTH throughput AND quality (not a trade-off). strength: medium
Explicit premises:
  • DORA research showed stability and throughput have an AND relation, not OR.
  • Co-creation eliminates wait time AND reduces processing time AND reduces rework.
  • Pair/Mob reduce WIP from ~2N (async) to 2 (pair) or less (mob).
Implicit premises:
  • The DORA finding (stability+throughput AND-relation) generalizes to throughput+quality — these are treated as analogous without argument.
  • The reduction in rework outweighs the doubled (or higher) people-cost of pairing/mobbing — an economic claim never quantified.
Inferences:
  • Teams measuring developer productivity by lines/PRs per individual will systematically under-credit co-creation.
  • If rework reduction is an order of magnitude, even pair programming at 2x people-cost is net positive on throughput.
Objections to co-creation (schedule flexibility, remote work, individual preference) are mostly mis-framed and dissolve under closer inspection. strength: medium
Explicit premises:
  • Remote and async are orthogonal — co-creation works remotely.
  • Schedule slippage signals low flow resilience, not a need for async — mob increases resilience.
  • Failed pair/mob attempts are typically failures to learn the practice, not failures of the practice (Woody Zuill anecdote).
Implicit premises:
  • Five hours of working-hours overlap is achievable for most teams — author asserts as a 'good target' without defending.
  • Individuals who prefer solo work after genuine exposure are rare enough to treat as edge cases.
Inferences:
  • Distributed teams across many time zones either need restructuring or are accepting a real cost.
  • 'This isn't for us' is often premature — but the author allows it as a legitimate landing point.

Narratives

  • Ema and Luka — Ema raises a PR, Luka is busy, Ema starts something else to feel productive, by the time Luka reviews Ema is no longer in context, the ping-pong drags on for days.
    → Embodies the WIP-inflation argument — shows in human terms why async review structurally produces more WIP than the headcount suggests, and why context decays.
  • Amazon Prime / shipping cost analogy — when shipping is $3 you batch your basket; when it's free (Prime) you order single items as needed.
    → Supports the proposition that batch size expands to match transaction cost; transfers the reader's everyday economic intuition onto the unfamiliar PR domain.
  • Slow test suite — if tests take 20 minutes, I won't run them after every line change, so I batch up changes before getting feedback.
    → Generalizes the transaction-cost argument beyond code review to any feedback loop; primes the reader to see the same mechanism in CI, deployment, etc.
  • Online clothing same-day vs. one-week delivery — under slow delivery you order multiple sizes; under fast delivery you order one and iterate.
    → Reinforces the transaction-cost-drives-batch-size point with a second consumer-economics analogy.
  • Production outage response — everyone jumps on a call together to resolve it.
    → Used as a Goldratt-style 'emergency rules are actually good rules' argument — co-creation is what teams already do when stakes are high; the question is why it isn't the default.
  • Woody Zuill's Mob Programming workshop — first whole day spent on working-together principles (kindness, consideration, timely contribution).
    → Supports the proposition that co-creation is a learnable skill teams typically lack, and that 'we tried it and it didn't work' often means 'we didn't actually try it.'
  • 40,000+ PRs across 40+ very active product development repos — the author's own study.
    → Empirical backbone of the engagement-per-size and wait-time-per-size claims; differentiates this argument from prior opinion-based critiques of PRs.

Questions addressed

  • ✅ How does PR-based async code review actually perform on throughput and quality in product development teams?
    Empirically poorly on both axes: large PRs lose engagement per size; small PRs lose throughput via exponential wait-time-per-size; flow efficiency collapses below a threshold size.
  • ✅ Why does shrinking PR size — the conventional Lean advice — fail to deliver throughput gains under async review?
    Because review transaction cost stays constant while batch size shrinks, the wait-to-work ratio explodes and flow efficiency tanks.
  • ✅ Is there a way to have both small batches and high throughput?
    Yes — drive review transaction cost toward zero via continuous synchronous review (Pair/Mob).
  • ✅ Is co-creation compatible with remote work and individual schedule flexibility?
    Yes — remote/async orthogonality; mob programming provides flow resilience for schedule slippage; flexible start/end times can be an advantage.
  • ⭕ What is code review actually FOR?
    Author assumes it's for building in quality and providing human judgement in a learning loop. Never engages with audit/gatekeeping/separation-of-duties/compliance framings.
  • ⭕ Can tooling (AI-assisted review, faster CI, response-time SLAs) reduce review transaction cost enough to make async viable?
    Not addressed. Article predates widespread AI coding agents and treats human availability as the only lever on transaction cost.
  • ⭕ What is the actual economic cost of pairing/mobbing relative to its rework reduction?
    Claimed 'order of magnitude less rework' but no comparison to the doubled (or higher) people-cost; net economics asserted, not shown.

Gaps

  • incompleteThe article treats human co-presence as the only lever on review transaction cost. It does not consider AI-assisted in-IDE review or AI pair-programmers as potential substitutes for some of the human availability cost. Given the explicit framing ('inventory expands to match transaction cost'), AI review tools that provide immediate, in-context feedback are a direct theoretical implication the author doesn't draw.
    Read next: Cursor / Copilot / Claude Code workflow studies; AI-as-pair-partner literature; any post-2024 work on continuous AI review folded into authoring.
  • incompleteNo engagement with audit, compliance, separation-of-duties, or regulatory framings of code review. For teams in regulated industries (finance, healthcare, defense), the 'PR as gate / paper trail' function may be the dominant requirement — not the learning function the author optimizes for.
    Read next: DORA's discussion of change approval; SOX/PCI-DSS code review requirements; literature on continuous compliance and pairing as audit substitute.
  • incompleteTrunk-based development and feature flags are notably absent. The article frames the alternative to async-PR review as Pair/Mob, but the modern stack often combines trunk-based + feature flags + small commits + light async review. The author doesn't engage with this hybrid.
    Read next: Accelerate (Forsgren/Humble/Kim); trunk-based development primary sources; Paul Hammant on TBD; Pete Hodgson on feature flags.
  • incompleteNo quantitative analysis of pairing/mobbing's people-cost vs. rework savings. The 'order of magnitude less rework' claim is anecdotal; whether 2N developers paired produces > N solo developers' net throughput is asserted, not shown.
    Read next: Williams/Kessler on pair programming productivity; Allan Kelly's mob programming case studies; any controlled studies comparing pair vs. solo throughput.
  • incompleteReviewer-as-knowledge-distributor function is largely unaddressed. PRs serve as a passive knowledge-sharing mechanism for team members not involved in authoring. Mob covers this for participants but raises the question of those NOT in the session.
    Read next: Codebase ownership patterns; bus factor / knowledge silos; mob rotation strategies for cross-team knowledge.
  • incompleteThe article assumes review is primarily about correctness/quality of code-as-written. It doesn't address review's role in architectural drift, dependency hygiene, security review, or other concerns where a fresh reviewer's distance from the work might be a feature, not a bug.
    Read next: Threat modeling literature; architecture decision records; literature on outside-in review for security and design coherence.
  • uninformedThe article predates and doesn't engage with the AI-accelerated coding context where a developer can produce a working diff in minutes that previously took hours. If authoring time collapses dramatically, the time-cost ratio between authoring and review shifts — possibly making async review even more dominated by wait, OR potentially making AI-assisted self-review viable in ways that change the calculus.
    Read next: Beck/Grinich on AI coding agents; Karpathy and Gartner-style claims about AI-native development workflows; empirical work on PR throughput in AI-heavy teams.
    closed byAgentic Code Review — Supplies the AI-accelerated analysis that gap flagged as missing: when authoring time collapses toward zero while human reading speed is fixed, the authoring/review ratio inverts and async review becomes the dominant, wait-heavy bottleneck rather than the cheap step — quantified by 2026 telemetry (median review duration up 441.5%, zero-review merges up 31.3%). It also answers the gap's second branch — AI-assisted self-review is viable but only as a sensor, since a model cannot reconstruct intent.
