Architecture as Code to Teach Humans and Agents About Architecture

Neal Ford and Mark Richards · rss · 2026-04-15 Reader 🖥️

Coding AgentsEvolutionary Architecture & Fitness FunctionsFeedback LoopsGuardrails
Software architecture should be defined as code — capturing structural constraints deterministically — so that both human developers and AI agents can receive fast feedback on whether implementations conform to design intent.

practical · argumentative — A short promotional essay for the forthcoming O'Reilly book 'Architecture as Code.' It argues that architects should define architectural constraints in code so that both humans and AI agents can use them as deterministic guardrails.

Key terms

  • Architecture as Code Evolutionary Architecture & Fitness Functions
    Expressing all aspects of software architecture — structure, constraints, relationships, dependencies — in executable, version-controlled code rather than diagrams or prose, so that conformance can be verified automatically.
  • feedback framework (vs. testing framework) Evolutionary Architecture & Fitness Functions
    A system that notifies architects when implementation diverges from design, framing divergence as information (potential feedback) rather than as failure (test breakage). A new component is not an error — it is a signal to assess.
  • deterministic constraints Evolutionary Architecture & Fitness Functions
    Objective, measurable architectural rules (e.g., cyclomatic complexity limits, dependency direction rules) that can be evaluated computationally without human judgment.
  • foundational constraints vs. desired behavior Guardrails
    A separation in agentic AI between inviolable rules (hard architectural boundaries) and goals the agent works toward. Foundational constraints are non-negotiable; desired behavior is aspirational.
  • nine intersections Evolutionary Architecture & Fitness Functions
    The nine areas where software architecture intersects other parts of the development ecosystem: implementation, engineering practices, infrastructure, generative AI, team topologies, business concerns, enterprise architecture, data, and integration architecture.

Arguments

Architecture should be defined as code to create feedback loops between architects and developers. strength: medium
Explicit premises:
  • Architects design components with careful thought about coupling, cohesion, and structural concerns.
  • Once a design is handed to a team, the architect has no way to verify the team implements it correctly.
  • Defining components in code with verifications allows architects to both illustrate and get feedback on the design.
  • Design should sometimes change to reflect implementation realities.
Implicit premises:
  • Diagrams and documents alone are insufficient to maintain architectural integrity over time.
  • The gap between design intent and implementation is a significant source of architectural drift.
  • Architects are not embedded deeply enough in implementation to catch divergence through code review alone.
Inferences:
  • Therefore, architecture-as-code is not just documentation but a living governance mechanism.
  • Therefore, the architect's role shifts from upfront designer to continuous feedback processor.
Architecture as Code is essential for effective agentic AI in software development. strength: medium
Explicit premises:
  • Agentic AI can work toward a solution as long as deterministic constraints exist.
  • LLMs tend toward brute-force solutions without explicit constraints.
  • Architectural constraints (complexity limits, coupling rules) can prevent brute-force code generation.
  • Separating foundational constraints from desired behavior is an increasingly common agentic AI practice.
Implicit premises:
  • AI agents will play a significant and growing role in software implementation.
  • The quality of agent-generated code is primarily bounded by the quality of constraints provided to the agent.
  • Human code review alone is insufficient to catch architectural violations in agent-generated code at scale.
Inferences:
  • Therefore, the architect's primary deliverable shifts from designs to constraints.
  • Therefore, organizations without codified architectural constraints will get worse outcomes from AI agents than those with them.

Narratives

  • The authors started writing a book about architecture-as-code through their usual iterative process (workshops, conference presentations, online classes), then 'the entire industry shifted' with the arrival of agentic AI, giving their concept an unexpected new audience and purpose.
    → Establishes the timeliness and serendipity of the concept — it was designed for human architects but turns out to be exactly what AI agents need, lending it a sense of inevitability.
  • An architect defines components via a diagram, hands the diagram to a team, and cannot be sure the team implements it correctly. With architecture-as-code, a new component triggers a notification rather than silently drifting.
    → Illustrates the feedback-framework concept and the key distinction from testing: divergence is information, not error.
  • An LLM asked for an algorithm that touches all 50 US states generates a 50-stage switch statement — brute force. A cyclomatic complexity limit would force it to find a better solution.
    → Concrete example of how deterministic architectural constraints prevent pathological agent behavior. The only specific example in the essay.

Questions addressed

  • ✅ How can architects maintain design integrity when implementation is done by others (humans or agents)?
    By defining architecture as code with verifications, creating a feedback framework that alerts architects to divergence.
  • ✅ How can AI agents be guided to produce architecturally sound code?
    By providing deterministic architectural constraints as foundational rules, separating them from desired behavior.
  • ⭕ What aspects of architecture can be expressed as code?
    Nine intersections are listed (implementation, engineering practices, infrastructure, generative AI, team topologies, business concerns, enterprise architecture, data, integration) but none are elaborated beyond naming.
  • ⭕ What tooling, languages, or frameworks support architecture-as-code?
    Not addressed at all. The essay describes the concept without any concrete tooling.
  • ⭕ How effective is architecture-as-code in practice — does it actually reduce architectural drift or improve agent output quality?
    Not addressed. No empirical evidence, case studies, or metrics are provided.
