Teaching an AI Agent to Debug Flaky Tests

Igor Kulakov · article · 2026-05-09 source ↗ Reader 🖥️

Agent-Native DesignAgent SkillsCoding Agents
Flaky tests can be debugged deterministically by diffing code-coverage reports from passing and failing runs to locate where execution paths diverge, and this entire procedure can be packaged as a reusable AI Agent Skill so that an agent follows it without human guidance.

practical · tutorial-narrative — A walkthrough that builds from intuition to tooling to a reusable AI Skill, using a concrete TOCTOU race condition as the running example. Part tutorial, part methodology argument.

Key terms

  • AI Agent Skill Agent Skills
    A plain-text document (SKILL.md) in a discoverable location that describes a reusable procedure, conventions, or domain knowledge an AI agent can follow — the AI equivalent of a knowledge base article
  • flaky test
    A test that returns both passes and failures despite no changes to code or test
  • coverage diff
    A comparison of per-line hit counts from coverage reports across multiple test runs, summarizing lines whose execution counts vary between passing and failing runs
  • hit counting
    Recording the number of times each line is executed (not just whether it was hit), enabling detection of concurrency bugs where the same code runs more times than expected
  • TOCTOU (time-of-check to time-of-use)
    A race condition where a thread checks a condition and then acts on it, but another thread changes the state between check and use

Propositions

  • explicitFlaky tests must take different code paths on passing vs. failing runs; otherwise results would be consistent.conf: high
  • explicitTest coverage tools, though not designed for tracing, can record the code-path information needed to diagnose flakiness.conf: high
  • explicitHit counting (execution counts per line) is essential — some flaky bugs differ not in which lines execute but in how many times they execute.conf: high
  • explicitDiffing coverage reports from passing and failing runs can deterministically point to the root cause of flakiness.conf: medium
  • explicitAI Agent Skills make hard debugging procedures accessible to agents that have no prior context about the problem.conf: high
  • implicitThe coverage-diff technique generalizes beyond TOCTOU races to other classes of flaky tests.conf: medium
  • explicitText interfaces are the composable building blocks that allow AI agents to solve complex problems — as long as you can stack text interfaces together, you can teach agents to solve virtually anything.conf: medium

Arguments

Coverage-diff is a sound diagnostic for flaky tests strength: medium
Explicit premises:
  • P1: Flaky tests must take different code paths
  • P2: Coverage tools can record code-path information
  • P3: Hit counting captures frequency differences, not just presence/absence
Implicit premises:
  • The code-path differences that cause flakiness are observable at the line/branch level of granularity
Inferences:
  • If flakiness implies path divergence, and coverage captures paths, then coverage-diff reveals the divergence point
Packaging the procedure as an AI Skill makes debugging deterministic strength: medium
Explicit premises:
  • The coverage-diff procedure is mechanical and repeatable
  • AI Skills encode procedures so agents follow them step-by-step
Implicit premises:
  • Determinism here means the agent consistently arrives at the correct diagnosis, not that the model output is literally identical
Inferences:
  • Deterministic procedure + faithful execution = deterministic outcome

Narratives

  • From intuition to tool to Skill: the article narrates a single debugging session that progresses from a vague hunch ('different results must mean different paths') through tool-building (coverage agent, text reporter, diff script) to a validated, reusable AI Skill.
    → Demonstrates the methodology by example and implicitly argues that AI-augmented developers should package their insights into reusable Skills

Questions addressed

  • ✅ Can we make AI deterministically find the root cause of flaky tests?
    Yes — by encoding a coverage-diff procedure as an AI Skill, the agent follows deterministic steps to locate the divergence point
  • ✅ What tool can record code paths for comparison between passing and failing test runs?
    IntelliJ IDEA's coverage agent with hit counting, run via CLI through Maven Surefire
  • ✅ How can we convert binary coverage data into a format useful for AI analysis?
    A custom TextCoverageStatistics reporter was added to the open-source IntelliJ coverage agent
  • ✅ Can the entire flaky-test debugging procedure be encoded as a reusable AI Agent Skill?
    Yes — packaged as SKILL.md following the agentskills.io specification; validated in a fresh agent session

Gaps

  • evidentialOnly one bug class demonstrated — the TOCTOU race condition. No evidence the technique works for other common flakiness causes: environment-dependent tests, time-sensitive tests, test-order dependencies, resource leaks, or non-deterministic external services.
  • conceptualThe claim of 'deterministic' debugging conflates procedural repeatability with outcome reliability. The agent's interpretation of the coverage diff is still probabilistic, and the article does not measure how often the agent correctly identifies root causes across varied bugs.
  • practicalThe technique requires the flaky test to actually fail during the collection runs. For tests that fail once in thousands of runs, the script's loop-until-both-outcomes approach may be impractical without guidance on how many runs are needed.
  • conceptualAssumes line-level coverage granularity is sufficient. Some flakiness may involve identical line-level execution with different data values, timing, or ordering that coverage cannot capture.
