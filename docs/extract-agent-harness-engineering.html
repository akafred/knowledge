Agent Harness Engineering

Addy Osmani · article · 2026-04-20 source ↗ Reader 🖥️

Agent HarnessAgent Tool ManagementAgentic LoopAgent Context Files (CLAUDE.md / AGENTS.md)Context Rot & Context LimitsMulti-Agent Orchestration
The performance gap between what current AI models can do and what users actually experience is primarily a harness-engineering gap — the scaffolding of prompts, tools, hooks, context management, and feedback loops wrapped around the model — and closing it requires treating that scaffolding as a continuously evolving engineering artifact shaped by observed failures.

practical · expository — A practitioner synthesis of an emerging engineering discipline. Primarily expository — surveys and organizes contributions from multiple sources (Viv Trivedy, HumanLayer, Anthropic, Simon Willison) into a coherent framework, with prescriptive guidance throughout.

Key terms

  • harness Agent Harness
    Every piece of code, configuration, and execution logic that isn't the model itself — prompts, tools, context policies, hooks, sandboxes, subagents, feedback loops, and recovery paths
  • harness engineering Agent Harness
    The discipline of designing, iterating, and maintaining the scaffolding around an AI model so it can reliably complete tasks; coined by Viv Trivedy
  • ratchet
    The practice of converting each observed agent failure into a permanent constraint (rule, hook, or check) that prevents recurrence, only relaxing constraints when model improvements make them redundant
  • context rot Context Rot & Context Limits
    The degradation of model reasoning and task-completion quality as the context window fills up
  • Ralph Loop Agentic Loop
    An autonomous coding loop where a hook intercepts the model's attempt to exit and re-injects the original prompt into a fresh context window, forcing continuation against a completion goal
  • HaaS (Harness-as-a-Service) Agent Harness
    The shift from building on LLM APIs (which give a completion) to building on harness APIs (which give a runtime with loop, tools, context management, hooks, and sandbox primitives)
  • skill issue
    Agent failures that appear to be model limitations but are actually harness configuration problems

Arguments

Engineers should invest in harness engineering rather than waiting for better models. strength: high
Explicit premises:
  • A decent model with a great harness beats a great model with a bad harness (author's direct experience).
  • On Terminal Bench 2.0, the same model (Claude Opus 4.6) scored far lower in Claude Code than in a custom harness.
  • Viv's team moved from Top 30 to Top 5 by changing only the harness.
Implicit premises:
  • Model improvements are outside the engineer's control, while harness improvements are within it.
  • The rate of harness improvement per unit of engineering effort exceeds the rate of model improvement per unit of waiting time.
Inferences:
  • Therefore the highest-leverage engineering work for agent performance is harness design, not model selection.
Harness engineering is a permanent discipline, not a temporary stopgap until models improve. strength: high
Explicit premises:
  • Every harness component encodes an assumption about what the model can't do on its own (Anthropic).
  • When models improve, old scaffolding becomes unnecessary but new scaffolding is needed for newly reachable tasks.
  • Opus 4.6 eliminated context-anxiety scaffolding but created need for multi-day memory policies and multi-agent coordination.
Implicit premises:
  • There will always be a frontier of tasks just beyond current model capabilities.
  • The gap between raw model capability and reliable task completion is structural, not incidental.
Inferences:
  • The harness moves with the capability frontier rather than being absorbed by it.
Agent mistakes should be treated as permanent signals that become harness constraints (the ratchet). strength: medium
Explicit premises:
  • Failures are usually legible and attributable to specific missing constraints.
  • Adding a convention to AGENTS.md, a hook that blocks a destructive command, or a back-pressure signal from typechecking can prevent recurrence.
  • Rules should only be removed when a capable model makes them redundant.
Implicit premises:
  • Agent failure modes are stable enough to be worth encoding permanently.
  • The cost of maintaining a constraint is lower than the cost of repeated failures.
Inferences:
  • A failure-driven, additive approach to harness configuration produces the most effective harness over time.

Narratives

  • Terminal Bench 2.0 benchmark comparison: Claude Opus 4.6 in Claude Code vs. the same model in a custom harness, where the custom harness moved from Top 30 to Top 5
    → Primary evidence that harness design, not model selection, is the dominant performance variable
  • An agent ships a PR with a commented-out test, which gets merged by accident. This leads to updating AGENTS.md, adding a pre-commit hook to grep for .skip() and xit(), and adding a reviewer subagent that flags commented-out tests as blockers.
    → Concrete illustration of the ratchet methodology — showing how a single failure cascades into three layers of prevention
  • Opus 4.6 killed the context-anxiety failure mode (premature task completion near context limits), eliminating a class of scaffolding the author had been writing six months earlier
    → Illustrates the 'harnesses move, not shrink' thesis — showing a real example of scaffolding becoming dead code as models improve
  • Fareed Khan's architectural breakdown of Claude Code showing all harness concepts as named components in a production system
    → Validates the theoretical framework by showing it instantiated in a shipping product

Questions addressed

  • ✅ What is the most impactful lever for improving AI coding agent performance?
    Harness engineering — the scaffolding around the model — rather than model selection or waiting for better models.
  • ✅ What are the concrete components of an agent harness?
    System prompts, tools/MCPs, filesystem/git, bash execution, sandboxes, memory/search, compaction, hooks, planning/verification loops, subagent orchestration, and observability.
  • ✅ Will better models make harness engineering obsolete?
    No — as models improve, the harness space moves rather than shrinks; new capabilities unlock new failure modes requiring new scaffolding.
  • ✅ How should engineers respond when an agent makes a mistake?
    Treat the mistake as a permanent signal and engineer a constraint (rule, hook, or check) that prevents recurrence — the ratchet.
  • ✅ How do model training and harness design interact?
    There is a feedback loop: useful harness primitives get standardized into products, used in training the next model generation, making the model better at those primitives, which enables new harness patterns.
  • ✅ What is the appropriate size and governance model for AGENTS.md?
    Keep it short (under 60 lines), earn each line by tracing it to a specific past failure — pilot's checklist, not style guide.

Gaps

  • incompleteThe article does not address when harness complexity itself becomes a liability — the scaling problem of accumulating ratchet rules. At what point does the constraint set become too large for the model to follow or for the engineer to maintain?
    Read next: Look for work on constraint prioritization, AGENTS.md hygiene, or automated rule pruning in harness configurations.
    closed byThe Anatomy of an Agent Harness — PARTIAL. Osmani g1 asks when harness complexity itself becomes a liability — the scaling problem of accumulating ratchet rules, and at what point the constraint set is too large. The anatomy supplies a directional criterion: the 'future-proofing test' (a design is sound only if performance scales up with more powerful models WITHOUT added harness complexity) plus the scaffolding-shrinks stance (Manus rebuilt 5x removing complexity; Anthropic deletes planning steps as models internalize them). So complexity that a stronger model would render redundant is, by this test, a liability. It does NOT answer the quantitative half of g1 — no threshold for 'too large for the model to follow / the engineer to maintain,' no maintenance-cost model — so this is a design principle, not the boundary the gap asks for.
  • incompleteThe Terminal Bench 2.0 data point (Top 30 to Top 5 via harness change) is presented without detail on what specific harness changes drove the improvement, making it impossible to evaluate which components had the most leverage.
    Read next: Read Viv Trivedy's original 'Anatomy of an Agent Harness' post for the detailed breakdown.
  • incompleteThe article focuses almost exclusively on coding agents. It does not explore whether the harness-engineering framework applies to non-coding agent domains (research, design, operations, etc.) or where it might break down.
    Read next: Look for harness-engineering writing in non-coding agent contexts — research agents, customer support agents, design agents.
    closed byProject Glasswing: what Mythos showed us — Extends harness engineering beyond coding into vulnerability research — multi-stage harness, parallel hunters, adversarial review — while showing where the coding-agent shape breaks down (single-hypothesis agents vs narrow-and-parallel work) (p2, p9, p10, p12).
  • incompleteThe model-harness co-training feedback loop is identified but its risks are not explored: overfitting to specific tool implementations, lock-in to specific harness vendors, or narrowing of model generality.
    Read next: Read research on post-training specialization tradeoffs and whether tool-specific fine-tuning degrades general capability.
  • incompleteThe article does not discuss cost or economics of harness engineering — when is the investment in harness refinement worth it vs. simply accepting some failure rate or switching models?
    Read next: Look for ROI frameworks for agent infrastructure investment.
    closed byDon't train the model, evolve the harness — Gives a partial answer to the economics-of-harness-investment question: keep refining while each added mechanism clearly beats the current best; once gains flatten, the residual misses are real capability gaps, so the answer to 'invest more vs. switch models' becomes 'switch/wait for a better model past the flatten point.' Adds a ~7x cost-per-task data point. Still qualitative — no explicit cost-benefit threshold.
