AI is not a new paradigm for software engineering — the fundamentals that make working with humans effective (small tasks, vertical slices, deep modules, fast feedback loops, alignment through dialogue, TDD) are exactly what make working with AI agents effective; the path from idea to shipped feature is therefore a deliberately planned, human-driven pipeline (align → PRD → vertical-slice Kanban → AFK implementation → QA) in which the codebase itself is treated as the battleground, not the spec.

practical · mixed — Primarily practical (a workshop that teaches a concrete workflow for shipping production code with AI agents), expository in its conceptual sections (smart zone / dumb zone, deep modules, push vs. pull), and narrative in its demos (the gamification feature lifecycle is one long worked example). The thesis is prescriptive: pre-AI software-engineering fundamentals are what make AI coding work in production.

Key terms

  • smart zone / dumb zone Context Rot & Context Limits
    The smart zone is the portion of an LLM's context window (Pocock pins it at roughly 100k tokens regardless of advertised window size) in which attention relationships are not yet strained and the model makes good decisions; the dumb zone is everything beyond, where token-to-token attention scales quadratically and quality collapses. Originally from Dex Horthy (Human Layer).
  • Memento agent (LLM as the guy from Memento) Context Rot & Context Limits
    An LLM session always resets to its system prompt; the developer should embrace and exploit this — clear context rather than compact — so every session starts in a known, optimal state. Pocock prefers clearing to compacting.
  • specs-to-code (the movement he opposes) Spec-Driven Development
    The view that you author specs, pass them to AI, and only edit the specs when output is wrong — never reading the code. Pocock equates this with 'vibe coding by another name' and rejects it: 'the code is your battleground.'
  • grill-me (skill) Spec-Driven Development
    A short skill that instructs the agent to interview the developer relentlessly, one question at a time, walking down each branch of the design tree, providing its own recommended answer with each question, until shared understanding is reached. Operationalises Brooks's 'design concept'.
  • design concept (Brooks) Software Design Fundamentals
    Borrowed from Frederick P. Brooks's The Design of Design: the shared idea among all participants in a design effort. Pocock's reinterpretation: what the developer needs to reach with the agent before any plan or code.
  • destination document / journey document Spec-Driven Development
    Two complementary planning artifacts. The destination document (the PRD) defines where you're going (problem, solution, user stories, definition of done, out-of-scope). The journey document (the vertical-slice Kanban) defines how to get there.
  • PRD (product requirements document) Spec-Driven Development
    A summary of the design concept written by the agent after grilling; contains problem, solution, user stories, implementation decisions, testing decisions, out-of-scope. Pocock explicitly does NOT read it — he trusts LLMs at summarization once alignment has been established.
  • tracer bullet / vertical slice Agent Planning Artifacts
    From Pragmatic Programmer: a thin slice of functionality that crosses every layer of the system (e.g., schema change + new service + minimal front-end representation) so you get end-to-end feedback after the first slice instead of after the last layer. Pocock contrasts this with horizontal slicing, which AI defaults to (do all schema, then all API, then all front-end).
  • agent-ready issue / independently grabbable issue Agent Planning Artifacts
    A markdown-file issue produced by the PRD-to-issues skill, sized as a vertical slice, with explicit blocking relationships to other issues so the whole backlog forms a DAG that multiple agents can work in parallel.
  • AFK (away-from-keyboard) task vs. human-in-the-loop task Human-in-the-Loop
    AFK tasks can be safely handed to an agent that runs autonomously — implementation almost always qualifies. Human-in-the-loop tasks require a human (or several humans, including domain experts) to sit there because the cost of misalignment is too high — alignment/PRD phases qualify. Issues are tagged in the Kanban as AFK or human-in-the-loop.
  • day shift / night shift Human-in-the-Loop
    Metaphor for the workflow split: humans do the day shift (idea, grilling, PRD, Kanban) which queues work; agents run the night shift (AFK implementation) consuming the queue.
  • Ralph (Wiggum) loop Agentic Loop
    A simple agent loop named after a Simpsons character: specify a destination, then repeatedly tell the agent 'make a small change that gets us closer'. Pocock acknowledges it works but prefers more structure; his once.sh / loop.sh scripts are a structured Ralph loop.
  • deep module vs. shallow module (Ousterhout) Software Design Fundamentals
    From John Ousterhout's A Philosophy of Software Design: a deep module exposes a small interface to a lot of functionality; a shallow module is a small file exporting a small surface. Pocock argues unaided AI produces shallow-module codebases, which fragment test boundaries and degrade agent performance.
  • feedback loops (as ceiling) Feedback Loops
    Automated tests + type checks + lint + build that an agent can run autonomously after each change. Pocock asserts: 'the quality of your feedback loops influences how good your AI can code. Essentially, that is the ceiling.'
  • red-green refactor / TDD with an agent Test-Driven Development
    The agent writes a failing test first, confirms red, implements until green, optionally refactors. Pocock claims this materially reduces test cheating because the test instruments the code before the code exists; the workflow is so valuable he 'warps his whole technique around getting TDD to work better.'
  • push vs. pull (for coding standards) Progressive Disclosure
    Push = unconditionally inject tokens into every agent context (e.g., CLAUDE.md). Pull = make information available behind a description header that the agent can choose to load (skills). Pocock's design: implementer pulls, reviewer pushes — the implementer must be free to ignore irrelevant rules, but the reviewer must always compare against the rules.
  • sub-agent Multi-Agent Orchestration
    An LLM call delegated to an isolated context window that returns only a summary to the parent. Pocock uses it to spend ~94k tokens on exploration without polluting the orchestrator's smart zone.
  • dock rot (doc rot) Technical Debt
    Old PRDs/plans persisting in the repo and being treated as authoritative by future agents even though the code, names, and requirements have drifted. Pocock's answer: don't keep them in the repo; close them as GitHub issues — visible-as-done but retrievable.
  • Sandcastle Coding Agents
    Pocock's TypeScript library for orchestrating parallel agent loops: a planner reads the backlog and chooses parallelizable issues; each issue runs in its own git worktree inside a Docker sandbox with its own Claude instance; a reviewer (Opus) reviews the implementer's (Sonnet) work; a merger merges branches and resolves conflicts.
  • QA (manual, by the developer)
    Pocock's most opinionated step. After AFK implementation, the developer manually exercises the feature to 'impose my opinions back onto the codebase, how I impose my taste'. Teams that try to automate QA 'end up with apps that lack taste and are bad'. QA is also the place new Kanban issues are generated.

Arguments

Therefore, the AI-coding workflow must be a deliberately planned pipeline (Align → PRD → Vertical-Slice Kanban → AFK Implementation → Manual QA) rather than an open-ended chat session or a spec-driven autopilot. strength: high
Explicit premises:
  • LLM quality degrades quadratically with context (smart zone vs. dumb zone).
  • Sessions reset (the Memento metaphor), so any workflow should exploit fresh starts.
  • Misalignment is the primary failure mode, not capability.
  • Specs-to-code/vibe coding fails because the code itself is the battleground.
  • Vertical slices give end-to-end feedback after the first slice; horizontal layering does not.
  • AFK tasks can be parallelized; alignment tasks cannot.
Implicit premises:
  • The developer remains accountable for product quality regardless of how much work was done by agents.
  • The codebase has (or can be brought to) deep modules, automated tests, types, and lint that an agent can run autonomously.
  • The developer is fluent enough in the codebase, tests, and standards to QA the agent's output meaningfully.
  • Token cost is not the binding constraint at this scale of work (sub-agent burns 94k tokens to explore; not problematized).
  • Trust in the agent is engineered through structure (skills, prompts, sandboxes), not through the agent itself becoming more reliable.
Inferences:
  • Teams that lack the test/type/lint infrastructure to provide tight feedback loops cannot use this workflow effectively, regardless of how good the agent gets.
  • Frontend-heavy or AI/ML/embedded codebases — where feedback loops are partial or human-eye-dependent — sit outside the comfort zone of this workflow.
  • The role 'engineering leader' must invest in test/type/architecture infrastructure as the prerequisite for the team capturing any AI-coding leverage at all.
  • Adopting this workflow likely INCREASES senior-developer load (more PRD writing + more code review + more taste imposition) and may not free up senior time the way naïve narratives suggest.
Codebase architecture (deep modules + tight feedback loops) is the primary lever for improving AI-agent output. strength: high
Explicit premises:
  • Bad codebases make bad agents.
  • Shallow modules fragment test boundaries and force the agent to manually traverse a dependency graph.
  • Feedback-loop quality is 'the ceiling' on agent quality.
  • Deep modules with simple interfaces let humans hold the codebase in their head and delegate implementation behind interfaces.
Implicit premises:
  • A developer who cannot personally judge what 'deep' means in their domain cannot benefit from this prescription.
  • There is time and authority in the organization to refactor toward deep modules before benefits accrue.
  • Ousterhout's notion of module depth transfers cleanly across domains (web app, embedded, ML pipeline, infra).
Inferences:
  • Teams stuck in shallow-module codebases (typical microservice sprawl, scattered utility files, legacy procedural code) will see disproportionately poor AI-coding leverage and should invest in architectural consolidation as an AI-readiness move.
  • 'Improve codebase architecture' is a legitimate first AI-coding investment, even before adopting any agent workflow.
Push vs. pull is the right framing for coding-standard enforcement: implementers pull, reviewers are pushed. strength: medium
Explicit premises:
  • An implementer drowning in always-loaded rules enters the dumb zone unnecessarily.
  • Skills with descriptive headers let the agent decide when to load context relevant to its current task.
  • A reviewer must always compare against the standard; the standard must therefore be in its context unconditionally.
Implicit premises:
  • The agent is sufficiently reliable at selecting which skills to pull when needed (centrally, this is not always true — Pocock acknowledges skills can be tuned).
  • Coding standards are explicit enough to be written down and machine-checked.
Inferences:
  • Tacit organizational coding norms must be reified into explicit skill files to be enforced; this is itself a senior-engineering activity.
  • The implementer-reviewer asymmetry argues for two-agent architectures (Sandcastle) rather than single-agent loops as the production-grade default.
Manual developer QA is irreplaceable, and is itself the primary site where new Kanban issues are generated. strength: high
Explicit premises:
  • Automated QA strips away taste, producing 'slop' apps.
  • QA is where developer opinions are imposed back onto the codebase.
  • While agents implement, the developer can be QA'ing prior work and adding blocking issues to the Kanban.
Implicit premises:
  • The developer doing QA has aesthetic and product judgement strong enough to materially improve outputs — i.e., this is a senior-developer activity.
  • QA happens at human speed; the parallelization of agent implementation does not relax the QA bottleneck.
Inferences:
  • Throughput is bounded by QA capacity, not by agent capacity.
  • Scaling AI-coding within a team requires growing QA capacity (and taste) — a non-trivial organizational ask.

Questions addressed

  • ✅ How do you stop your agent sliding into the dumb zone?
    Track tokens visibly (status line); clear instead of compact; use sub-agents for exploration; size tasks via vertical slices.
  • ✅ How do you reach alignment with an agent before generating any plan?
    Adversarial dialogue via the grill-me skill, optionally seeded with a meeting transcript from a domain expert.
  • ✅ How do you split a feature into work an agent can do AFK?
    PRD as destination document; vertical-slice Kanban as journey document; each issue tagged AFK or human-in-the-loop with explicit blocking relationships.
  • ✅ How do you parallelize agents safely?
    Independently grabbable issues forming a DAG; per-issue git worktrees in Docker sandboxes (Sandcastle); planner + implementer + reviewer + merger roles.
  • ✅ How do you enforce coding standards on an agent?
    Pull standards in the implementer (skills); push standards into the reviewer (always-loaded).
  • ✅ How do you stop AI from writing bad tests?
    TDD with red-green-refactor as the inner loop; the test instruments behaviour before the implementation exists.
  • ✅ What do you do about a bad codebase?
    Refactor toward deep modules (Ousterhout); use the improve-codebase-architecture skill to find candidate consolidations; design module interfaces yourself and delegate implementation.
  • ✅ Should you keep PRDs and plans in the repo?
    No — dock rot. Close them as issues to retain a visual record without polluting future agent context.
  • ⭕ How do you reconcile parallel agent loops with small-PR review discipline?
    Pocock explicitly admits he doesn't know: 'I don't honestly know what the answer to this yet... I do think it's probably the way things are going.'
  • ⭕ How do you make this work for a team rather than a solo developer?
    Pocock's answer is essentially: the upstream phases (idea → PRD) become team-shared assets argued over with RFCs; downstream the workflow is one or two developers managing the pipeline. He does not address onboarding, role design, review queues, or shared infrastructure.
  • ⭕ How do you conform to existing architecture, API contracts, and security constraints?
    Pocock defers ('I'm going to answer that in a bit') and never fully returns; the answer is implicitly absorbed into 'push reviewer / pull implementer' standards, but no concrete demonstration is given for security or compliance constraints.
  • ✅ How do you handle frontend work?
    Use the AI for throwaway parallel prototypes (three variants in a disposable route), let humans pick, feed the chosen prototype back into the grilling pipeline.
  • ✅ How do you avoid losing your mental model of the codebase under aggressive delegation?
    Design the module interfaces yourself; treat modules as gray boxes whose behaviour-under-interface you know and whose implementation you delegate.
