Building API Products in the AI Era

Janet Wagner · article · 2026-05-09 source ↗ Reader 🖥️

Agent-Native DesignAgentic CommerceAPI Products & SprawlMCP (Model Context Protocol)
APIs must now serve both human developers and AI agent consumers, which requires rethinking API product design across multiple dimensions — granularity, protocol diversity, monetization models, and developer experience — to remain competitive in an AI-driven market.

practical · conference recap / panel synthesis — Synthesizes advice from four industry panelists at Platform Summit 2025 into prescriptive guidance for API product managers and architects. Primarily practitioner-oriented — surveys current thinking rather than advancing original theory.

Key terms

  • API product API Products & Sprawl
    An API built around business capabilities to solve a specific problem, where the business team drives the product decisions and IT provides the technical solution
  • API-as-a-product API Products & Sprawl
    An API that is itself the standalone product being sold, serving as the core value of the business (e.g., Twilio, Stripe)
  • multi-protocol API Products & Sprawl
    Offering the same API product through multiple architectural styles and protocols (REST, gRPC, webhooks, MCP, A2A) so consumers can choose their preferred interaction pattern
  • MCP server MCP (Model Context Protocol)
    A Model Context Protocol server used to orchestrate multiple APIs for AI agent consumers, providing structured context and workflow coordination
  • agent-based payments Agentic Commerce
    Payment protocols that enable AI agents to autonomously execute financial transactions when consuming APIs, exemplified by Google's AP2 and Stripe/OpenAI's ACP
  • outcome-based pricing Agentic Commerce
    API monetization that charges based on the value or outcome delivered rather than the volume of API calls

Arguments

API providers must design for AI agent consumers alongside human developers. strength: medium
Explicit premises:
  • 44% of enterprises assessed or deployed AI agents in 2025 (NVIDIA report).
  • AI agents consume APIs differently — they need structured workflows, contextual understanding, and orchestration across multiple APIs.
  • MCP servers are emerging as the orchestration layer for AI-to-API interaction.
Implicit premises:
  • The 44% figure will continue to grow, making AI agents a dominant consumer class.
  • Providers who ignore this shift will lose competitive ground.
Inferences:
  • Therefore API product design must account for AI agent consumption patterns from the start.
Monolithic APIs should be broken into smaller, targeted APIs. strength: medium
Explicit premises:
  • Large APIs increase cognitive load and are harder to understand.
  • Developers search for simpler services that get them to 'Hello World' faster.
  • Smaller APIs can be tailored to specific business use cases with clear documentation.
Implicit premises:
  • Time-to-first-call is a primary driver of API adoption.
  • The cost of maintaining multiple smaller APIs is justified by higher adoption and retention.
Inferences:
  • Therefore smaller, purpose-built APIs produce better developer acquisition, retention, and product-market fit.
API monetization must evolve beyond per-call pricing for the AI era. strength: medium
Explicit premises:
  • Per-call pricing becomes impractical with millions of AI agent requests.
  • Subscription models don't cover unpredictable AI usage patterns.
  • Customers expect pricing aligned with outcomes, not volume.
Implicit premises:
  • AI agent traffic volumes will be orders of magnitude larger than human-driven API traffic.
  • Providers who don't adapt pricing will either lose money or lose customers.
Inferences:
  • Therefore outcome-based, dynamic, or agent-payment-protocol-based pricing models are needed.

Questions addressed

  • ✅ How should API products be designed differently now that AI agents are major consumers?
    Design for multiple consumer types, offer multi-protocol access, use MCP servers for AI orchestration, break monolithic APIs into targeted services, and evolve monetization to outcome-based models.
  • ✅ What is the difference between an API product and an API-as-a-product?
    An API product is built around business capabilities with business-team ownership; an API-as-a-product is a standalone product where the API is the core value (e.g., Twilio, Stripe).
  • ⭕ How should API pricing change for AI consumers?
    The article identifies the problem (per-call and subscription models are inadequate) and points to emerging solutions (outcome-based pricing, AP2, ACP) but acknowledges no consensus exists yet.
  • ✅ Should internal APIs be treated as products?
    Yes — with developer portals, documentation, and proper onboarding to boost productivity and reduce information loss.
  • ✅ What makes for good API developer experience?
    Solution-oriented documentation, easy onboarding, community building, active social channels, and portals that are both functional and engaging.
