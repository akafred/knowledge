Fighting Tool Sprawl: The Case for AI Tool Registries

Peter Richards · rss · 2026-04-20 Reader 🖥️

Agent Governance (Internal)Agent SecurityAgent Tool ManagementPlatform Engineering
Enterprise AI agent adoption is producing a coordination failure identical to the one package managers solved for shared code: without a centralized, internal tool registry that provides discovery, versioning, certification metadata, and access-control foundations, organizations compound duplication, security exposure, and governance opacity with every new agent deployment.

practical · opinion essay — infrastructure advocacy piece arguing for enterprise-internal AI tool registries, structured as problem-diagnosis-prescription — Written in the register of enterprise platform-engineering thought leadership. Draws on McKinsey research, vendor announcements (Kong, Gravitee, AgilityFeat, Frontegg), and historical analogy to package managers. Advocacy, not reporting.

Key terms

  • tool registry Agent Tool Management
    An enterprise-internal, centralized catalog of AI agent tools with discovery, versioning, certification metadata, and access-control foundations — explicitly not a public package manager.
  • tool sprawl Agent Tool Management
    The uncoordinated, ad hoc proliferation of undocumented, ungoverned AI agent tools across teams — producing duplication, security blind spots, and operational opacity.
  • coordination failure Agent Tool Management
    The predictable outcome of asking teams to solve an infrastructure problem at the application layer — not a failure of discipline but of missing shared infrastructure.
  • Internal Developer Portal (IDP) Platform Engineering
    Existing platform-engineering infrastructure for service discovery and governance; the author positions the tool registry as the agent-era equivalent.
  • deny-by-default Agent Security
    A security posture where agent tool access is blocked unless explicitly allowed — the inverse of the current permissive default that creates scaling attack surfaces.

Narratives

  • The package-manager redemption arc: decades ago, software was fragmented and duplicated; package managers solved this; the agent era is re-creating the same problem; the registry is the same solution, one layer up.
    → Provides historical legitimacy and inevitability framing — positions the registry not as a novel proposal but as a known pattern being applied to a new domain.
  • The governance crisis: adoption is outpacing governance, old lessons are being retaught, and the window for structural intervention is closing before costs become unmanageable.
    → Creates urgency and frames inaction as the risky choice, inverting the usual framing where building new infrastructure is the risk.

Questions addressed

  • ✅ Why do enterprise AI agent deployments produce duplicated, ungoverned tooling?
    It is a coordination failure caused by missing shared infrastructure, not poor discipline.
  • ✅ What is the structural relationship between tool registries and security governance?
    A registry is a necessary precondition — it makes tools discoverable so security review can attach — but not sufficient on its own.
  • ✅ What does a production-grade enterprise tool registry require?
    Discovery, versioning, certification metadata, and access-control foundations.
  • ✅ Why should tool registries be enterprise-internal rather than cross-organizational?
    Cross-org standardization would be premature in a nascent, fast-moving space.
  • ✅ What historical precedent supports the case for centralized tool infrastructure?
    Package managers (npm, PyPI, Maven) solved the identical coordination problem for shared code.

Gaps

  • unstated assumptionThe package-manager analogy elides a critical disanalogy: library dependencies are static, versioned, and deterministic, while agent tool invocations are runtime-dynamic, context-sensitive, and may produce different effects depending on prompt, model, and state. Versioning a tool's code does not version its behavior.
  • omissionNo engagement with the possibility that tool registries themselves become high-value attack targets — a lesson that public package managers have taught repeatedly (typosquatting, registry compromises). The article acknowledges this for public registries but does not examine the risk for internal ones.
  • unstated assumptionThe article assumes that tool duplication is primarily a discovery problem, but in many organizations duplication persists even when shared components are discoverable — teams duplicate because integrating shared infrastructure carries coordination costs, API coupling, and dependency risks that local solutions avoid.
  • omissionNo discussion of the organizational incentive structures that produce tool sprawl: teams are rewarded for shipping agents quickly, not for contributing to shared infrastructure. A registry does not change these incentives.
