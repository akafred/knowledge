Code maintainability plummets in the AI coding era

Bill Doerrfeld · article · 2026-07-07 source ↗ Reader 🖥️

Changeability & OptionalityCode QualityDORA & Engineering MetricsLegacy & Organizational ForgettingMeasuring AI ImpactSoftware Design FundamentalsTechnical Debt
New longitudinal telemetry (GitClear/GitKraken, 623M commits, 2023-2026) shows that since the AI-coding era began, software maintainability has degraded across eight code-quality metrics -- duplication and error-masking are up while code reuse and legacy refactoring have collapsed -- trading short-term generation speed for compounding technical debt.

practical · expository journalism reporting an empirical study — A short trade-press write-up of GitClear/GitKraken telemetry research, framed as a warning to engineering leaders: it reports findings and ends with prescriptions ('leaders should build awareness of AI-coding best practices, review outputs, measure obfuscation'), so it is practical in aim even though its substance is empirical.

Key terms

  • maintainability
    The capacity to keep understanding and changing a codebase over time, operationalised across eight code-quality metrics (duplication, reuse/'Move', legacy refactoring, obfuscation, functional connectivity, churn); the axis of value AI-assisted code systematically underdelivers on.
  • duplication / copy-paste
    Occurrences of five or more consecutive repeated meaningful lines; a DRY violation the author attributes to AI 'creating a new package' for every need rather than reusing existing code.
  • code reuse ('Move' / refactor)
    A metric of how often commits edit existing codebases rather than adding new code; declining reuse means each new implementation discards prior developer progress.
  • legacy refactoring
    GitClear's term for changes that remove or update code last touched more than 12 months ago -- a proxy for whether teams tend existing systems versus only building new.
  • obfuscation / error masking
    The density of rescue/catch blocks, safe-navigation operators, and stubbed methods that squelch unexpected-input signals -- code that silently catches errors without evaluating their underlying cause, added for 'AI expediency' rather than for a real error condition.
  • DRY (do not repeat yourself)
    The principle that every piece of knowledge must have a single, unambiguous, authoritative representation within a system -- the norm AI-generated output is said to violate.
  • code churn
    The volume of recently written code that is soon rewritten or replaced; cited as one of the eight degrading metrics whose recent slowing ('churn only somewhat') hints at a possible plateau.

Arguments

In the AI-coding era, software maintainability is degrading -- codebases are bloating with duplication, suppressing errors, and leaving legacy code to rot -- so short-term AI output speed is being paid for with compounding technical debt. strength: medium
Explicit premises:
  • Duplication is up 81%, 'Move'/reuse down 70%, legacy refactoring down 74% since 2023, obfuscation up 47%, and functional connectivity down 35%, across 623M commits (2023-2026).
  • AI-assisted commits are ~25% of all commits during the same period.
  • Google's DORA finds instability rising with AI usage (7.2% per +25% AI use).
  • Prior GitClear studies (2025, 2026) already validated AI-generated code bloat and technical debt.
Implicit premises:
  • Generation is cheap but comprehension and refactoring are where maintainability actually lives, so an AI that optimises for satisfying the prompt optimises the wrong half of the work.
  • Duplication and never-throw error handling are being traded for speed and for not being flagged as a defect.
  • The eight measured metrics are valid proxies for the underlying construct 'maintainability', and their co-movement reflects one real decline rather than several unrelated shifts.
Inferences:
  • Multiple independent maintainability metrics moving adversely in the same period, corroborated by a separate dataset (DORA), indicates a genuine construct-level decline rather than metric noise.
  • Because the shifts coincide with the rise of AI-assisted commits and match a plausible mechanism (new-package-per-need, never-throw code), AI is the most likely driver of the decline.

Questions addressed

  • ✅ Is AI-assisted coding improving or degrading code maintainability?
    Degrading -- eight maintainability metrics moved adversely across 623M commits in the AI era, led by an 81% duplication increase and a 74% collapse in legacy refactoring.
  • ✅ Why does AI produce this pattern rather than clean, reusable code?
    Because its objective is to satisfy the prompt cheaply -- spinning up a new package per need and writing code that never throws so it won't be labelled a defect -- which shifts the eventual cost onto human maintainers.
  • ⭕ Is the decline slowing, and could the downsides plateau?
    Open: duplication and churn are flattening, which is 'encouraging', but the report's author believes it is 'far from over.'
  • ✅ What should engineering leaders do about it?
    Build awareness of AI-coding best practices, review outputs, measure common obfuscation patterns, and watch specific LLMs or junior cohorts more prone to these faults -- but this prescription is asserted, not evidenced.
