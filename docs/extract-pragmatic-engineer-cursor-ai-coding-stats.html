Cursor's two years of aggregated usage data show that AI-coding output is extremely power-law skewed across developers, that its cost is dominated by input/context-reading tokens (making caching decisive), and that developers are rapidly abandoning manual review of AI-written commits.

theoretical · expository data commentary in a tech newsletter — curated statistics with interpretation — Primarily informative: it reports and interprets Cursor's aggregated two-year usage data (what is the case about AI-coding usage and cost). It carries practical implications for engineering leaders (cost control, model routing, review policy), but the article's own work is reporting numbers and reading meaning into them, not prescribing a method.

Key terms

  • p50 / p90 / p99 (percentile usage tiers)
    Percentile buckets of Cursor developers ranked by lines of code generated per week — median (p50 ~700), 90th (~9,000), 99th (~30-40K).
  • input tokens vs output tokens
    Tokens the agent consumes reading the existing codebase and docs (input) versus tokens it emits as generated code (output); the article's central cost distinction.
  • 10:1 read-to-write ratio
    The observation that ~90% of usage is reading context, echoing Uncle Bob's 2008 claim that developers spend >10x as long reading code as writing it.
  • context caching (cache read / cache write)
    Reusing previously processed context so it need not be re-sent as fresh input tokens; when counted, cache reads are ~90% of token usage and output falls to 0.6%.
  • cost per agent request
    The billed cost of one agent turn/request for a given model — the metric on which Opus 4.7 is ~10x Composer 2.5 and ~2x GPT-5.5.
  • cost-per-line-accepted (a.k.a. cost-per-line-added)
    An outcome-adjusted cost metric dividing spend by lines actually accepted, which rewards expensive-but-accurate models and shrinks the apparent cost gap.
  • commits without manual review
    AI agents creating commits that the developer no longer personally checks before they land — the share that rose from ~10% to ~40% in a month.
  • tokenmaxxing Token Throughput
    Raised as a possible explanation for power users' extreme volume — spending tokens to top usage leaderboards rather than to ship value.

  • explicit90% of Cursor's token usage is input tokens spent reading the existing codebase and documentation, not output tokens generating code — a roughly 10:1 read-to-write ratio that mirrors the classic developer read/write ratio.conf: high
  • explicitEven though output tokens are priced about 5x higher than input tokens (e.g. Opus 4.7 at $5 per 1M input vs $25 per 1M output), input tokens still account for close to 70% of the cost of AI coding agents because they so dominate usage.conf: high
  • explicitContext caching cuts token cost roughly 10x: once cache reads and writes are counted, output is only 0.6% of token usage, with cache reads at about 90%, cache writes 2.5%, and raw input 7%.conf: high

Arguments

Input/context tokens, not generated output, are the dominant driver of AI-coding cost, so caching is the decisive efficiency lever. strength: high
Explicit premises:
  • 90% of Cursor's token usage is input tokens for reading the codebase.
  • Input tokens account for ~70% of AI-coding-agent cost despite output being priced ~5x higher per token.
  • With caching counted, output is only 0.6% of tokens and cache reads are ~90%; without caching, cost would be 10x higher.
Implicit premises:
  • Cursor's aggregated usage is representative of AI-coding workloads generally, not just its own product.
  • The measured token/cost mix reflects genuine efficiency rather than an artifact of Cursor's specific pricing arrangements.
Inferences:
  • If reading context consumes most tokens and most cost, then reducing the re-processing of that context (caching/reuse) is where the largest savings live.
Developers are rapidly increasing their trust in AI-written code, delegating even the review step.
Explicit premises:
  • The share of devs letting AI commit without manual review rose from ~10% to ~40% in a single month.
  • The jump correlates with the release of more capable models (Opus 4.7, GPT-5.5) and sentiment that hand-writing code is dying.
Implicit premises:
  • Choosing not to review is evidence of justified trust in model output, rather than approval fatigue, volume overload, or complacency.
  • The absence of manual review is not (yet) producing enough visible failures to reverse the behavior.
Inferences:
  • Because the behavior change tracks model capability improvements, it is read as a capability-driven shift rather than a fad.

Narratives

  • The top 1% of Cursor users generate 30-40K lines per week, the equivalent of about 45 median developers.
    → Dramatizes the power-law skew of AI-coding output and sets up the author's skeptical question of whether that volume is value or noise.
  • Robert C. Martin's 2008 Clean Code observation that the read-to-write time ratio is 'well over 10 to 1.'
    → Historical analogy that makes the surprising 90%-input-token finding feel intuitive — agents, like humans, spend most effort reading.
  • Gemini and Grok were omitted from Cursor's model comparisons; Cursor told the author it was because those models see very little usage on the platform.
    → Addresses a conspicuous omission and reinforces that the data is Cursor-usage-specific, not a neutral cross-model benchmark.
  • The 10%-to-40% jump in unreviewed commits coincided with new model releases and with many devs concluding that 'writing code by hand is dying.'
    → Ties the headline behavioral shift to a specific cause (a model-capability inflection) and to a broader cultural sentiment.

Questions addressed

  • ✅ What does Cursor's aggregated usage data reveal about how AI coding is actually used and paid for?
    Usage is power-law skewed across developers, cost is dominated by input/context tokens, caching cuts cost ~10x, and unreviewed AI commits are rising fast.
  • ✅ Why do input tokens dominate AI-coding cost even though output tokens are priced higher?
    Agents spend most tokens reading the existing codebase and docs, so sheer input volume outweighs the higher per-token output price, pushing input to ~70% of cost.
  • ✅ Are the most capable, most expensive models worth their price?
    Partially: on cost-per-agent-request they look ~10x worse than cheap models, but on outcome-adjusted cost-per-line-accepted the gap narrows sharply, so 'expensive' is metric-dependent.
  • ⭕ Do the extreme power users actually deliver proportional business value (or just more code and bugs)?
    The author explicitly raises this but the report only measures line volume, not value, bug rate, or maintainability, so it goes unanswered.
  • ⭕ Is the sharp rise in unreviewed AI commits justified trust or accumulating risk?
    The report documents the behavior and its correlation with model releases but provides no downstream defect, incident, or quality data to judge whether skipping review is safe.

Gaps

  • incompleteThe report measures lines-of-code volume per developer but never links that volume to delivered business value, defect rates, or maintainability — leaving the author's own question ('are they adding a lot of business value?') unanswered.
    Read next: Read outcome-anchored AI-impact studies that trace code activity through to shipped value (five-studies-ai-software-engineering on delivery-hierarchy attenuation; DORA/outcome metrics) to test whether high line volume corresponds to value.
  • incompleteAcceptance-without-review is reported as an adoption signal, but the report does not examine downstream defect, incident, or rework rates from unreviewed AI commits, so it cannot distinguish justified trust from accumulating cognitive and technical debt.
    Read next: Pair with studies of cognitive debt and verification bottlenecks (five-studies-ai-software-engineering; guardrails literature) to model the consequences of removing the human review checkpoint.
  • incompleteAll figures come from Cursor, which has a commercial incentive for its Composer model to appear cheapest and for caching to look like its moat; there is no independent verification and the cost comparison is a single-agent-request figure, not a token-to-token one.
    Read next: Seek independent, cross-vendor AI-usage and cost studies to corroborate or challenge the caching, cost-share, and acceptance-rate claims before treating them as general truths.
