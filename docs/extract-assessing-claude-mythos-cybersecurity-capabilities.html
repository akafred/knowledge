Assessing Claude Mythos Preview's cybersecurity capabilities

Nicholas Carlini, Newton Cheng, Keane Lucas, Michael Moore, Milad Nasr, et al. (Anthropic) · article · 2026-04-08 source ↗ Reader 🖥️

AI Alignment & SafetyAI-Driven Vulnerability DiscoveryMythosProject Glasswing
Claude Mythos Preview represents a qualitative leap in autonomous vulnerability discovery and exploitation — finding and exploiting zero-days in every major OS and browser, including bugs that survived decades of human auditing — and the security industry must begin urgent coordinated defense now, during the transitional period before these capabilities become broadly available, because the same general improvements in reasoning and autonomy that make models better at fixing vulnerabilities make them equally better at exploiting them.

practical · mixed — Practical: argues for urgent coordinated defensive action. Mixed form: part technical report (vulnerability case studies with code-level detail), part capability assessment (benchmark comparisons), part policy argument (responsible disclosure, Project Glasswing). Unusually rigorous for a company blog post — includes SHA-3 commitments for undisclosed vulnerabilities, specific cost figures, and honest limitations. The tone is carefully calibrated: alarming enough to drive action, measured enough to maintain credibility.

Key terms

  • qualitative leap / different league AI-Driven Vulnerability Discovery
    Not an incremental improvement but a discontinuous jump in capability. Opus 4.6: near-0% exploit success rate. Mythos: 181/several hundred. The authors argue this is not the same thing done better but a fundamentally new capability category.
  • emerged / emergent
    The cybersecurity capabilities were not explicitly trained — they are a downstream consequence of general improvements in code understanding, reasoning, and autonomy. This means future models will also have these capabilities, regardless of whether they are trained on security data.
  • transitional period AI Alignment & Safety
    The window between when AI models gain these capabilities and when the security ecosystem reaches a new equilibrium. During this period, attackers may have the advantage because defensive tooling and practices haven't adapted. The restricted release of Mythos is specifically designed to shorten this window.
  • Project Glasswing Project Glasswing
    Anthropic's program to give critical infrastructure partners and open-source maintainers early access to Mythos for defensive vulnerability scanning, before the model is broadly available.
  • vulnerability chaining AI-Driven Vulnerability Discovery
    Combining multiple individually-limited vulnerabilities into a complete exploit. A read vulnerability alone is limited; a write vulnerability alone is blind. Chain them together (read to bypass ASLR, then write to escalate privileges) and you get root. Mythos can chain 2-4 vulnerabilities autonomously.

Arguments

The security industry must begin urgent coordinated defensive action now, during the transitional period before these capabilities become broadly available strength: high
Explicit premises:
  • Mythos can find and exploit zero-days in every major OS and browser
  • The capability jump is discontinuous (181x vs. near-0%)
  • These capabilities emerged from general improvements, meaning future models will also have them
  • Non-experts can leverage these capabilities with minimal scaffolding
  • Thousands of critical vulnerabilities exist undiscovered in production software
Implicit premises:
  • Other labs will produce models with similar capabilities within months — the restricted release only buys time, not permanent advantage
  • Attackers with access to open-weight models (or future equivalents) will not observe responsible disclosure norms
  • The current security ecosystem's human-speed processes (90+45 day disclosure, manual code review, penetration testing) cannot keep pace with model-speed vulnerability discovery
Inferences:
  • The security industry needs to fundamentally restructure around AI-speed vulnerability discovery and patching — not just use AI as a tool within existing processes
  • Open-source projects face an existential capacity problem: if AI can find thousands of critical bugs, but maintainers can barely keep up with current bug reports, the discovery rate may overwhelm the fix rate
  • The window for defenders to get ahead is measured in months, not years — the time between Mythos and broadly available models with similar capabilities is shrinking
  • National security implications are immediate for countries running critical infrastructure on open-source software (including Norway's public sector)
Language models discover qualitatively different vulnerabilities than existing automated tools (fuzzers, static analyzers) strength: high
Explicit premises:
  • The FFmpeg H.264 bug survived 16 years of fuzzing because it requires understanding sentinel initialization semantics
  • The OpenBSD SACK bug requires understanding signed integer overflow interaction with protocol state — a reasoning chain, not a crash path
  • The VMM bug requires understanding that memory-safe languages have unsafe escape hatches
Implicit premises:
  • Fuzzers explore input space; language models explore semantic/logical space — they are complementary, not competitive
  • The remaining bugs in well-audited codebases are precisely the ones that require semantic understanding to find — the easy crashes have already been found by fuzzers
Inferences:
  • The security model that says 'if it passes fuzzing, it's probably safe' is now invalidated — a new threat class has emerged
  • Security budgets and practices need to add 'LLM-driven vulnerability assessment' as a category alongside fuzzing, penetration testing, and code review

Narratives

  • The OpenBSD SACK vulnerability: a 27-year-old bug requiring two interacting flaws (missing range check + signed integer overflow at 2^31 boundary) that together create an 'impossible' condition — the SACK start is simultaneously below and above the same point. Crashed a security-focused OS.
    → Demonstrates that Mythos finds bugs requiring multi-step logical reasoning — not just pattern matching but understanding how signed integer arithmetic interacts with protocol state. The OpenBSD angle (known for security) amplifies the impact.
  • The FFmpeg H.264 sentinel collision: -1 used as 'no slice' sentinel stored as 16-bit 65535; an attacker creates exactly 65536 slices and the real slice number collides with the sentinel. A 2003 design decision became exploitable after a 2010 refactor.
    → Shows that Mythos finds bugs across time — understanding how code written years apart can interact to create vulnerabilities. No fuzzer would generate exactly 65536 slices because the trigger requires semantic understanding, not random mutation.
  • The FreeBSD NFS exploit: Mythos found the stack overflow, noticed the stack canary was absent (int32_t buffer, not char array), discovered the handle-guessing trick via NFSv4 EXCHANGE_ID, built a 20-gadget ROP chain, and split it across 6 RPC packets to fit the 200-byte limit — all autonomously.
    → The most complete demonstration of end-to-end autonomous exploitation. Every step required a different type of reasoning: compiler behavior, protocol interaction, exploit engineering, constraint satisfaction. The detail about splitting across 6 packets shows genuine problem-solving creativity.
  • Non-experts at Anthropic asking Mythos to find RCE overnight and waking up to a working exploit
    → Makes the democratization threat concrete — this isn't about nation-state actors with security teams, it's about anyone with access to the model. The 'overnight' framing emphasizes autonomy.

Questions addressed

  • ✅ Has AI reached a capability level that genuinely changes the cybersecurity landscape?
    Yes — the jump from Opus 4.6 (near-0% exploit success) to Mythos (181x) is discontinuous, and the capabilities emerged from general improvements, meaning they will continue to improve.
  • ✅ Can AI find bugs that humans and fuzzers have missed for decades?
    Yes — demonstrated with specific examples (27-year OpenBSD, 16-year FFmpeg, 17-year FreeBSD). The model finds semantically complex bugs that require reasoning about code interactions, not just crash-inducing inputs.
  • ⭕ Will these capabilities ultimately benefit defenders or attackers?
    Authors believe defenders in the long run (analogous to fuzzers), but acknowledge the transitional period favors attackers. The outcome depends on how carefully frontier labs manage release.
  • ⭕ Can the open-source maintainer ecosystem absorb AI-scale vulnerability discovery?
  • ⭕ What happens when multiple labs have models with these capabilities and not all observe responsible disclosure norms?

Gaps

  • incompleteThe paper does not address what happens when open-source maintainers receive thousands of vulnerability reports. Anthropic acknowledges human validation is already a bottleneck and may need to be relaxed. But the deeper question is whether the volunteer-maintainer model of open-source can survive AI-scale vulnerability discovery at all. The economic model (free labor maintaining critical infrastructure) was already strained; AI-driven bug discovery may break it entirely.
    Read next: Nadia Eghbal's 'Working in Public'; the xkcd 'dependency' comic made real; EU Cyber Resilience Act implications for maintainer liability; Willison's piece on the vulnerability tsunami
  • incompleteThe paper argues the transitional period will end with defenders ahead, by analogy to fuzzers. But the analogy may not hold: fuzzers are symmetric (both sides can fuzz), but AI vulnerability discovery has asymmetric economics. An attacker needs ONE working exploit; a defender must patch ALL vulnerabilities. If AI finds 10,000 bugs, the attacker wins by exploiting any one before it's patched.
    Read next: Asymmetric warfare literature; the 'defender's dilemma' in cybersecurity; Bruce Schneier on the economics of security
    closed byPredict, Don't Enumerate — Addresses the defender's-dilemma asymmetry: defenders need not patch everything; exploitability prediction plus local environmental context is a structural defender advantage that survives AI-scale discovery.
  • incompleteThe paper doesn't discuss the implications for memory-safe language migration strategies. If the remaining vulnerability surface is increasingly in unsafe blocks (as the VMM case shows), does the industry need to rethink how unsafe code is audited, isolated, and minimized? The 'just rewrite in Rust' narrative needs updating given that Rust's unsafe blocks are now a primary target.
    Read next: Rust safety literature; formal verification of unsafe blocks; the 'unsafe Rust' audit ecosystem
  • incompleteThe cost figures ($20,000 for OpenBSD, $10,000 for FFmpeg) imply that nation-state actors — or well-funded criminal organizations — could run comprehensive vulnerability scans against all critical infrastructure software for less than the cost of a single human security researcher's annual salary. The geopolitical implications of this cost structure are not discussed.
    Read next: Norwegian intelligence assessments (PST, NSM) on cyber threats; the economics of cyber offense vs. defense at national scale
