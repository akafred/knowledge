Agentic Code Review

Addy Osmani (addyosmani.com) · rss · 2026-07-06 source ↗ Reader 🖥️

Agent SecurityCode Was Never the BottleneckCognitive DebtCognitive SurrenderHuman-in-the-LoopIntent DebtMeasuring AI ImpactOrchestration Tax / Human Review BottleneckPull Requests & Async Code Review
Because agents made writing code nearly free while understanding it stayed exactly as costly as ever, code review has become the binding constraint on shipping software — and since review must now reconstruct an intent that never existed in any human mind, the right amount of human review is not a fixed rule but a dial set by each change's blast radius.

practical · argumentative practitioner essay, data-driven — A prescriptive essay in Osmani's ongoing series on agentic engineering (comprehension debt, cognitive surrender, orchestration tax, intent debt, the verification bottleneck). It is practical — it diagnoses a problem and prescribes conduct (tier by risk, require evidence, run heterogeneous AI reviewers, keep a human on the merge) — but grounds every prescription in a theoretical account of WHY review became the bottleneck and in named 2026 benchmark datasets.

Key terms

  • agentic code review
    Code review in the regime where an agent, not a human, authored most of the code — so the reviewer is often 'the first human being to ever lay eyes on this code' and review must manufacture an intent that never existed rather than check an author's reasoning.
  • review bottleneck / the bottleneck moved to verification
    The binding serial constraint on shipping is no longer writing code but a trusted human being confident a change is correct; machine-speed output was poured into a system built for human-speed reading and review is where that bill comes due.
  • blast radius Agent Security
    What happens when a change breaks — nothing, versus angry users, money and PII on the line — the primary dial that (with code lifespan and how many people must understand it) sets how much review a change actually needs.
  • reconstructing intent
    The new, slower work of review: because no human ever held the reasoning behind agent-written code, the reviewer has to rebuild a rationale that never existed in any mind — a fundamentally different task from checking an author's thinking, which is why review now takes 441% longer.
  • borrowed confidence Cognitive Surrender
    When an AI review says 'looks good' in a calm, confident voice and no human is anywhere in the loop, the system's certainty silently becomes yours though nobody actually understood anything — a closed loop of same-data models can be both very sure and very wrong.
  • human on the loop (vs. in the loop)
    The human stops reading every diff and instead samples, spot-checks and audits the system — spending scarce attention where being wrong would actually hurt (accountability, whether it is the right change, high-blast-radius gates) rather than on every PR.
  • heterogeneity / cross-family review
    Running reviewers that are built differently so they surface non-overlapping classes of bug; four copies of one model is a single reviewer with a bigger invoice, whereas four genuinely different reviewers find bugs no single one could — demonstrated by four tools that never once flagged the same line.
  • the senior-engineer tax / comprehension debt
    Review time up by triple digits falls hardest on the senior people you can least afford to bottleneck, and it is invisible to any metric that only counts merged PRs; unreviewed, un-understood changes accumulate as comprehension debt that becomes someone's on-call incident.

Arguments

Review, not writing, is now the binding constraint on shipping software. strength: high
Explicit premises:
  • Writing code has collapsed toward zero cost while human reading speed is unchanged.
  • Code review only ever kept pace by the accident that reading was faster than writing.
  • Four independent 2026 datasets show ~4x more code for ~12% more value, with quality and reviewability down.
Implicit premises:
  • Understanding a change genuinely requires human-speed reading and cannot itself be accelerated by the same agents that wrote the code.
  • The value of a change is gated on a human understanding it, not merely on it passing tests.
  • AI's added output is overwhelmingly review burden rather than delivered value (this is what the 4x/12x gap is taken to mean).
Inferences:
  • If the cheap, fast half of a coupled pair collapses in cost, the remaining human-speed half becomes the throughput ceiling.
  • Therefore the pipeline now waits on review, and the bottleneck moved to verification.
Reviewing agent code is a fundamentally different and slower task than reviewing human code, because it must reconstruct an intent that never existed. strength: high
Explicit premises:
  • With human code the author's reasoning existed and review checked it.
  • With agent code no human ever formed that reasoning.
  • A model can only fabricate plausible intent from the code, not supply the actual intent.
Implicit premises:
  • Review's essential purpose was always to check reasoning — so removing the reasoning changes the nature of the task (this downplays review's other jobs: catching typos, distributing knowledge, enforcing style).
  • Intent cannot be reliably recovered from the code artifact itself.
  • A confident but fabricated rationale is worse than an admission of not-knowing, because at merge time it is indistinguishable from real rationale.
Inferences:
  • Manufacturing intent from scratch is strictly harder than verifying intent that is presented, so review time rises (the 441% figure is read as evidence of exactly this).
A human must remain accountable on the loop; the review loop must not be allowed to close completely on anything that can hurt someone. strength: high
Explicit premises:
  • Writer, reviewer and judge models share training data and therefore share blind spots.
  • AI review is a sensor, not a verdict; a model cannot be paged or held responsible.
  • The judgment of whether a change is the right one to build is distinct from whether the code is correct.
Implicit premises:
  • Whether a change is the right change is not checkable against the code, so it cannot be delegated to any code-reviewing model.
  • Heterogeneous reviewers reduce correlated error (backed by the finding that four tools never flagged the same line), whereas a homogeneous loop amplifies it.
  • The requirement nobody thought to write down will never be flagged by a model that reviews only the code that exists.
Inferences:
  • A fully closed model loop maximizes shared-blind-spot risk while eliminating the only party who can catch unspecified behavior and be accountable — so retained human review must scale with blast radius.
There is no one-size answer; review depth must be set by blast radius, not by guilt or a universal rule. strength: high
Explicit premises:
  • Three variables (blast radius, code lifespan, number of people who must understand it) determine what 'good review' means.
  • A solo builder with no users and a team on a decade-old payments system share almost no constraints.
  • Bolting an enterprise pipeline onto a prototype adds friction for no benefit; running 'tests pass, ship it' on payments builds an incident generator.
Implicit premises:
  • A developer can correctly self-locate on the blast-radius spectrum and honestly assess their own position in real time — yet the doc itself concedes the dangerous-middle crossing 'is rarely noticed at the time'.
  • Deferring review behind real tests is safe, whereas skipping verification is not — 'no users is permission to defer review, not to skip verification'.
Inferences:
  • Since the purpose of review changes with position, the rules must change with it; matching review effort to cost-of-being-wrong is the general form.

Narratives

  • Code review's origin as a 'happy accident': a senior engineer could read faster than a junior could write, so review kept pace without anyone designing it to, and the team absorbed system knowledge as a side effect — none of it deliberate.
    → Sets up the entire argument by naming the precise precondition that agents just broke, so the reader feels the loss rather than being told about it.
  • A developer describing an agent's PR as making them 'the first human being to ever lay eyes on this code', drawn from the 2026 paper 'AI Slop and the Software Commons', which analyzed 1,154 posts across 15 Reddit and Hacker News threads.
    → Crystallizes the intent-reconstruction thesis in one sentence — review now happens before any human has ever understood the change.
  • Kun Chen, an ex-Meta L8 engineer now shipping ~40 PRs a day as a solo builder who has largely stopped reviewing code — running 20-30 agents in parallel, writing detailed up-front plans, and gating merges with an automated 'No Mistakes' review.
    → A concrete existence proof that 'stop reviewing' can be rational at one extreme of the spectrum — but only because intent was written up-front and a net exists; the conditions most readers do not have. Copy it onto a team and you reproduce the Faros numbers.
  • Osmani's own workflow: pointing Claude Code or Codex at a batch of incoming PRs (including on popular open-source projects) for a risk-sorted first pass — low-risk gets a few minutes of confirmation, high-risk gets careful human time; the merge decision stays his.
    → Demonstrates 'human on the loop' in practice and embodies the framing irony — the same tools that cause the flood are what make the queue survivable.

Questions addressed

  • ✅ Why has code review become the most expensive part of software work?
    Writing collapsed toward zero while human reading and understanding did not, breaking the relative-speed accident that made review keep pace; the bottleneck moved to verification and review is where the bill comes due.
  • ✅ Is there one right way to do AI code review?
    No — how much and what kind of review a change needs is set by its blast radius, lifespan, and how many people must understand it; most bad advice is one spectrum-position prescribing to another.
  • ✅ Can agents just review other agents' code?
    Only half — a differently-primed model catches real bugs, but it cannot reconstruct human intent, decide whether a change is the right one, catch unspecified behavior, or be held accountable; treat AI review as a sensor, not a verdict.
  • ✅ Should we deliberately let AI do more of the reviewing?
    The machine already reviews more than humans do; make it deliberate, keep a human 'on the loop' owning the load-bearing paths, scale retained human review by blast radius, and never let the loop close completely on anything that can hurt someone.

Gaps

  • incompleteThe vendor-interest and measurement caveats are flagged but not reconciled. Osmani notes CodeRabbit and Faros sell into this market and that some of GitClear's 12% is selection bias, yet never asks whether the effect sizes survive de-biasing, nor whether 'four independent datasets that agree' truly measure one construct — churn, incidents-to-PR, defect rate, review duration and issue counts are different metrics, and their agreement is asserted rather than shown to be about the same underlying thing.
    Read next: Independent, non-vendor telemetry on AI code quality; how churn / defect-rate / review-duration metrics are operationally defined across Faros, CodeRabbit and GitClear and whether they are comparable; selection-bias correction in AI-productivity studies.
  • incompleteReview's knowledge-distribution function is named as real ('the team absorbed how the system fit together as a side effect of reading each other's diffs') but never re-examined. Once humans stop reading most diffs, the question of whether AI reviewers, evidence-gates or written intent can carry that team-comprehension role — or whether it simply erodes — is left open, even though it is a distinct job of review from bug-catching and intent-reconstruction.
    Read next: Empirical code-review research (Greg Wilson) on what review actually accomplishes; distributed-cognition literature (Hutchins, 'Cognition in the Wild') on knowledge spread as a side effect of shared reading.
  • incompleteThe whole prescription assumes a team can correctly self-locate on the blast-radius spectrum in real time, but the doc itself concedes the dangerous-middle crossing (from no-users to real-users) 'is rarely noticed at the time'. How a team detects it has crossed — before the postmortem, not after — is exactly the load-bearing detection problem the essay leaves unaddressed.
    Read next: Leading indicators that a project's blast radius has changed (first real users, first PII, first money at risk); instrumentation and tripwires for tightening review before an incident forces it.
  • incomplete'Human on the loop — sampling, spot-checking, auditing' is offered as the destination but never operationalized. What sampling rate is enough, what selects which PRs get human eyes, and how you know the sample is representative are all unspecified — especially since the 'behavior nobody specified' gap is by definition invisible to the reviewing model and so cannot be sampled toward.
    Read next: Statistical acceptance sampling applied to code review; risk-based audit methodology; how to surface unspecified-requirement gaps that no code-reading reviewer (human or model) can see in the diff.
  • incompleteThe economics of the recommended stack are never priced. Running two-plus heterogeneous AI reviewers on everything, plus a human triage layer and strict deterministic gates, has a token and attention cost that is never set against the modest ~12% real productivity gain the same data reports — leaving open whether the review apparatus consumes the very productivity it is meant to protect.
    Read next: Cost/benefit accounting for multi-reviewer setups (token spend per PR, false-positive triage time); whether heterogeneous review is cost-effective only when invoked sparingly at high-signal moments rather than on everything.
