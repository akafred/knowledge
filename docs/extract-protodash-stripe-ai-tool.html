Key terms

  • Protodash Non-Engineers Building
    Stripe's internal AI prototyping tool that lets designers and PMs generate clickable, production-quality prototypes from the company's own design system (Sail) using AI code generation.
  • blurple slop AI Slop
    The low-fidelity, uncanny-valley output produced by generic AI design tools (like v0) that don't know a company's design system — wrong nav, wrong fonts, mismatched components in generic Tailwind.
  • Sail
    Stripe's internal design system, exposed to AI via an MCP server so that code generation uses real components rather than imagining approximations.
  • demos not memos Non-Engineers Building
    The cultural shift Owen advocates: instead of writing documents proposing product ideas, build working prototypes that teams can interact with directly.
  • Cursor rules Agent Context Files (CLAUDE.md / AGENTS.md)
    Instruction files that teach the AI code editor how to use a specific project — which components to use, when to check the MCP server, how to handle the design system. The foundational layer of Protodash.
  • MCP server (for Sail) MCP (Model Context Protocol)
    A Model Context Protocol server that provides AI tools with structured access to Stripe's design system components, enabling high-fidelity code generation.

Arguments

Companies with strong design systems should build internal AI prototyping tools rather than using generic ones.
Explicit premises:
  • Generic AI tools don't know your design system and produce 'blurple slop'
  • Stripe has a high quality bar and a well-defined design system (Sail)
  • Embedding the design system via MCP + Cursor rules produces prototypes that look correct
Implicit premises:
  • The quality bar for internal prototypes must match the company's shipping standard to be useful in design reviews
  • The cost of building and maintaining an internal tool is now lower than the accumulated cost of cleaning up generic AI output
Lowering the barrier to prototyping transforms organizational decision-making from document-driven to demo-driven.
Explicit premises:
  • Protodash lowered the barrier so designers only need 'npm run dev'
  • PMs became surprise power users
  • Conversations shifted from staffing questions to design quality discussions
Implicit premises:
  • The previous cost of prototyping was high enough that many ideas were never prototyped
  • People make better decisions when they can interact with a working thing rather than read about it
Code-based prototyping is superior to static design prototyping for exploring edge cases.
Explicit premises:
  • Dashboards have many states: zero, low-volume, high-volume, different business models
  • Getting all those states into Figma was 'a mess'
  • In code you can 'just toggle between them'
Implicit premises:
  • Edge case exploration is a critical part of good design that is often skipped due to tooling friction
  • The value of code-based prototypes scales with product complexity

Narratives

  • Owen observed designers in design reviews struggling with v0-generated prototypes that looked wrong (wrong nav, wrong fonts, mismatched components), leading him to build Protodash to solve the 'blurple slop' problem.
    → Origin narrative — personal frustration driving a tool that solves a real, observed problem
  • Protodash evolved from a bundle of Cursor rules and React components into a full browser-based platform, with the explicit goal of requiring designers to know only 'npm run dev' and nothing else.
    → Technical evolution narrative — shows pragmatic, iterative approach to internal tooling
  • Owen's initial nervousness about PMs using Protodash ('Oh my goodness, PMs designing') transformed into enthusiasm as PM adoption improved cross-functional collaboration rather than undermining designers.
    → Reversal narrative — challenges the assumption that democratizing design tools threatens designers

Questions addressed

  • ✅ Why do generic AI design tools fail at companies with mature design systems?
    They lack knowledge of company-specific components, producing 'blurple slop' — uncanny-valley output that fails quality bars.
  • ✅ How can a company embed its design system into AI-assisted prototyping?
    Bundle design system components, create an MCP server for the design system, and write extensive Cursor rules that teach the AI when and how to use the system.
  • ✅ What happens when non-designers (PMs) get access to high-fidelity prototyping tools?
    Counter to initial fears, it improves collaboration — PMs build correctly-styled explorations, communicate better with designers, and advocate for design resources with working demos.
  • ✅ What are effective heuristics for working with AI coding tools?
    Be specific in initial prompts; abandon deteriorating sessions early (slash-clear and restart) rather than falling into sunk-cost debugging.
