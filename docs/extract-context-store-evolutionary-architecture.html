Comprehension at AI Speed: Building a Context Store for Evolutionary Architecture

Stella Berhe, Stephan Bragner, Vikram Maran, Anand Jayaraman · rss · 2026-07-14 source ↗ Reader 🖥️

Code Was Never the BottleneckCognitive DebtContext StoreEvolutionary Architecture & Fitness FunctionsIntent DebtSpec-Driven DevelopmentTest-Driven DevelopmentVerifiability (as the Quality Bar)
Because AI has decoupled writing code from understanding it, engineering organizations must run spec-anchored specification-driven development, test-driven development, and architectural fitness functions together as one verification system so that their combined artifacts form a versioned, queryable "context store" — making comprehension a lifecycle property of evolutionary architecture that both human reviewers and AI agents can draw on, rather than a pre-merge concern that AI-generated speed otherwise outruns.

practical · mixed — Argumentative core (data + framework building toward a prescription) wrapped around two narrative anchors (the Copilot/METR productivity reversal, the Amazon outage) and closed with concrete first-step actions per discipline — the piece ultimately argues for what teams should do, not merely what is true.

Key terms

  • Context store Context Store
    A deterministic, versioned record of design intent, behavior, and architectural conformance, produced as a byproduct of spec-anchored SDD, TDD, and fitness functions run together, persisted in the repository and queried by both humans and AI agents across the system's whole lifecycle.
  • Context gap
    The decoupling, unique to AI-assisted development, between code shipping at machine speed and the understanding (intent, behavior, architectural reasoning) that used to be produced automatically as a side effect of a human typing the code.
  • Comprehension (as an architectural characteristic)
    A property of a system, on par with availability, elasticity, resilience, and security, that a team must deliberately design for and protect rather than assume — because AI has made it possible to lose.
  • Spec-anchored SDD
    Specification-driven development where the spec is a living, machine-readable artifact authored and reviewed like code, amended whenever reality diverges from intent — as opposed to a one-time document generated to brief an agent and never revisited.
  • The 80/20 gap
    The observation that AI dramatically accelerates the first 80% of a feature (scaffolding, compiling, generated tests, working demo) while the last 20% — integration, edge cases, undocumented constraints, institutional memory — remains slow and is where architecture actually lives.

Arguments

Organizations building software with AI need a context store — a persisted, queryable record of intent, behavior, and conformance — rather than simply accepting faster raw code production as the win. strength: medium
Explicit premises:
  • AI decouples code writing from the understanding-building that used to happen automatically alongside it.
  • Two divergent studies (Copilot +55.8% in a controlled task, METR -19% in a real-codebase RCT) show that raw completion-speed measurement is unreliable and is not the thing that actually matters.
  • The 80/20 shape of AI-assisted work means the current pace hides architectural cost until an incident, departure, or re-platforming decision exposes it.
  • Amazon's outage and DORA's 2025 instability finding are both symptoms of the same underlying gap, showing up at the team altitude and the leadership altitude respectively.
Implicit premises:
  • Slowing down AI-generated code production is not an acceptable fix, because that speed is a real productivity win and "not the problem worth solving" — so any solution must add a parallel comprehension mechanism rather than throttle AI.
  • A gap that stays invisible until a production incident, key departure, or re-platforming event exposes it is nonetheless costing the organization continuously in the interim, even though nothing in the article measures that ongoing, unexposed cost directly.
Inferences:
  • If comprehension really is architecture's newly binding constraint, the tools evolutionary architecture already built for other characteristics (fitness functions, executable guardrails) should be reusable for comprehension too — which is exactly the move the article makes next, without the article explicitly flagging this as an inference from the two-altitudes evidence to the toolkit-reuse strategy.
Running spec-anchored SDD, TDD, and architectural fitness functions together — rather than practicing any one in isolation — is what produces a queryable context store capable of closing the AI comprehension gap. strength: medium
Explicit premises:
  • Spec-anchored SDD establishes the intent layer, with the spec authored and reviewed like code and amended when reality diverges from it.
  • TDD establishes the behavior layer, with failing tests written first serving as both the agent's brief and the verification of its output.
  • Architectural fitness functions establish the structural layer, as automated CI-blocking checks of architectural conformance.
  • None of the three disciplines is new individually — each already has an established literature and tooling (Beck's TDD, Ford/Parsons/Kua/Sadalage's evolutionary-architecture fitness functions, domain-driven design's ubiquitous-language tradition for specs).
Implicit premises:
  • The three disciplines' artifacts, once persisted and versioned together in the repository, compose into something qualitatively more than their sum — i.e. it is the integration itself, not any single discipline done well, that turns three separate practices into a "store."
  • A human reviewer's PR-level question about why the system behaves a certain way can always be traced to one of exactly these three artifact types (spec, test, fitness function), implying the three layers are jointly exhaustive of the "why" a reviewer or agent would ever need.
Inferences:
  • If the three layers really are jointly exhaustive of comprehension's "why," then a codebase missing any one of the three disciplines has a structural blind spot in precisely the dimension that discipline alone would have covered — e.g. no fitness functions means structural conformance has no persisted record even where intent and behavior are well captured.
Architectural conformance should be enforced by automated, CI-blocking fitness functions rather than left to human reviewers, freeing reviewer time to interrogate intent rather than check structure. strength: medium
Explicit premises:
  • Senior-reviewer capacity in AI-accelerated codebases is finite.
  • Cognitive overload slows teams down and increases failure rates.
  • A synchronous architecture review board, or a single senior engineer who catches everything, is slow, burns that person out, and lets architectural drift surface only when production exposes it.
Implicit premises:
  • Architectural conformance is the kind of property that can be fully captured by automatable, executable checks — implying there is no residual architectural judgment call that only a human reviewer, and not a fitness function, can make once the sensors are in place.
Inferences:
  • If some architectural judgments genuinely cannot be reduced to executable fitness functions — for example, whether a new bounded context is actually warranted, as opposed to whether an existing dependency rule was violated — then offloading "structure" entirely onto sensors while reviewers focus on "why" risks silently losing exactly that residual judgment, a limitation the article does not discuss.

Narratives

  • GitHub Copilot's 2023 Microsoft Research study reporting a 55.8% task-completion speedup became "the working definition of what AI does to software development" through vendor decks and hiring plans; two years later, METR's randomized controlled trial on experienced developers in their own large codebases found the opposite — developers were 19% slower, while believing themselves 20% faster.
    → Implicitly supports the proposition that raw code-production speed is the wrong thing to optimize for or measure, motivating the pivot to comprehension as the article's real subject.
  • Amazon's March 2026 storefront outages were traced to AI-assisted changes merged without proper review, prompting a code safety reset and a new senior-engineer approval requirement for all AI-assisted code — a procedural fix that, the article argues, sits one layer above the untouched structural problem of nobody holding the reasoning behind the changes.
    → Implicitly supports the claim that the context gap has real, visible organizational consequences at the team altitude, and that procedural reactions (approval gates) treat the symptom rather than the cause the context store is meant to fix.
  • A reviewer asking why a service rejects requests over 5MB finds the spec, the test, and the fitness function that jointly explain the constraint, all in the repository — and an AI agent picking up the same feature a quarter later starts from that same spec and fitness-function catalog rather than reconstructing the reasoning from scratch.
    → Concretely illustrates what the abstract four-layer context store cashes out to in an ordinary review moment, grounding the framework's payoff in a recognizable, mundane scenario rather than leaving it abstract.

Questions addressed

  • ✅ What happens to an organization's understanding of a system when AI accelerates the writing of the system's code?
    The understanding no longer ships automatically with the code; it must be deliberately captured and persisted through a context store fed by spec-anchored SDD, TDD, and fitness functions.
  • ⭕ Why do developers report feeling faster with AI tools even when controlled studies (METR) show they were measurably slower?
    The article names the 39-point perception gap as a symptom but does not explain the underlying causal or psychological mechanism behind the mismatch between felt and measured productivity.
  • ✅ How should architecture practice change now that architectural characteristics must be enforced at AI-generation speed rather than at human-typing speed?
    Run spec-anchored SDD, TDD, and architectural fitness functions together as one verification system so conformance checks run continuously in CI rather than depending on a human reviewer's read-through pace.
  • ⭕ If spec-authoring and review must scale to match AI-generation speed the same way code review currently cannot, does spec-authoring capacity become the next finite, overloaded resource — just as senior-reviewer capacity already has?

Gaps

  • incompleteThe article treats finite senior-reviewer capacity as the crisis that pushed conformance checking onto automated fitness functions, but never asks whether spec-authoring capacity is symmetrically finite. The social practice it prescribes for spec-anchored SDD — "the spec is authored and reviewed like code, with diffs and approval, and amended when reality diverges from intent" — reintroduces a human bottleneck at the intent layer even as it removes one at the structural layer. If spec review cannot keep pace with AI-generation speed any better than code review currently does, the context store risks simply relocating the reviewer-capacity crisis from PRs to specs rather than resolving it.
    Read next: Read against extract-eisele-ai-coding-agents-need-clear-specs' U-shaped cost curve of specification completeness (heavier specs front-load human effort) and extract-agentic-code-review's documentation of reviewer capacity already collapsing under AI-generated PR volume, to test whether spec-authoring is next in line for the same collapse.
  • incompleteThe Amazon outage is presented as an outcome of the exact context gap this article diagnoses, but the article's own cited source (CNBC) attributes the outage to "changes merged without proper review" — a review-gate failure — not to a documented absence of persisted intent, behavior, or conformance records specifically. The causal attribution to the context gap, versus a more generic review-diligence lapse, is asserted rather than demonstrated from the available reporting.
    Read next: Follow further Amazon incident reporting or post-mortems, if published, to see whether investigators identified a missing intent/behavior/conformance record specifically, or a simpler story of a skipped approval step that a context store would not by itself have prevented.
  • incompleteThe article's context store distinguishes a machine-readable knowledge graph from a human-readable system spec, but doesn't interrogate whether prose-form specs can actually carry the relational structure that intent needs — priorities among constraints, non-negotiables versus preferences, time-bound versus principled rationale. This is the same gap the corpus has already flagged in Osmani's intent-debt series: existing artifact formats (specs, AGENTS.md, ADRs) may be "flat" against intent's real structure. The two-projection move is a step toward richer structure but stops short of specifying what the knowledge graph's schema actually captures that a flat spec would miss.
    Read next: Read against extract-osmani-intent-debt's gap on intent-as-data versus intent-as-prose, and knowledge-representation literature (description logics, semantic web) on what a genuinely structured, queryable intent schema would need to encode.
