AI-assisted software development is best understood as an amplifier of a team's existing practices rather than a disruptive break, so the field's real work is rebuilding the human-era practices that are predictably breaking—through emerging disciplines like the supervisory middle loop, risk tiering, TDD-as-prompt-engineering, and platform 'bullet trains'—amid far more shared uncertainty than settled answers.

mixed · mixed — A 'fragments' collection: loosely linked tidbits and open questions from the Thoughtworks Future of Software Development Retreat plus three external pointers (O'Grady, Tornhill, a TDD testimonial). Partly expository (reporting takeaways), partly argumentative (Fowler's own verdicts), partly a list of unresolved questions.

Key terms

  • supervisory engineering middle loop
    A newly emerging category of engineering work between generating code and shipping it, in which humans supervise and steer AI-produced output
  • risk tiering Risk Tiering
    Classifying work by risk level and applying proportionate rigor and oversight, framed as 'the new core engineering discipline' for AI-assisted development
  • TDD as the strongest form of prompt engineering
    The reframe that writing tests first is the most effective way to specify intent to and constrain an LLM coding agent
  • debt accelerator Velocity Debt / Debt Accelerator
    AI's effect on a team lacking delivery best practices: because writing code was never the bottleneck, faster code just accelerates the accumulation of debt
  • funhouse mirror / amplifier
    AI (per the 2025 DORA report) reflects and magnifies the good, bad, and ugly of an existing pipeline rather than transforming it
  • AI-friendliness AI-Friendly Code (Code Health for Agents)
    Adam Tornhill's metric: the probability that AI-generated refactorings preserve behavior and improve maintainability, shown to be higher in healthy code
  • bullet trains
    A platform-team metaphor for a fast-but-safe path that lets application teams adopt AI with good security built in by default
  • besieged
    Stephen O'Grady's characterization of how much of the software profession feels under threat from AI

Arguments

Faster AI code generation does not by itself improve software delivery; teams must first fix their upstream practices. strength: high
Explicit premises:
  • AI is an amplifier of whatever pipeline a team already has (2025 DORA 'funhouse mirror').
  • Writing code was never the delivery bottleneck.
  • Where delivery best practices are absent, the velocity multiplier becomes a debt accelerator.
Implicit premises:
  • The binding constraints on delivery lie upstream of code creation—in design, testing, review, and release.
  • A dysfunction that is amplified rather than removed grows faster under AI.
Inferences:
  • Therefore the payoff from AI is contingent on pre-existing delivery discipline, not on code-generation speed.
TDD is a central technique for using LLM coding agents effectively. strength: medium
Explicit premises:
  • Leading-edge LLM users report that clear tests and the TDD cycle are essential to using LLMs well.
  • TDD is framed as the strongest form of prompt engineering.
Implicit premises:
  • LLMs need an executable, legible specification to constrain and verify their output.
  • A red-green test loop gives the agent an unambiguous target and fast feedback.
Inferences:
  • Therefore investing in tests is a primary lever for driving agents.
Security in AI-assisted development must be delivered through platforms, not left to individual discipline. strength: medium
Explicit premises:
  • Security is tedious and naturally deprioritized ('make it work, then reliable, then secure').
  • Other engineering disciplines bake a significant safety factor into their designs.
  • Platform teams can provide a fast-but-safe path ('bullet trains') that makes secure AI deployment the default.
Implicit premises:
  • Relying on individual developer diligence for security fails at AI's velocity and scale.
  • Failing to build in safety factors could cause damage larger than a falling bridge.
Inferences:
  • Therefore platform thinking is essential to safe AI adoption.

Questions addressed

  • ✅ Was the retreat meant to produce a new manifesto for AI-enabled development?
    No—the aim was a shared understanding and a shared set of questions, not a doctrine.
  • ✅ Is AI's primary role in software development that of a disruptor or an amplifier?
    An amplifier/accelerator of existing practice—a funhouse mirror per the 2025 DORA report.
  • ⭕ Will the rise of specifications drag development back toward waterfall?
    Fowler's instinct: it depends on workflow; LLMs don't reduce the value of small releasable slices and should increase the frequency of that cycle.
  • ⭕ Will LLMs be cheaper than humans once token subsidies end?
    Unknown—there is little visibility into the true cost of tokens now, let alone in a few years.
  • ✅ How should security be handled in AI-assisted development?
    Through platforms that provide a fast-but-safe path with security built in, plus engineered safety factors.
