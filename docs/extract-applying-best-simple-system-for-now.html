Applying Best Simple System for Now for Software Design

Ben Linders · article · 2026-05-09 source ↗ Reader 🖥️

Changeability & OptionalityMalleable SoftwareSoftware Design FundamentalsTechnical Debt
The trade-off between technical debt and missed deadlines is a false dichotomy; by building the simplest system that meets only current requirements to a high standard (Best Simple System for Now), teams can ship continuously while keeping future change cheap — because simplicity, not speculative flexibility, is the real source of adaptability.

practical · interview-article — An InfoQ article reporting on Daniel Terhorst-North's GOTO Copenhagen talk, supplemented with a follow-up interview. Primarily practical — prescribes a design discipline (BSSN) and offers concrete heuristics for adoption.

Key terms

  • Best Simple System for Now (BSSN) Software Design Fundamentals
    A design approach where code meets all current requirements to a high standard, contains no speculative or extraneous code, and is future-proof through simplicity rather than extension points
  • dimensions of change Changeability & Optionality
    The multiple, unpredictable directions in which a product could evolve at any time; anticipating one dimension introduces speculative complexity that trades off against all the others
  • First Sight Software Design Fundamentals
    The ability to see what is actually there — the real requirements and real system — rather than what you expect or wish to see (borrowed from Terry Pratchett)
  • gold-plating Software Design Fundamentals
    Adding unnecessary polish, generalization, or speculative features beyond what current requirements demand
  • sketches Malleable Software
    Exploratory code written to test ideas, held to a lower standard than production code but still requiring good naming and simplicity

Propositions

  • explicitThe trade-off between technical debt and missing deadlines is a false dichotomy.
  • explicitWith the right design decisions, you can have a shippable product all the time while keeping quality high.
  • explicitcontestedSpeculative code and extension points make future changes harder, not easier, because the actual direction of change is unpredictable.
    contradicts ←Worse is better: JSON versus XML · p9 — Seemann treats XML's latent, ready-to-use standards (schema/versioning/XQuery/SAX) as a positive reason to prefer the richer format; BSSN holds that speculative extension points and unused capacity make future change harder, not easier — an unaddressed worldview tension, not a rebuttal Seemann makes.
  • explicitcorroboratedTrue future-proofing comes from simplicity — a simple system is easy to change in any direction, not just the direction you predicted.
    supports ←Worse is better: JSON versus XML · p5 — XML's gradual enhancement — start with plain nesting, add a schema only when a real need appears — is BSSN applied to data-format design: defer formalization and let simplicity carry adaptability. On this proposition Seemann is Beck-aligned, even as p9 pulls the other way.
  • explicitProgrammers' love of generalization is the primary cause of unnecessary system complexity.
  • implicitExperience makes engineers worse at simplicity, not better, because they become more confident in their ability to predict the future.
  • explicitBSSN is a practice discipline that must be learned through repeated experience, not through intellectual persuasion.
  • explicitPair programming is an effective guard against over-engineering because a partner challenges speculative assumptions.

  • → Literary metaphor that reframes the design problem as a perceptual one — programmers need to see what is actually there (First Sight) and monitor their own thinking for speculation (Second Thoughts)

Gaps

  • incompleteNo discussion of how BSSN handles genuinely uncertain or conflicting current requirements. The framework assumes 'current requirements' are knowable and stable, but in many contexts they are ambiguous or in flux.
  • incompleteNo treatment of team or organizational dynamics. BSSN is presented as an individual design discipline. How does it interact with architectural governance, platform teams, or mandated standards that may require speculative infrastructure?
  • uninformedNo engagement with the large body of work on evolutionary architecture (Ford/Parsons) or fitness functions, which addresses the same problem space — keeping systems adaptable — with complementary but different mechanisms.
  • illogicalMinor tension: the 'Best' characteristic requires telemetry, automated tests, API docs, and automated deployment paths. These are themselves forward-looking investments that go beyond 'current requirements'. The boundary between 'necessary quality infrastructure' and 'speculative code' is not clearly drawn.
