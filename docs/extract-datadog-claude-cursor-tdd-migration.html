How Datadog Used Claude and Cursor for Test-Driven Production Migration

Datadog's AI-assisted migration of a critical production system off a key-value backend onto a relational one succeeded not because Claude and Cursor generated code autonomously, but because a strong human-authored test suite plus a live blue/green validator supplied the pass/fail oracle that bounded and verified every AI-generated change — making the coverage of that verification, not the capability of the model, the ceiling on how much AI output could be trusted.

Key terms

  • Test-driven refactoring / migration
    A per-method migration loop in which a human-authored failing test defines the target behavior, the AI generates an implementation against the old code + new schema, and the passing/failing test — not the human reading the code — decides whether the output is accepted.
  • Pass/fail criterion (the test suite as oracle)
    A comprehensive test suite treated as the clear, mechanical accept/reject gate for every AI-generated change — the artifact that converts non-trusted AI output into trusted output.
  • Blue/green with a validator service
    Running the old FoundationDB-backed and new PostgreSQL-backed Stream Routers side by side on the same live requests, with a dedicated validator per cluster continuously comparing their routing responses and alerting the instant they diverge — a correctness check referenced live from the running old system rather than from frozen test cases.
  • Code modularity
    The property that let the new Stream Router implement the same API on PostgreSQL instead of FoundationDB without requiring changes elsewhere — the structural precondition that made a swap-the-backend migration tractable at all.
  • Suboptimal queries / niche optimizations
    Queries that return the correct results but issue far more round trips than necessary; the specific optimization patterns (batching, UNNEST tricks, common table expressions) that the AI could not discover on its own even though they exist in the literature, and had to be supplied by humans before the model could replicate them.

Arguments

The migration succeeded because of the verification infrastructure (test suite + live validator), not because the AI could autonomously produce trustworthy code. strength: high
Explicit premises:
  • The tests were the pass/fail criterion that told the team whether each AI-generated method was correct.
  • The models were not generating code autonomously — humans supplied the old implementation, new schema, and a failing test per method.
  • A live validator continuously compared old and new routers on real requests and alerted on divergence.
  • Wakim explicitly concludes the test suite's strength determined how much AI-generated code they could trust.
Implicit premises:
  • AI-generated code cannot be trusted on its own and requires an external, mechanical check before acceptance.
  • A sufficiently comprehensive, checkable oracle can substitute for a human reading and vouching for each change.
  • How much of the AI's output you can safely accept is bounded by how much of the intended behavior your verification actually covers.
Inferences:
  • Therefore the binding constraint on AI-assisted migration is the quality of the verification harness, not the capability of the model.
  • Therefore the engineering effort that pays off is invested in tests, schema design, and comparison tooling rather than in more autonomous prompting.
AI accelerates mechanical code migration but high-level optimization judgment remains a human responsibility. strength: high
Explicit premises:
  • AI-generated queries returned correct results but issued far more round trips than necessary.
  • The models did not discover known optimization patterns (batching, UNNEST, CTEs) independently, though these exist in the literature.
  • Humans wrote the optimized versions, after which the models could replicate the pattern in subsequent methods.
  • Higher-level prompts had limited effectiveness.
Implicit premises:
  • LLMs apply patterns present in their immediate context rather than reliably retrieving and instantiating techniques merely because they exist 'in the literature'.
  • Replication-from-an-exemplar is a different and easier capability than discovery-from-first-principles.
Inferences:
  • Therefore providing a worked exemplar, not a higher-level instruction, is the effective way to get optimized output from the model.
  • Therefore the non-obvious performance work stays with humans even in an otherwise AI-driven migration.

Narratives

  • The Datadog Stream Router migration told as 'what worked, what didn't': a KV-backed API that reconstructed relational logic in-process and took up to 45 minutes on its hardest operations was refactored, method by method, onto a relational PostgreSQL schema using Claude and Cursor against failing tests, deployed blue/green against the old system, and ended with ~1s operations, 40x smaller storage, and ~90% lower DB cost.
    → Serves as the single concrete existence proof grounding the article's thesis that a strong test suite (and a live comparison oracle), not model autonomy, is the enabling condition for trustworthy AI-assisted migration of a critical production system.

Questions addressed

  • ✅ What made an AI-assisted, test-driven migration of a critical production system actually succeed?
    A comprehensive human-authored test suite acting as the pass/fail oracle per method, strong modularity that let the same API run on a new backend, a per-method loop feeding the AI old code + new schema + a failing test, and a live blue/green validator comparing old and new responses on real traffic.
  • ✅ Where did the AI-assisted approach fall short?
    Higher-level prompts were less effective than narrow test-anchored ones; Claude generated correct-but-suboptimal queries and could not discover known optimizations independently; and token consumption was high due to feeding full test-output dumps and the iterative test/code/schema loop.
  • ✅ Can AI be trusted to refactor a critical production system?
    Conditionally yes — trust is proportional to the strength of the verification harness: the test suite gates each change and the live validator catches any divergence from the old system in production.

Gaps

  • incompleteThe blue/green validator verifies behavioral EQUIVALENCE to the old Stream Router, not correctness: it checks that the new router's responses match the old one's on each live request. Any latent bug or quirk in the old FoundationDB-backed system would be faithfully preserved and pass the validator silently, and the per-method TDD loop likewise inherits whatever blind spots the human-authored failing tests contain. The article never examines this — it treats 'agrees with the old system / passes the tests' as equivalent to 'correct', so the oracle's own fidelity to true intent is unquestioned.
    Read next: Read against extract-what-survived-contact#g4 (the Goodhart case where a passable oracle diverges from true, uncheckable intent) and the reference-oracle concept: a referenced-live oracle covers every input but only relative to the reference's own behavior.
  • incompleteThe human/AI division of labor is characterized qualitatively (AI does first-pass translation; humans supply schema, tests, and optimizations) but never quantified: no data on what fraction of methods the AI got right on the first pass versus needed manual fixes, how many iterations the loop took, or how the AI-driven migration compared in effort or time against a conventional hand-refactor.
    Read next: Read the original Datadog engineering post (datadoghq.com/blog/engineering/ai-assisted-storage-migration) for any first-party figures the InfoQ summary dropped; then compare with extract-doordash-copilot-ios-test-migration#g1, which flags the same missing intervention-fraction data for an AI-assisted test migration.
  • incompleteHigh token consumption is named as a shortcoming but never costed: there is no total token or dollar figure for the migration, so the economic viability of the AI-assisted approach relative to a manual refactor cannot be assessed — the impressive 90% DB-cost and 40x storage wins are outcome metrics of the new architecture, not of the AI method itself.
