Agent-assisted software work is best handled not as ad-hoc prompting but as an explicit, skill-mediated development lifecycle — interactively grill the human into a spec, split the spec into tickets each scoped to a single agent session, implement, then code-review for refactoring smells — fronted by a dedicated planning phase (Wayfinder) that externalizes large, foggy plans onto the shared issue tracker so they survive beyond any one context window, with research, prototyping and TDD as first-class supporting stages.

The durable, transferable content is the set of workflow patterns the release demonstrates — staging agent work as an explicit spec->tickets->implement lifecycle, fronting it with a dedicated planning phase, reviewing for refactoring smells, and designing skills to be thin and AFK-usable. The extraction favours those patterns over the ephemeral specifics.

Key terms

  • flow skills
    The sequence of skills that together form the intended end-to-end development flow — grill/Wayfinder for planning, /to-spec, /to-tickets, /implement, /code-review, /commit; 'flow' means the main workflow path users should follow, not a psychological flow state. The two 'main flow skills' renamed in this release are /to-spec and /to-tickets.
  • /to-spec (formerly /to-PRD)
    The flow skill that turns grilled understanding into a specification. Renamed off 'PRD' because what it produced was not really a product requirements document but a specification — a broader term for a description of the thing you want to build, which can be technical, non-technical or a blend; the spec 'defines the destination'.
  • /to-tickets (formerly /to-issues)
    The flow skill that decomposes a spec into individual work items. Renamed off 'issues' because that felt biased toward GitHub/Linear; tickets are the units 'underneath the spec', 'the journey you take to enact the spec', each sized so development can be spread across multiple agent sessions.
  • Wayfinder
    A new pre-spec planning skill for a loose idea 'too big for one agent session and wrapped in fog', where the route to the destination isn't yet visible. It charts the way as a shared map on the repo's issue tracker — a GitHub issue whose typed sub-issues (research / grilling / prototype / task) carry blocking relationships and are each scoped to one agent session — worked one at a time until the route is clear, then converted to a spec. Removes the anxiety of hand-offs and 'blowing out of the smart zone', and is collaborative because it lives in GitHub.
  • grilling
    A central reference skill that teaches the LLM how to interrogate the human, used by grill-me and grill-with-docs (and Wayfinder's grilling tickets). It replaces plan mode: the agent asks questions one at a time, distinguishes facts (which it discovers by exploring the codebase) from decisions (which only the user can make), captures rationale in ADRs and a glossary, and gates on explicit human confirmation before enacting the plan.
  • refactoring smells
    Martin Fowler's named code smells (mysterious name, duplicated code, feature envy, data clumps, primitive obsession, repeated switches, divergent change, speculative generality, message chains, middle man). Because Fowler's Refactoring is old and heavily cited, these are 'deep in the agent's priors', so the code-review skill only needs to name each one in a sentence to make the agent identify it, name it back, and fix it.

  • explicitAgent-assisted development should be run as an explicit, staged lifecycle in which each stage is a discrete, named skill — grill/plan, then /to-spec, then /to-tickets, then /implement (one ticket per session), then /code-review, then commit — rather than ad-hoc prompting or a single 'plan mode'; this answers the recurring user question 'what is the main flow?'.conf: high

  • explicitWork should be decomposed into agent-session-sized units: a spec defines the destination, and tickets are the journey — sized so 'you can spread the development out over multiple agent sessions' and implement one ticket per fresh session — so the plan is chunked to fit the agent's effective context rather than to human sprint conventions.conf: high
  • explicitAn implementation skill should be deliberately thin, leaning on the agent's priors and the harness rather than encoding detailed instructions: /implement is 'very, very simple' ('implement the work described... use TDD where possible at pre-agreed seams, run type-checking and single test files regularly and the full suite once at the end, then code review, then commit') and earns its place mainly by naming and anchoring a stage in the flow, not by adding instruction volume.conf: high

Arguments

Agent-assisted development should be structured as an explicit, skill-mediated lifecycle — a dedicated planning phase (Wayfinder), then grill -> spec -> tickets -> implement -> code-review-for-smells -> commit — rather than ad-hoc prompting or a single plan mode.
Explicit premises:
  • Users kept asking 'what is the main flow?', i.e. there was no legible end-to-end path from idea to shipped code.
  • Planning that is too big for one agent session 'blows out of the smart zone' or the context window, so it needs its own externalized, chunked phase.
  • Agents carry canonical engineering knowledge (e.g. Fowler's code smells) in their priors that a thin skill can evoke by naming.
  • Naming artifacts correctly (spec, tickets) and staging them as discrete skills makes the workflow legible and re-runnable.
Implicit premises:
  • Codifying a workflow into reusable, installable skills produces more repeatable, higher-quality outcomes than ad-hoc prompting (asserted, never measured).
  • Large plans genuinely need a dedicated planning phase distinct from implementation, rather than the agent planning inline as it goes.
  • A skill's value can come from naming and anchoring a stage even when it adds little instruction — workflow legibility is itself worth a skill.
  • The author's personal, weeks-long usage generalizes across other users, models and codebases.
Inferences:
  • If the binding constraint is a legible, context-safe path from idea to shipped code, then discrete skills that stage and externalize the work supply that path.
  • Therefore the flow, not any single skill, is the product — each skill earns its place by occupying a stage in the lifecycle.
A prompt/skill for a well-known domain should evoke the model's priors by naming canonical concepts, not re-teach them.
Explicit premises:
  • Fowler's Refactoring is old and heavily cited, so its named smells are deep in the model's training data.
  • When each smell is named in a single sentence, the agent identifies it, repeats the name back, and fixes it.
  • The resulting code-review addition is ~10 lines and was 'outrageously useful' over a couple of weeks of testing.
Implicit premises:
  • The model's priors on a canonical source are reliable enough to act on from a one-line evocation.
  • Naming rather than teaching cuts token cost without losing fidelity.
  • What is canonical to human engineers is well and consistently represented across models' training data.
Inferences:
  • If the knowledge is already latent, then the skill's job is to activate it, not to supply it.
  • Therefore prompt design for canonical domains is subtractive: reference and name, don't re-explain.

    → Motivates the interactive-intent-elicitation design and illustrates that skill behaviour is model-dependent and needs iterative hardening, not one-shot authoring.
  • The author reveals he planned his next product — an upcoming self-paced 'AI coding crash course' — using Wayfinder, 'even non-coding stuff', and shows it off as a live example.
    → Generalizes Wayfinder beyond code to any large, foggy planning problem, and doubles as a course plug.

Questions addressed

  • ✅ What is the main flow you are supposed to use with the skills?
    A staged lifecycle: grill (or Wayfinder for large/foggy work) -> /to-spec -> /to-tickets -> /implement each ticket in its own session -> /code-review -> commit.
  • ✅ Why rename /to-PRD to /to-spec and /to-issues to /to-tickets?
    The artifacts produced were specifications and tickets, not PRDs and issues; 'spec' is the broader, accurate term for the destination, and 'tickets' is vendor-neutral where 'issues' leaned toward GitHub/Linear.
  • ✅ When should you use Wayfinder rather than grill-with-docs?
    Default to Wayfinder when the work is too big for one agent session and 'wrapped in fog' (the route isn't visible yet), and especially for anything touching front-end code; grill-with-docs suits work that fits a session.
  • ✅ How do you migrate to v1.1 given the renames?
    Delete and re-add skills via `npx skills add` (the installer won't rename in place), pass over the skills folder to remove stale ones, or point your agent at the repo and release notes to pull the new stuff.
  • ⭕ Does this workflow actually improve code quality or delivery outcomes versus ad-hoc prompting?
    Not addressed with evidence. Every claim rests on the author's personal use ('I tested this for a couple of weeks', 'outrageously useful') and popularity metrics (160K stars, 7M downloads), which measure adoption, not outcomes; there is no comparison against ad-hoc prompting or alternative flows.
