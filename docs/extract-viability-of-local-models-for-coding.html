Viability of local models for coding

Martin Fowler (martin@martinfowler.com) — memo by Birgitta Böckeler · rss · 2026-07-07 source ↗ Reader 🖥️

Agent HarnessAgent Tool ManagementCoding AgentsContext Rot & Context LimitsInference OptimizationLocal AI
The practical viability of local models for agentic coding is not a property of the model alone but an emergent, hard-to-predict outcome of interacting hardware (RAM, processing power, memory bandwidth), model (parameters, architecture, quantization, reasoning) and harness (context overhead, tool-calling schema) factors — which is why local coding assistants are still not a reliable 'plug and play' option for developers, even though specific configurations already deliver acceptable results for narrow tasks.

practical · expository — The memo is structured as a systematic breakdown of the factors that determine whether local models are usable for agentic coding — mostly expository (defining and explaining each factor) with an argumentative undertone driving toward a practical recommendation (a go-to local setup) and a practical verdict ('not yet plug and play').

Key terms

  • Viability
    Whether a local model is usable for genuinely agentic coding (not just autocomplete) by developers who are not willing to dive into specs, extra tooling and tweaks — a bar that includes ease of setup, not just raw capability.
  • Runnability
    Whether a model's weights (plus context) fit into available RAM/VRAM without crashing or dropping to unusably slow speeds — a binary-ish threshold distinct from speed or quality.
  • Harness Agent Harness
    The coding agent tool (OpenCode, Pi, Claude Code) plus its own configuration (skills, MCP servers) wrapped around the model — a source of context-window overhead and tool-call schema variance that becomes a bigger constraint locally than in the cloud.
  • MoE (Mixture of Experts) architecture
    A model architecture with a large total parameter count that activates only a subset of weights per inference step, so a MoE model of a given size needs less RAM and runs faster than a dense model of the same size — and, per the author's tentative hypothesis, may load more experts on machines with more RAM, improving quality.
  • QAT (Quantization-Aware Training) Inference Optimization
    Quantization simulated during training (as opposed to applied after training) so as to better preserve model quality at a given compression level; mentioned as 'the newest buzz' but not yet personally tested by the author.

Arguments

Local models are not yet a plug-and-play option for developers who don't want to spend significant time tuning them. strength: medium
Explicit premises:
  • Quality of outcomes was 'very hit and miss'.
  • Tool calling was tricky and models often failed at it.
  • Many interacting factors (hardware, model, harness) each require separate tuning decisions.
  • Even carefully chosen setups crashed unpredictably (e.g. a 48GB model on a 64GB machine 'went great at first, but then it quickly crashed').
Implicit premises:
  • 'Plug and play' is being judged against the near-zero-configuration UX bar set by cloud/frontier model products.
  • Typical developers have low tolerance for spec-reading, tooling setup and iterative tuning.
Inferences:
  • If the blocker is largely tuning burden rather than raw capability, better defaults and packaged configurations (not only bigger/better models) could close much of the remaining gap.
Reasoning is not always necessary for small local models on coding tasks, and can sometimes be counterproductive. strength: low
Explicit premises:
  • Reasoning-off runs were faster, as expected.
  • Reasoning-off runs performed the same to slightly better than reasoning-on runs in the automated eval.
  • Smaller reasoning models were observed looping ('Wait...', 'Actually...', 'But wait...') without converging.
Implicit premises:
  • The automated eval's tasks are representative enough of real coding tasks to generalize from.
  • Unproductive reasoning loops don't just waste tokens/time but can actively degrade the eventual answer, rather than being harmless overhead.
Inferences:
  • Local coding harnesses and runtimes should expose (and possibly default) a reasoning-off toggle, at least while context and speed are the binding constraints.

Narratives

  • Loading a 48GB model on the 64GB machine: it 'went great at first, but then it quickly crashed.'
    → Illustrates that sustained runnability depends on RAM headroom over time (context growth, other applications), not just whether the model weights initially fit — supporting the RAM proposition and foreshadowing that model size proximity to hardware limits is fragile, not a stable pass/fail line.
  • A model self-correcting a malformed tool call by recovering from a wrong parameter name (file.path instead of filePath).
    → Shows that agentic coding viability is less about zero tool-calling errors and more about a model's resilience/self-correction ability when errors occur — supporting the tool-calling proposition.

Questions addressed

  • ✅ How viable are local models for agentic coding today?
    Partially viable: acceptable for autocomplete and small, well-supported setups; not yet a plug-and-play option for full agentic coding without tuning effort.
  • ✅ What factors determine that viability, and how do they interact?
    The memo enumerates and explains roughly eleven interacting factors: RAM, processing power, memory bandwidth, parameter count, reasoning, tool calling, format, quantization, architecture, context window size, runtime, and harness.
  • ⭕ Why does the same model with identical settings produce different-quality code on machines with different RAM?
    The author offers only a tentative, self-flagged-as-unconfirmed hypothesis (MoE loading more experts on the higher-RAM machine); she states outright 'I'm not sure if that's true.'
  • ⭕ Does model format (GGUF vs. MLX) meaningfully affect perceived speed on Apple Silicon?
    The author felt no clear difference, but attributes this to her own unstructured evaluation method rather than concluding format doesn't matter.

Gaps

  • incompleteNo quantitative measurements appear anywhere in the memo — no tokens/sec, no benchmark scores, no proportions of tasks succeeding. All judgments ('quite good', 'hit and miss', 'nowhere near') are qualitative and felt, so the reader cannot independently verify or reproduce the viability verdict.
    Read next: Look for (or commission) instance-level benchmarking of local coding setups with reported tokens/sec, task-success rates and RAM-usage curves.
  • incompleteThe RAM-dependent quality mystery (same model + settings, different machine, better code) is raised but not investigated beyond a guess; no diagnostic was run (e.g. logging which experts get activated, comparing memory-pressure metrics between the two machines).
    Read next: MoE internals and expert-utilization instrumentation; any follow-up work (the author's own or others') that actually measures expert activation under memory pressure.
  • incompleteThe harness's own contribution to viability is only lightly explored — the author names context overhead and tool-schema differences as factors, and reports avoiding Claude Code for context-budget reasons, but never tests how much a harness purpose-built for small models (context budgeting, tolerant tool-call parsing, schema simplification) could shift the overall 'not plug and play' verdict.
    Read next: Harness-engineering approaches specifically designed for small/local models, e.g. a harness that narrows tool schemas and tolerates malformed calls.
  • incompleteQuantization is treated almost entirely by anecdote at a single setting: every model tried was Q4/4BIT, and QAT (flagged as 'the newest buzz' and 'supposed to preserve quality better') was never tried. The memo's viability verdict therefore rests on the most compressed, least-quality-preserving setup available, not on a comparison across quantization levels.
    Read next: Direct comparisons of Q4 vs Q6/Q8 vs QAT variants of the same model on the same coding tasks.
