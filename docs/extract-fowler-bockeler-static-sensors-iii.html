Three more static code analysis sensors

Birgitta Böckeler (published on martinfowler.com) · rss · 2026-05-20 source ↗ Reader 🖥️

Agent SkillsCode QualityEvolutionary Architecture & Fitness FunctionsFeedback LoopsModularity & Coupling
For maintainability concerns that cross file and module boundaries, computational static-analysis sensors (dependency rules, coupling metrics) can usefully enforce a few structural invariants but cannot judge what coupling is appropriate; the most productive sensor is an inferential, prompt-driven AI modularity review that reads the code itself, with the deterministic tools relegated to enforcement of known rules and to triage of review attention.

practical · mixed — An experience report that argues for a practical position: how to combine computational and inferential sensors for cross-module maintainability when AI is doing the coding. Mostly expository/narrative ('here is what I tried') but with clear normative takeaways.

Key terms

  • sensor Feedback Loops
    Any tool — deterministic or AI-driven — that feeds back signals about a code change to the agent (or human) so that self-correction is possible before issues reach human review. Sensors run at different cadences: in-session, in CI, on a slow schedule, or in production.
  • computational sensor Feedback Loops
    A deterministic, rule-based tool whose output is reproducible given the same inputs (linter, type-checker, dependency-cruiser, coupling-analyser, SAST).
  • inferential sensor Feedback Loops
    An LLM-driven review (prompt + model + sometimes auxiliary CLI tools) whose output is interpretive and not bit-reproducible. Useful when 'what counts as bad' depends on context.
  • self-correction guidance Feedback Loops
    Extra prose embedded in a sensor's error/warning messages, written specifically so that an AI agent reading the output understands what to do — effectively a controlled prompt injection at the point of failure.
  • dependency rules Modularity & Coupling
    Declarative constraints (in this case dependency-cruiser config) that forbid imports across designated layer boundaries — e.g. clients must not import from services.
  • coupling data Modularity & Coupling
    Per-file metrics about incoming and outgoing imports/calls, produced by her custom 'coupling-analyser' CLI, optionally visualised as e.g. a DSM.
  • AI modularity review Modularity & Coupling
    A scheduled, prompt-driven LLM analysis that reads the actual code (not just metrics) using a curated 'skills' pack (Vlad Khononov's Modularity Skills) and emits concrete refactoring recommendations.
  • Modularity Skills (Khononov) Agent Skills
    An external, reusable prompt/skill bundle by Vlad Khononov that operationalises modularity heuristics for LLM-based codebase review.
  • garbage collection (metaphor)
    Used by analogy: the AI modularity review acts like a periodic sweep that finds accumulated waste (duplicate routes, scattered parameters, misplaced responsibilities) that the agent would not refactor on its own initiative.
  • impact radius Modularity & Coupling
    The set of callers/dependents potentially affected by a change to a given file — proposed as a future use for coupling data during human code review triage.

Arguments

Computational dependency rules are worth adopting in an AI-coding harness despite their narrow expressive range. strength: high
Explicit premises:
  • Once authored, they actively enforce structure rather than describe it.
  • AI eliminates the configuration-syntax cost that previously made them unattractive.
  • Agents observably self-correct from dependency-cruiser feedback.
  • Custom error messages can be used as self-correction guidance.
Implicit premises:
  • Layered structure expressed via folders/imports is a good-enough proxy for the architectural intent the team actually cares about.
  • The maintenance cost of the rule-set is lower than the maintenance cost of an equivalent markdown guide plus human enforcement.
Inferences:
  • Teams who skip writing such rules are silently accepting that the agent will mis-place files within a few iterations.
  • If the rules ARE the architecture documentation, then dependency-cruiser config becomes load-bearing for onboarding, and losing the maintainer of that config becomes a knowledge-loss event.
Raw coupling metrics are not, on their own, a useful sensor for an AI agent. strength: high
Explicit premises:
  • Strong models reading the metric output misclassify intentional DI factories and shared schemas as smells.
  • What counts as appropriate coupling is context-dependent.
  • Without a suppression mechanism, false positives accumulate as noise.
Implicit premises:
  • The model's misclassifications are not addressable purely by better prompting — the problem is information-theoretic (metrics omit intent).
Inferences:
  • The right place for coupling metrics is upstream of human/agent attention allocation (impact-radius triage), not downstream as a quality verdict.
  • Anyone selling a 'coupling dashboard for AI codebases' is selling a sensor that fires noise without architectural context.
An LLM modularity review that reads the actual code, driven by a curated skill pack, is the best available sensor for cross-module maintainability under AI-assisted development. strength: high
Explicit premises:
  • It produced concrete, valid refactoring findings (duplicate routes, scattered parameters, misplaced auth code).
  • It correctly contextualised the 'god module' false positives that the metric-only run produced.
  • Repeated runs add coverage; single runs miss issues.
  • Without it (and without expert human review), the agent compounded inadvertent debt.
Implicit premises:
  • The skill pack (Khononov's) encodes enough good architectural taste to be trusted as a sensor; its quality is largely outside the team's control.
  • An LLM reading the full codebase periodically is affordable in tokens and wall-clock — i.e. inference cost continues to fall or the codebase stays small enough.
Inferences:
  • Modularity quality in AI-built systems becomes a function of the modularity skills you import, not of the implementing model.
  • Organisations now depend on third-party skill maintainers in a way structurally analogous to dependency on linter rule-set maintainers — but with much less mature ecosystem hygiene.
Computational and inferential sensors are not substitutes; they belong in different parts of the harness.
Explicit premises:
  • Deterministic rules excel at enforcement of known invariants.
  • Inferential reviews excel at judgement calls about what is appropriate in context.
  • Grounding the inferential review in the deterministic data made little difference in this case.
Implicit premises:
  • The two sensor classes fail in non-overlapping ways, so coverage is additive.
  • The cost of running both is justified relative to letting the agent compound debt.
Inferences:
  • Harness design becomes a portfolio problem: which deterministic rules to enforce, which inferential skills to run, and at which cadence.

Narratives

  • The `clients-no-services` rule: she shows the dependency-cruiser JSON for forbidding API clients from importing from the services orchestration layer, plus the expanded error message that recaps the entire layering convention as self-correction guidance for the agent.
    → Concretely demonstrates the 'custom error messages as agent guidance' pattern; supports the proposition that deterministic rules can substitute for a markdown guide.
  • The agent declared her dependency-injection factory a 'god module' and her shared frontend/backend zod schema a 'god module', based purely on coupling metrics — both being deliberate, well-known patterns.
    → Carries the argument that raw coupling metrics strip away intent and therefore mislead even a strong LLM; without telling the abstract claim, the anecdote makes it.
  • Changing the way a user specifies a date range forced edits in 40+ files; the modularity review then surfaced 'Request parameters repeated at every level' as a finding, confirming her intuition that the design was leaking.
    → Validates Sensor 3 by closing the loop between a felt pain (40-file change) and a structural finding (no parameter object).
  • Three nearly-identical route files, each reimplemented when a new backend endpoint was added, because the agent will copy-paste rather than refactor on the third or fourth repetition.
    → Supports the proposition that AI agents do not autonomously generalise across repetitions and that a separate sweep is needed.
  • Auth logic accidentally lurking inside the wiring/DI factory.
    → Illustrates the broader claim that responsibilities drift to whatever module the agent happened to be editing, and that only a semantic reviewer notices.
  • Running the modularity review a second time, with no memory of the first, surfaced a new issue.
    → Supports the proposition that LLM reviews are usefully stochastic and should be run multiply.

Questions addressed

  • ✅ Can deterministic static analysis enforce architectural boundaries in an AI-coded codebase?
    Yes, but only for what imports / file paths can express; pairs well with custom error messages that double as agent guidance.
  • ✅ Are raw coupling metrics, fed to an LLM, a useful modularity sensor?
    No — even Claude Opus 4.7 misclassifies intentional patterns; the data lacks the context the LLM needs.
  • ✅ Is a prompt-driven LLM modularity review (that reads the code) a useful sensor?
    Yes — it found valid, concrete refactorings; run it more than once.
  • ⭕ Where in the change lifecycle should the AI modularity review live?
    She wants it nearer the commit, scoped to changed files, but hasn't built that yet.
  • ⭕ Who maintains the dependency rules and the custom coupling-analyser once the original author moves on?
    Not addressed.
  • ⭕ What happens to a team's design intent if it lives only in dependency-cruiser config and an imported Modularity Skill pack?
    Not addressed.
  • ⭕ How does the architecture get debated and evolved when it's expressed as enforcement rules rather than as prose?
    Not addressed.
  • ⭕ Does this approach generalise beyond a single TypeScript/Next.js app rebuilt by one expert?
    She notes risk profiles vary by software type but doesn't test generalisation.

Gaps

  • incompleteNo treatment of the social-and-organisational fate of the artefacts themselves. Dependency-cruiser configs, custom coupling-analyser CLIs, and imported Modularity Skill packs are all new load-bearing artefacts. Who owns them? Who updates them when the architecture should change? In the absence of an owner, the rules outlive the intent behind them and become either silently violated or silently obsolete — the classic linter-suppression-debt pathology, scaled to architecture.
    Read next: Cutler on institutionalised overload and waste; Fearless Change (Manns & Rising) on what it takes for organisational invariants to survive; Tornhill's Software Design X-Rays on attaching social/ownership data to code-quality signals; Vlad Khononov's modularity work directly (since his Skills are now part of the harness).
  • incompleteThe article treats the dependency-cruiser rule set as a substitute for a markdown structure guide, but it skips the question of whether enforcement and explanation are interchangeable. A rule that says 'clients cannot import services' enforces a constraint but does not teach why the layering exists — and the moment the constraint is wrong, the team has no shared prose to reason with. This silently relocates design intent into tooling that few people maintain, and few people can read.
    Read next: Ousterhout 'A Philosophy of Software Design' on the role of design documents; Parnas on module specifications; Hickey 'Simple Made Easy' for the distinction between rules-as-policy and rules-as-understanding.
  • incompleteHeavy reliance on Khononov's externally maintained Modularity Skills pack as the thing that makes Sensor 3 work. The article does not examine: skill-pack versioning, drift, security/supply-chain assumptions, or what happens when the skill author changes their mind. The harness now has a third-party dependency at the level of taste, not just at the level of code.
    Read next: Khononov's Learning Domain-Driven Design and his coupling talks; literature on prompt drift; the emerging 'skills-as-dependency' discussion (Osmani's agent-skills writing).
  • incompleteThe coupling-analyser is described as a vibe-coded custom CLI. The article acknowledges it would benefit from suppression of legitimate high-coupling hubs but does not engage with the more interesting question: if your coupling tool requires hand-curated exceptions for every architectural pattern, you are effectively encoding architectural intent in two places (the rules AND the exception list). At that point, why not skip the metric layer and go directly to the inferential review?
    Read next: Tornhill, Software Design X-Rays; the dependency-cruiser community's own discussion of when DSM-style tools earn their keep.
  • incompleteThe article observes that the AI modularity review found duplicate routes, scattered parameters, and misplaced auth — i.e. exactly the things a senior reviewer would find in a half-hour read of the code. There is no contrastive baseline against human-only review, against pair-review with the original implementing agent, or against a much cheaper rubric like 'review every file changed >X times in the last 30 days'. Without such a baseline, the proposition 'the review is the best available sensor' is unfalsifiable on the evidence given.
    Read next: DORA reports' methodology on baselines; Beck on test-commit-revert and 'starving genies' as alternative scaffolds for keeping AI-edited code honest.
  • incompleteNo engagement with the relationship between this approach and existing fitness-function / architecture-as-code tooling (ArchUnit, NetArchTest, ts-arch, Modulith, even Structure101). dependency-cruiser is one point in a much larger design space; treating it as 'the' dependency-rules sensor leaves the reader without a sense of where the boundary lies between custom configuration and adopting an established framework.
    Read next: Ford/Parsons/Kua 'Building Evolutionary Architectures' on fitness functions (which she links to in the framing); ArchUnit; ts-arch.
  • incompleteThe framing assumes the agent is always the implementing actor and the human (plus other agents) are the reviewers. It does not consider the inverse case — where humans implement and the AI modularity review is the rubric against which their code is judged — even though the same tooling would work. This matters because it shapes whether the harness is 'AI guardrails' or 'organisational architecture discipline that happens to use AI'.
    Read next: Cutler on operating-system design; Beck on team practices.
  • illogicalShe claims dependency-cruiser rules are 'a useful replacement for describing code structure in a markdown guide', but a markdown guide and an enforcement rule do different epistemic work: one teaches, one constrains. Treating them as substitutes is the same move that has historically led teams to delete their architecture docs in favour of 'the tests are the spec', then discover that nobody can reason about the system any more.
    Read next: Ousterhout on documentation; the long literature on tacit-vs-explicit knowledge in software teams.
    closed byTony Seale's Post — That gap warns that migrating meaning into machine-only enforcement tooling (dependency-cruiser rules replacing a markdown structure guide) destroys the human-readable prose that teaches — the 'tests are the spec' failure. Vault-LD's core move directly answers it: keep prose AND typed structure/semantics in one file so the two epistemic jobs — teaching (markdown) and constraining/validating (YAML-LD) — coexist per-file rather than one substituting for the other.
