Vibe coding and agentic engineering are getting closer than I'd like

Simon Willison · email · 2026-05-08 source ↗ Reader 🖥️

Agentic EngineeringCoding AgentsNear-Zero-Cost CodeTrustVibe Coding
As AI coding agents become more reliable, the experiential distinction between 'vibe coding' (unreviewed, low-accountability coding) and 'agentic engineering' (expert-supervised AI coding) is collapsing, which forces practitioners to find new trust signals — usage over inspection — while the entire software development lifecycle must restructure around dramatically cheaper code production.

Key terms

  • vibe coding Vibe Coding
    Writing software by describing what you want to an AI without reviewing the generated code, accepting it if it works, iterating if it doesn't — no concern for code quality, security, or maintainability
  • agentic engineering Agentic Engineering
    A professional software engineer using AI coding agents while applying their full expertise in security, performance, maintainability, and operations — the goal is higher quality, not just faster output
  • normalization of deviance Trust
    The gradual erosion of safety standards as repeated success without incident makes people comfortable skipping precautions — each time AI code works without review, the practitioner trusts it more at the margin

Arguments

Practitioners should accept that vibe coding and agentic engineering are converging and find new heuristics for trust rather than clinging to the line-by-line review standard. strength: medium
Explicit premises:
  • AI agents now reliably produce correct code for straightforward tasks
  • Even an experienced engineer like Willison has stopped reviewing all agent output
  • Engineers in large organizations already trust other teams' code as semi-black boxes
Implicit premises:
  • The line-by-line code review standard was itself a proxy for trust, not intrinsically necessary
  • If a practice is already widespread in a different form (trusting other teams), doing the same with AI is a matter of degree, not kind
Inferences:
  • If code review was always a proxy for trust, then any reliable trust signal (usage history, test suites, reputation) could substitute — including demonstrated AI reliability over time
Usage history is a better trust signal than artifact quality in an era of cheap AI-generated code. strength: high
Explicit premises:
  • AI can now produce repos with 100 commits, comprehensive tests, and beautiful READMEs in 30 minutes
  • These artifacts are indistinguishable from carefully crafted human work
  • The author cannot tell the quality difference even for his own projects
Implicit premises:
  • Quality signals that can be cheaply faked cease to be reliable signals (Goodhart's Law applied to code artifacts)
  • Usage over time is harder to fake than artifact quality
Inferences:
  • The entire open-source evaluation heuristic (stars, commit count, test coverage, documentation quality) may need to be supplemented or replaced by usage-based metrics
The SDLC must restructure because its timing assumptions are broken. strength: medium
Explicit premises:
  • Code production has gone from ~200 lines/day to ~2000 lines/day
  • Design processes were calibrated to the cost of building the wrong thing being high
  • If implementation is cheap, the cost of wrong designs drops
Implicit premises:
  • Code volume is a reasonable proxy for development cost (ignoring that debugging, integration, and ops may not scale the same way)
  • Organizations can actually restructure their processes fast enough to capture these gains
Inferences:
  • Testing, code review, deployment, and monitoring processes must also scale by 10x or become bottlenecks that negate the production speed gains

Questions addressed

  • ⭕ Is there a meaningful distinction between vibe coding and agentic engineering?
    Willison argues the distinction was clearer in theory than it is in practice — the boundary is eroding as agents improve, and he has not found a stable new boundary.
  • ✅ How should practitioners evaluate software quality when AI can produce perfect-looking artifacts cheaply?
    Shift the trust signal from artifact quality to usage history — has someone actually used this in production for weeks?
  • ✅ Will AI replace professional software engineers?
    No — AI amplifies expertise; the work remains 'ferociously difficult'; most people would rather buy professionally managed software.
  • ⭕ What happens to the SDLC when code production becomes 10x faster?
    Willison identifies that both upstream (design) and downstream (review, ops) processes must change, but does not specify how.
  • ⭕ When does the normalization of deviance in AI-assisted coding lead to a serious failure?
    Willison flags the risk explicitly but has no answer — he acknowledges he will likely 'get burned' at some point.

Gaps

  • incompleteWillison identifies that the SDLC must restructure but does not address the downstream bottlenecks — if you produce 10x more code, testing, review, deployment, monitoring, and incident response must also scale, and it is unclear whether they can. The essay focuses on code production and design but says nothing about ops, security review, or organizational change management.
    Read next: Charity Majors on observability-driven development; Google's DORA research on deployment frequency vs. stability; Will Larson on engineering org design under scaling pressure.
  • incompleteThe 'usage as trust signal' argument assumes that usage reveals bugs, but many serious defects (security vulnerabilities, data races, subtle data corruption) can lurk for months or years without being triggered by normal usage. Usage is a better signal than artifact appearance, but it is not a sufficient signal for safety-critical properties.
    Read next: Literature on latent defects in software; Nancy Leveson's work on system safety; the Therac-25 case.
  • incompleteThe essay treats the accountability gap between human teams and AI agents as an acknowledged discomfort but does not explore solutions — contractual liability, organizational accountability structures, or the possibility that AI vendors could offer guarantees or insurance for agent-produced code.
    Read next: Legal scholarship on AI liability; Anthropic's own responsible scaling policy; discussions of AI insurance markets.
  • incompleteWillison's 'amplifier of expertise' framing implies that junior engineers benefit less from AI tools, but the essay does not address the apprenticeship problem: if AI handles the straightforward tasks that juniors traditionally learn on, how do new engineers develop the expertise that makes AI an amplifier?
    Read next: Matt Rickard and Addy Osmani on the junior engineer pipeline; apprenticeship models in medicine and law for comparison.
