Don't train the model, evolve the harness

Akshay · tweet · 2026-04-06 Reader 🖥️

Agent HarnessAgentic LoopContext EngineeringEvalsMulti-Agent OrchestrationRecursive Self-ImprovementRiding Model Improvements
For a model you don't control — a frozen, rented commodity whose weights you can't change — the reliable leverage on agent performance lives in the harness (the code that feeds the model context, runs its tool calls, and decides when a run ends), not in the weights: a benchmark score conflates model and harness, so most apparent 'model failures' are harness failures (often mundane output-plumbing, not reasoning), and the disciplined way to capture that leverage is an automated fitness-selection loop that adds one mechanism at a time and keeps it only when it clearly beats the current best — a process that compounds until the harness runs out of tricks and the residual misses are genuine capability gaps.

practical · argumentative thread (with an appended expository deep-dive) — The thread proper is argumentative: a thesis ('don't train, evolve the harness') defended by one worked example and three ablation-style findings. The quoted article is expository/taxonomic — it defines the harness, enumerates 12 components and 7 design decisions, and asserts rather than argues. Prescriptive intent throughout (what to invest in, where to look when an agent fails), so practical.

Key terms

  • harness Agent Harness
    The complete non-model software wrapping the LLM — the runtime that feeds it context, runs its tool calls, and decides when a run ends; concretely the 12 components (orchestration loop, tools, memory, context management, prompt construction, output parsing, state, error handling, guardrails, verification loops, subagent orchestration). 'If you're not the model, you're the harness.'
  • evolve (the harness)
    Improve the harness by an automated fitness-selection loop that ADDS and REMOVES mechanisms to fit the current model — not monotonic growth. The quoted deep-dive is explicit that complexity should shrink as models absorb capabilities. Opposed to 'train'/'fine-tune' (change the model's weights).
  • the automated harness loop (proposer + outer loop)
    A Claude 'proposer' adds exactly one mechanism per iteration; an outer loop accepts it only if it clearly beats the current best. Accepted mechanisms therefore compound. This is the concrete method behind 'evolve' — recursive self-improvement of the code around a fixed model.
  • benchmark-score conflation
    A benchmark score measures the model AND its harness together; until the harness is fixed you cannot attribute a failure to either. The 0% legal-agent score measured file-plumbing, not reasoning.
  • harness thickness / scaffolding
    How much logic lives in the harness versus the model. Scaffolding is temporary construction infrastructure — removed when the building is complete; as models improve, harness complexity should decrease (Anthropic deletes planning steps as models internalize them).

Arguments

Evolve the harness rather than train the model. strength: high
Explicit premises:
  • A frozen model plus a rewritten harness matched a frontier model at ~7x lower cost per task, with zero weights changed.
  • Five harnesses on the same model/judge/tasks spanned 3.5% to 80.1% — the harness dominates outcome.
  • The harness search can be automated by a cheap proposer + keep-if-better outer loop whose accepted mechanisms compound.
  • Code-layer harness fixes transfer across models.
Implicit premises:
  • You rent a frontier model whose weights you cannot change, so training is not actually available as a lever for most teams.
  • Harness iteration is cheap, fast, and controllable, whereas training is expensive, slow, and gated by the model owner.
  • The binding constraint for your task is the harness, not the model's capability ceiling — the load-bearing and undischarged assumption.
Inferences:
  • Therefore spend marginal effort on the layer you own and can iterate cheaply (the harness) rather than on the model.
A benchmark score (even a 0%) does not tell you the model failed. strength: high
Explicit premises:
  • The judge only grades files saved in the right place under the exact requested filename.
  • The model did the legal analysis correctly but kept saving it under the wrong name, in a scratch folder, or not at all.
  • Five harnesses on the same model produced scores from 3.5% to 80.1%.
Implicit premises:
  • A benchmark score is a joint function of model capability and harness quality.
  • Harness failure modes (output plumbing, tool orchestration) are systematically mistaken for reasoning failures.
Inferences:
  • Therefore fix the harness before attributing any failure to the model; 'the next time your agent fails, look at the harness.'

Narratives

  • Hugging Face took a frozen open model scoring 0% on a hard legal-agent benchmark, left its weights alone, and let an automated loop rewrite only the surrounding code. By the end the system essentially matched Sonnet 4.6 on the headline metric at ~7x lower cost per task, zero weights changed. The 0% turned out to be misfiling (correct analysis saved under the wrong name / in a scratch folder / never written); the biggest single gain was an automatic file-handling step, not intelligence; code fixes transferred across models while tuned prompts did not; and five harnesses on the same model spanned 3.5-80.1%.
    → The entire empirical spine of the thesis — one worked case carrying every generalization (leverage, portability, dominance, ceiling).
  • LangChain changed only the infrastructure wrapping their LLM (same model, same weights) and jumped from outside the top 30 to rank 5 on TerminalBench 2.0; a separate research project reached a 76.4% pass rate by having an LLM optimize the infrastructure itself, surpassing hand-designed systems.
    → A second, independent instance that the harness — not the model — drives benchmark rank; corroborates the thread's single case.
  • Manus was rebuilt five times in six months, each rewrite removing complexity: complex tool definitions became general shell execution; 'management agents' became simple structured handoffs.
    → Demonstrates that 'evolving' a harness includes shrinking it — the concrete basis for reconciling the title with 'prune your scaffolding as the model improves.'

Questions addressed

  • ✅ When an agent underperforms on a benchmark, is it the model or the harness that failed?
    Largely the harness, until harness gains flatten — and you cannot attribute failure to the model until the harness is fixed, because the score measures both together.
  • ✅ Can harness tuning be made systematic rather than slow, manual, and model-specific?
    Yes — automate it with a proposer that adds one mechanism per iteration and an outer loop that keeps it only if it clearly beats the current best, so accepted mechanisms compound.
  • ✅ Do harness improvements port across models?
    Partially: code-layer fixes transfer (a smaller same-family model gained 14 points), but prompt playbooks do not and can even hurt a different model family.

Gaps

  • incompleteNo binding-constraint diagnostic. The whole thesis presupposes the harness — not the model's capability ceiling — is what limits your task, but the piece offers no way to tell in advance whether a given task is harness-bound or capability-bound; you only learn which after gains flatten. The headline example (file-plumbing, 0% -> parity) is a selected case where the harness was the entire story and does not generalize to reasoning-bound tasks. This is the Meadows leverage-point question left unanswered.
    Read next: Read Boeckeler's adaptive-control harness framing and Anthropic's 'remove components one at a time, keep only the load-bearing' methodology, looking for a procedure that predicts, before investment, which constraint binds — e.g. a cheap probe that separates 'model got the answer but the wrapper lost it' from 'model can't do this at all.'
  • incompleteNo eval-construction guidance, yet the whole method depends on it. The automated proposer + keep-if-better loop presupposes a reliable task judge to select against — and the cited judge was a brittle exact-filename grader that itself created much of the 'failure.' The piece never says how to build the objective the loop optimizes, which is where the real difficulty (and the risk of optimizing to a bad metric) lives.
    Read next: Read the evals literature and the extract-20-loop-design-patterns Score-and-Retry / Judge Ensemble / LLM-as-judge patterns for how to construct and calibrate the gradable objective a self-optimizing harness loop requires.
  • incomplete'Don't train' vs 'evolve the harness' is a soft strawman. Fine-tuning is never actually analyzed — no cost, no cases where training genuinely IS the right lever (domain shift, latency/cost at scale, proprietary capability the frontier lacks). For a rented frontier model the real contrast is harness-vs-nothing, so the title over-generalizes into 'never train.'
    Read next: Read a fine-tuning cost/benefit source to bound when weight-changing is warranted, so the 'harness-first' heuristic has an explicit boundary rather than an absolute.
  • incompleteThe automated 'add one mechanism, keep if better' loop has no pruning or garbage-collection step, so accreted mechanisms can ratchet harness complexity upward — precisely the cost the quoted deep-dive warns against (scaffolding should shrink; tight model-coupling creates lock-in). The maintenance and portability cost of an ever-growing auto-generated harness is named in the abstract but never quantified or bounded.
    Read next: Read on harness co-training overfitting and the 'delete components one at a time' methodology; look for how an automated harness search can be given a complexity penalty or removal operator so it thins rather than only accretes.
  • incompleteSecond-hand, single case, no first-party numbers. The entire empirical spine is one relayed Hugging Face experiment (7x cost, 14-point transfer, 3.5-80.1% spread) with no reproducible benchmark, dataset, task definition, or independent replication — thin ground for a rigorous adopter to act on.
    Read next: Read the source Hugging Face post directly and seek independent harness-ablation studies before treating the 7x / 3.5-80.1% figures as generalizable.
