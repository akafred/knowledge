Key terms

  • ultrathink
    A prompt keyword the thread says signals xhigh reasoning effort for a single response, without changing session settings or triggering workflow orchestration.
  • ultracode (/effort ultracode)
    A Claude Code session setting the thread says combines xhigh reasoning with automatic dynamic workflow orchestration for every substantive task, and that requires an xhigh-capable model (Fable 5, Opus 4.8, Opus 4.7; not Sonnet 4.6 or earlier).
  • effort
    Reframed by the thread as 'not a quality dial you crank up' but a cost-and-latency trade-off calibrated to the task. This drops the official guide's framing of effort as the intelligence/latency/cost control and omits its over-deliberation warning.
  • thinking.display
    An API setting the thread says controls thinking-block content: 'summarized' returns a readable summary, 'omitted' (default) returns empty thinking fields; raw chain of thought is never returned for Fable 5 / Mythos 5.
  • audit (audit-language)
    The load-bearing verb in the progress-verification block; the thread claims Anthropic tested multiple phrasings and found audit-language produced more reliable self-checking than softer equivalents.

Narratives

  • The progress-verification block: 'Before reporting progress, audit each claim against a tool result from this session. Only report work you can point to evidence for... if tests fail, say so with the output... When something is done and verified, state it plainly without hedging.'
    → Demonstrates the anti-fabrication technique the thread calls its single most useful prompt.
  • The proactivity constraint block: 'When the user is describing a problem, asking a question, or thinking out loud rather than requesting a change, the deliverable is your assessment. Report your findings and stop. Do not apply a fix until asked...'
    → Demonstrates constraining unrequested actions in unattended workflows — an assessment is a complete deliverable.
  • The vision UI prompt: 'Here is the design target and a screenshot of the current implementation. Use vision to identify differences and generate the changes needed to close the gap. Crop and zoom into any areas where the comparison is unclear before reporting findings.'
    → Demonstrates using Fable 5's vision to self-evaluate a build against a design mockup, and how the explicit crop/zoom instruction unlocks the bash-preprocessing behavior.
  • The keyword invocations: put 'ultrathink' in a prompt for single-turn xhigh; set '/effort ultracode' in Claude Code for session-level xhigh plus dynamic workflow orchestration.
    → Demonstrates thread-only shortcuts for forcing maximum reasoning without building a parallel-agent workflow.

Questions addressed

  • ✅ What actually has to change when I move an Opus 4.8 workflow to Fable 5 — is swapping the model name enough?
    No; audit six areas first (timeouts, refusal handling, proactivity constraints, progress verification, memory structure, final-response format) and validate on a task that was genuinely hard/impossible for Opus 4.8. First-attempt completion of that hard workflow is the benchmark.
  • ✅ Why are Fable 5 responses so slow, and should I keep my old timeouts?
    Slowness is the model planning and self-checking correctly (minutes at high effort, hours autonomous); extend timeouts rather than shorten them.
  • ✅ How do I force maximum reasoning without building a full parallel-agent workflow?
    The thread says use 'ultrathink' in a single prompt for xhigh, or '/effort ultracode' for a whole Claude Code session — though these controls are uncited and absent from the captured official guide, so they need verification.
  • ✅ How do I stop a long autonomous run from fabricating progress?
    Add the audit-each-claim-against-a-tool-result block to the system prompt (the guide's technique, relayed here).
