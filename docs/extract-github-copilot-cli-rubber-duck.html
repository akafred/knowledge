GitHub Copilot CLI combines model families for a second opinion

Nick McKenna · article · 2026-02 source ↗ Reader 🖥️

Coding AgentsLLM LimitationsLLM Reliability FailuresMulti-Agent Orchestration
A single coding agent reviewing its own work is limited by its training biases; pairing it with a second model from a different AI family at key checkpoints closes most of the quality gap to a stronger model, catching architectural errors, subtle bugs, and cross-file conflicts that self-reflection alone misses.

practical · product announcement / technical blog post — Announces and explains Rubber Duck, a cross-model review feature in GitHub Copilot CLI, combining product marketing with a technical rationale grounded in evaluation data.

Key terms

  • Rubber Duck Multi-Agent Orchestration
    A focused review sub-agent powered by a model from a different AI family than the primary orchestrator, invoked at key checkpoints to critique the agent's plan and work.
  • cross-family review Multi-Agent Orchestration
    Having a model from a different training lineage (e.g., GPT-5.4 reviewing Claude's work) to surface blind spots that same-family self-reflection cannot.
  • training biases LLM Reliability Failures
    Systematic blind spots arising from a model's training data and techniques that persist even during self-reflection.
  • orchestrator Multi-Agent Orchestration
    The primary model driving the coding agent session, selected by the user from the model picker.
  • checkpoint
    A moment in the agent loop (after planning, after implementation, after test-writing) where a review has the highest return on investment.

Arguments

Cross-family model review (Rubber Duck) meaningfully improves coding agent quality. strength: medium
Explicit premises:
  • Self-reflection is bounded by same-family training biases.
  • Sonnet + Rubber Duck closes 74.7% of the Sonnet-to-Opus gap on SWE-Bench Pro.
  • The improvement is larger on harder, multi-file problems.
Implicit premises:
  • Models from different families have complementary (non-overlapping) blind spots.
  • SWE-Bench Pro is a representative proxy for real-world coding quality.
  • Opus alone is the appropriate quality ceiling for comparison.
Inferences:
  • Therefore a cheaper model + cross-family review approaches the quality of a more expensive model alone.
Review should happen at specific checkpoints rather than continuously. strength: medium
Explicit premises:
  • Feedback has highest return after planning, after complex implementation, and after writing tests.
  • Invoking Rubber Duck sparingly avoids getting in the way.
Implicit premises:
  • Continuous review would be too slow or too expensive.
  • The marginal value of review diminishes rapidly outside these checkpoints.
Inferences:
  • Therefore a sparse, targeted invocation strategy is optimal.

Narratives

  • Rubber Duck caught an async scheduler in OpenLibrary that would start and immediately exit, running zero jobs — and that even if fixed, one of the tasks was itself an infinite loop.
    → Illustrates the 'architectural catch' category — errors in design that cascade if undetected.
  • Rubber Duck caught a loop silently overwriting the same dict key on every iteration, dropping 3 of 4 Solr facet categories from search queries with no error thrown.
    → Illustrates the 'silent bug' category — small code errors with outsized impact and no visible failure signal.
  • Rubber Duck caught three files reading from a Redis key that the new code stopped writing, which would have silently broken the confirmation UI and cleanup paths on deploy.
    → Illustrates the 'cross-file conflict' category — changes whose consequences span multiple files and are invisible at the local level.

Questions addressed

  • ✅ Why does a coding agent sometimes produce confidently wrong plans?
    Because early planning errors compound, and self-reflection cannot escape the model's own training biases.
  • ✅ Can a second model from a different family materially improve agent quality?
    Yes — Sonnet + Rubber Duck closes 74.7% of the gap to Opus on SWE-Bench Pro.
  • ✅ When should the second opinion be invoked?
    After planning, after complex implementation, after writing tests, reactively when stuck, and on user demand.
  • ⭕ What is the cost/latency overhead of cross-family review?
    Not addressed. The article says Rubber Duck is invoked 'sparingly' but provides no data on added latency, token cost, or wall-clock time.
